1. Что такое Git — объяснение для всех

Представьте что вы пишете книгу. Каждый день вносите изменения. Через месяц понимаете что три главы назад текст был лучше — но все черновики выброшены. Именно эту проблему решает Git.

Git — система контроля версий. Она запоминает каждое изменение в вашем коде: кто изменил, когда, что именно и зачем. Вы можете вернуться к любой точке в истории проекта.

Без Git С Git
файл_v1.py commit: начало проекта
файл_v2.py commit: добавил авторизацию
файл_v2_final.py commit: исправил баг
файл_ТОЧНО_ФИНАЛ.py commit: добавил API
файл_ФИНАЛ_2_РЕАЛЬНО.py ← можно вернуться к любому моменту

💡 Git — это не просто резервная копия. Это полная история проекта с возможностью работать в команде над одним кодом одновременно.

1.1 Основные концепции

Repository (Репозиторий)

Папка проекта которую отслеживает Git. Внутри есть скрытая папка .git где хранится вся история.

ls -la ~/git-practice

# drwxr-xr-x .git ← вся история здесь

Commit (Коммит)

Снимок состояния проекта в определённый момент. Каждый коммит имеет уникальный ID (хэш), автора, дату и сообщение.

commit a3f4b2c

Author: Kirill <kirill@example.com>

Date: Thu May 8 10:00:00 2026

Message: feat: добавил nginx конфигурацию

Branch (Ветка)

Параллельная линия разработки. Как альтернативная реальность — вносите изменения не затрагивая основной код.

main ──●──●──●──────────────●

└──●──●──●──┘

feature/auth

Staging Area (Индекс)

Промежуточная зона перед коммитом. Вы выбираете какие именно изменения включить в следующий коммит.

Рабочая папка → Staging Area → Repository

(изменили) (git add) (git commit)

Remote (Удалённый репозиторий)

Копия репозитория на сервере (GitHub, GitLab). Используется для резервного копирования и командной работы.

2. Установка и первоначальная настройка

2.1 Установка Git

# Fedora / РЕД ОС

sudo dnf install -y git

# Ubuntu / Debian

sudo apt install -y git

# Проверить версию

git --version

# git version 2.47.0

2.2 Обязательная настройка после установки

Это первое что нужно сделать на любой новой машине. Git требует знать кто делает коммиты.

# Установить имя — будет отображаться в коммитах

git config --global user.name "Kirill"

# Установить email

git config --global user.email "kirill@example.com"

# Установить ветку по умолчанию main вместо устаревшего master

git config --global init.defaultBranch main

# Установить редактор для сообщений коммитов

git config --global core.editor vim

# Проверить все настройки

git config --global --list

⚠️ Реальная ошибка из практики: при первом коммите Git выдал ошибку 'Author identity unknown'. Причина — не были настроены user.name и user.email. Всегда настраивайте их сразу после установки Git!

❌ git commit без настройки user.name и user.email → ошибка 'Please tell me who you are'

✅ Выполните git config --global user.name и user.email перед первым коммитом

2.3 Глобальный .gitignore

Файл который защищает секреты во ВСЕХ ваших репозиториях автоматически. Настроить один раз — работает везде.

# Создать глобальный .gitignore

nano ~/.gitignore_global

Содержимое файла:

.env

.env.*

!.env.example

*.pem

*.key

id_rsa

id_ed25519

*.log

__pycache__/

*.pyc

venv/

.venv/

.idea/

.vscode/

*.swp

.DS_Store

# Подключить глобальный .gitignore

git config --global core.excludesfile ~/.gitignore_global

# Проверить что подключён

git config --global --list | grep excludes

# core.excludesfile=/home/kirill/.gitignore_global

3. Создание репозитория

3.1 Новый репозиторий

# Создать папку и инициализировать репозиторий

mkdir ~/my-project

cd ~/my-project

git init

# Инициализирован пустой репозиторий Git в /home/kirill/my-project/.git/

⚠️ Реальная ошибка из практики: при git init создалась ветка 'master' вместо 'main'. Исправление: git branch -m master main. Чтобы этого не происходило — настройте git config --global init.defaultBranch main заранее.

❌ Ветка называется master — устаревший стандарт

✅ git branch -m master main — переименовать ветку

3.2 Клонирование существующего репозитория

# Клонировать репозиторий

git clone https://github.com/user/repo.git

# Создаст папку repo/ со всем содержимым

# Клонировать в конкретную папку

git clone https://github.com/user/repo.git my-folder

# Клонировать только последний коммит (быстрее для больших репо)

# --depth=1 — глубина истории: только 1 последний коммит

git clone --depth=1 https://github.com/user/repo.git

# Клонировать конкретную ветку

# -b — указать ветку для клонирования

git clone -b develop https://github.com/user/repo.git

4. Файл .gitignore

Файл .gitignore указывает Git какие файлы НЕ отслеживать. Это критично для безопасности — чтобы пароли и секреты не попали в репозиторий.

4.1 Как работает .gitignore

Все строки в .gitignore работают ОДНОВРЕМЕННО — это не конфиг где последнее значение побеждает. Файл игнорируется если подходит хотя бы под одно правило.

Паттерн Что игнорирует Пример
.env Только файл .env .env
*.log Все файлы с расширением .log app.log, error.log
.env.* Файлы начинающиеся с .env. .env.local, .env.prod
logs/ Папку logs целиком logs/app.log, logs/error.log
!important.log НЕ игнорировать этот файл important.log будет отслеживаться
**/temp Папку temp в любом месте src/temp/, dist/temp/

⚠️ Реальная ошибка из практики: написали *.env вместо .env. Разница: *.env игнорирует файлы ЗАКАНЧИВАЮЩИЕСЯ на .env (prod.env, dev.env), а .env игнорирует ИМЕННО файл .env. Для защиты файла .env используйте именно .env в .gitignore.

4.2 Важное правило — .gitignore работает только с неотслеживаемыми файлами

Если файл уже был закоммичен — добавление его в .gitignore НЕ поможет. Git продолжит его отслеживать.

# Проверить отслеживается ли файл

git ls-files | grep .env

# Убрать файл из отслеживания НО оставить на диске

# --cached — только из индекса Git, не с диска

git rm --cached .env

# Для папки

# -r — рекурсивно

git rm --cached -r папка/

# После этого закоммитить

git commit -m "chore: убрал .env из отслеживания"

⚠️ Даже после git rm --cached файл остаётся в ИСТОРИИ старых коммитов! Если файл содержал реальные секреты — их нужно считать скомпрометированными и немедленно сменить.

4.3 Правильная структура .gitignore для Python проекта

# Секреты — никогда не коммитим

.env

.env.*

# НО шаблон без секретов можно

!.env.example

# Python

__pycache__/

*.pyc

*.pyo

venv/

.venv/

# Логи

*.log

logs/

# IDE

.idea/

.vscode/

💡 Золотое правило: .gitignore должен быть ПЕРВЫМ коммитом в любом проекте. Создайте его до того как добавлять остальные файлы.

4.4 Проверка что файл игнорируется

# Проверить почему файл игнорируется

# -v — verbose, показать какое правило сработало

git check-ignore -v .env

# .gitignore:3:.env .env

# Файл: строка 3 в .gitignore, правило: .env

5. Основные команды работы с файлами

5.1 git status — состояние репозитория

Самая часто используемая команда. Показывает что изменилось, что добавлено в staging, что не отслеживается.

git status

# Короткий вывод

# -s — short, компактный формат

git status -s

# M app.py — изменён и добавлен в staging

# MM config.py — изменён в staging И после этого ещё раз

# ?? new.py — новый файл не отслеживается

# A added.py — новый файл добавлен в staging

5.2 git add — добавление в staging

# Добавить конкретный файл

git add app.py

# Добавить несколько файлов

git add app.py config.py utils.py

# Добавить все изменения в текущей папке

git add .

# Добавить все файлы с расширением .py

git add *.py

# Добавить всю папку

git add src/

# Интерактивное добавление по кускам — ВАЖНАЯ команда!

# -p — patch, показывает каждый кусок изменений

git add -p app.py

Подробно о git add -p

Позволяет добавлять изменения не весь файл целиком а по кускам. Незаменимо когда в одном файле несколько несвязанных изменений.

Интерфейс показывает каждый кусок и спрашивает:

Клавиша Действие
y yes — добавить этот кусок в staging
n no — пропустить этот кусок
s split — разбить кусок на меньшие части
e edit — отредактировать кусок вручную
q quit — выйти, остальные куски не трогать
a all — добавить этот и все следующие куски
d don't — пропустить этот и все следующие
? help — показать все команды

💡 git add -p — признак профессионала. Позволяет делать атомарные коммиты: один коммит = одно логическое изменение. Работодатели это ценят.

5.3 git commit — создание коммита

# Создать коммит с сообщением

# -m — message, сообщение коммита

git commit -m "feat: добавил авторизацию"

# Добавить все изменённые файлы и закоммитить

# -a — all modified files, -m — message

git commit -am "fix: исправил баг в авторизации"

# ВНИМАНИЕ: -a добавляет только уже ОТСЛЕЖИВАЕМЫЕ файлы

# Новые файлы нужно сначала добавить через git add

# Изменить последний коммит (если ещё не отправили на сервер)

# --amend — изменить последний коммит

git commit --amend -m "Новое сообщение"

# Добавить забытый файл в последний коммит

git add забытый_файл.py

# --no-edit — не менять сообщение коммита

git commit --amend --no-edit

⚠️ Реальная ошибка из практики: попытка сделать git commit без предварительного git add. Git выдал 'индекс пуст'. Порядок всегда такой: изменить файл → git add → git commit.

❌ git commit без git add → ошибка 'nothing to commit' или 'индекс пуст'

✅ Всегда: изменить файл → git add файл → git commit -m 'сообщение'

5.4 Правила написания сообщений коммитов

Хорошие сообщения коммитов — это профессионализм. Работодатели смотрят на историю коммитов на собеседовании.

Плохо ❌ Хорошо ✅
fix fix: исправил утечку памяти в обработчике запросов
asdf feat: добавил авторизацию через JWT токены
изменения docs: обновил README с инструкцией по Docker
работает наконец chore: обновил зависимости до актуальных версий
WTF refactor: вынес логику авторизации в отдельный модуль

Стандарт Conventional Commits — формат: тип: краткое описание

Тип Когда использовать Пример
feat Новая функциональность feat: добавил поиск по пользователям
fix Исправление бага fix: исправил падение при пустом запросе
docs Документация docs: добавил примеры в README
chore Обслуживание chore: обновил зависимости
refactor Рефакторинг refactor: упростил функцию авторизации
test Тесты test: добавил тесты для API
ci CI/CD ci: добавил GitHub Actions
style Форматирование style: исправил отступы

6. Просмотр истории и изменений

6.1 git log — история коммитов

# Полная история

git log

# Компактная история — каждый коммит в одну строку

# --oneline — краткий формат

git log --oneline

# 93730bf chore: Добавил .gitignore

# 5f1e318 feat: Добавил первый скрипт

# 80a0a90 docs: добавил README

# История с графом веток

# --graph — показать ASCII граф веток

# --all — показать все ветки

git log --oneline --graph --all

# Последние N коммитов

# -5 — показать последние 5

git log -5

# История конкретного файла

git log --oneline app.py

# Что изменилось в каждом коммите

# -p — patch, показать diff каждого коммита

git log -p -3

# Поиск по сообщению коммита

# --grep — фильтр по тексту в сообщении

git log --grep="авторизация"

# История за период

# --since и --until — временной диапазон

git log --since="2 weeks ago"

git log --since="2026-01-01" --until="2026-05-01"

# История конкретного автора

# --author — фильтр по автору

git log --author="Kirill"

6.2 git diff — просмотр изменений

# Что изменилось но ещё не добавлено в staging

git diff

# Что добавлено в staging (будет в следующем коммите)

# --staged — показать изменения в staging area

git diff --staged

# Разница между двумя коммитами

git diff abc123 def456

# Разница между ветками

git diff main feature/auth

# Только имена изменённых файлов

# --name-only — только имена без содержимого

git diff --name-only

7. Ветки — Branch

Ветки — одна из главных возможностей Git. Позволяют разрабатывать новые функции не затрагивая основной код.

7.1 Управление ветками

# Показать все локальные ветки

git branch

# * main ← звёздочка = текущая ветка

# feature/auth

# Показать все ветки включая удалённые

# -a — all, все ветки

git branch -a

# Создать новую ветку

git branch feature/auth

# Переключиться на ветку (современный способ)

git switch feature/auth

# Создать И сразу переключиться

# -c — create

git switch -c feature/auth

# Старый способ (тоже работает)

# -b — branch, создать и переключиться

git checkout -b feature/auth

# Переименовать ветку

# -m — move/rename

git branch -m старое-имя новое-имя

# Удалить ветку (только если смержена)

# -d — delete

git branch -d feature/auth

# Принудительно удалить ветку

# -D — force delete

git branch -D feature/auth

7.2 Слияние веток — git merge

# Переключиться в ветку куда сливаем

git switch main

# Слить ветку feature/auth в текущую

git merge feature/auth

# Слить с созданием коммита слияния (рекомендуется)

# --no-ff — no fast-forward, всегда создавать коммит слияния

git merge feature/auth --no-ff -m "merge: добавил авторизацию"

# Отменить слияние если возник конфликт

# --abort — отменить и вернуться к состоянию до merge

git merge --abort

7.3 Конфликты слияния

Конфликт возникает когда две ветки изменили одну и ту же строку в файле. Git не знает какую версию оставить и просит вас решить.

Как выглядит конфликт в файле:

<<<<<<< HEAD

Версия из текущей ветки (main)

=======

Версия из сливаемой ветки (feature/auth)

>>>>>>> feature/auth

Как решить конфликт:

  1. Откройте файл с конфликтом в редакторе

  2. Удалите маркеры <<<<<<, ======, >>>>>>>

  3. Оставьте правильную версию (или объедините обе)

  4. Сохраните файл

  5. git add конфликтный_файл

  6. git commit -m "merge: решил конфликт"

# Посмотреть файлы с конфликтами

git status

# both modified: app.py

8. Отмена изменений

Одна из самых важных тем — уметь отменять изменения. Разные команды для разных ситуаций.

Ситуация Команда Что делает
Отменить изменения в файле (до git add) git restore файл Возвращает файл к последнему коммиту
Убрать файл из staging git restore --staged файл Файл остаётся изменённым но убирается из индекса
Отменить последний коммит (сохранить изменения) git reset HEAD~1 Коммит удаляется, изменения остаются
Отменить последний коммит (удалить изменения) git reset --hard HEAD~1 Коммит И изменения удаляются НАВСЕГДА
Безопасная отмена (для командной работы) git revert HEAD Создаёт новый коммит отменяющий изменения

# Отменить изменения в конкретном файле

git restore app.py

# Отменить ВСЕ незакоммиченные изменения

git restore .

# Убрать файл из staging

# --staged — убрать из индекса

git restore --staged app.py

# Отменить последний коммит сохранив изменения

# HEAD~1 — один коммит назад от HEAD

git reset HEAD~1

# Отменить последний коммит УДАЛИВ изменения

# --hard — жёсткий сброс, изменения теряются

git reset --hard HEAD~1

# Безопасная отмена — создаёт новый коммит

# Подходит когда уже отправили на сервер

git revert HEAD

⚠️ git reset --hard необратим! Изменения удаляются навсегда. Используйте только если уверены. Для командной работы используйте git revert — он безопаснее.

9. Stash — временное сохранение

Stash (тайник) — временное хранилище для незакоммиченных изменений. Полезно когда нужно срочно переключиться на другую ветку не делая коммит.

# Сохранить изменения в stash

git stash

# Saved working directory and index state WIP on main

# Сохранить с описанием

# save — сохранить с именем

git stash save "работа над авторизацией"

# Посмотреть список сохранений

git stash list

# stash@{0}: WIP on main: работа над авторизацией

# stash@{1}: WIP on feature: другие изменения

# Восстановить последнее сохранение и удалить из stash

# pop — восстановить и удалить

git stash pop

# Восстановить без удаления из stash

# apply — восстановить но оставить в stash

git stash apply

# Восстановить конкретное сохранение

git stash pop stash@{1}

# Удалить все сохранения

git stash clear

💡 Типичный сценарий: работаете над функцией, пришёл срочный баг-фикс. git stash → переключились на main → исправили баг → закоммитили → вернулись на ветку → git stash pop.

10. Работа с удалённым репозиторием

10.1 Настройка remote

# Показать удалённые репозитории

# -v — verbose, показать URL

git remote -v

# origin https://github.com/user/repo.git (fetch)

# origin https://github.com/user/repo.git (push)

# Добавить удалённый репозиторий

# origin — стандартное имя основного репозитория

git remote add origin https://github.com/user/repo.git

# Изменить URL удалённого репозитория

# set-url — изменить URL

git remote set-url origin https://github.com/user/new-repo.git

# Удалить удалённый репозиторий

git remote remove origin

10.2 Получение изменений

# Скачать изменения НЕ применяя их

git fetch origin

# Изменения скачаны в origin/main но не влиты в main

# Скачать И применить изменения

# = git fetch + git merge

git pull origin main

# Скачать с перебазированием (чище история)

# --rebase — вместо merge использовать rebase

git pull --rebase origin main

10.3 Отправка изменений

# Отправить изменения

git push origin main

# Отправить новую ветку и установить upstream

# -u — upstream, связать локальную ветку с удалённой

git push -u origin feature/auth

# После -u можно просто писать git push

# Отправить все ветки

git push --all origin

# Отправить теги

git push --tags

# Принудительная отправка (ОСТОРОЖНО!)

# --force — перезаписать историю на сервере

git push --force origin main

⚠️ git push --force перезаписывает историю на сервере. В командной работе это может удалить коммиты других людей! Используйте только для своих веток и только если точно знаете что делаете.

11. Теги — Tags

Теги используются для маркировки важных точек в истории — обычно версий релизов.

# Создать лёгкий тег

git tag v1.0.0

# Создать аннотированный тег с описанием

# -a — annotated, -m — message

git tag -a v1.0.0 -m "Первый стабильный релиз"

# Показать все теги

git tag

# Отправить теги на сервер

git push origin --tags

# Удалить тег

git tag -d v1.0.0

13. Типичные ошибки и их исправление

Ошибка 1 — Коммит без настройки пользователя

❌ fatal: unable to auto-detect email address

✅ git config --global user.name "Имя"

✅ git config --global user.email "email@example.com"

Ошибка 2 — Ветка master вместо main

❌ Создалась ветка master — устаревший стандарт

✅ git branch -m master main — переименовать

✅ git config --global init.defaultBranch main — исправить для будущих репо

Ошибка 3 — Опечатка в названии файла

❌ Создали READMI.md вместо README.md

✅ git mv READMI.md README.md — переименовать через Git

✅ git commit -m "docs: переименовал README"

Ошибка 4 — Забыли git add перед commit

❌ nothing to commit / индекс пуст

✅ git add файл — сначала добавить в staging

✅ git commit -m "сообщение" — потом коммитить

Ошибка 5 — Закоммитили секрет

❌ Пароль или API ключ попал в коммит

✅ Немедленно отозвать скомпрометированный ключ/пароль!

✅ git rm --cached .env

✅ git commit -m "chore: убрал .env"

✅ Добавить .env в .gitignore

Ошибка 6 — gitignore вместо .gitignore

❌ Создали файл gitignore без точки в начале — Git его не видит

✅ mv gitignore .gitignore — переименовать добавив точку

✅ Точка в начале имени делает файл скрытым в Linux

Ошибка 7 — Неправильный паттерн в .gitignore

❌ *.env не защищает файл .env (только файлы вида prod.env)

✅ Используйте .env для защиты именно файла .env

✅ Используйте .env.* для файлов вида .env.local, .env.prod

14. Шпаргалка — ежедневные команды

Начало работы

git pull # получить последние изменения

git status # что изменилось

Работа с кодом

git add . # добавить всё

git add -p файл # добавить по кускам

git commit -m "тип: текст" # закоммитить

git push # отправить

Ветки

git switch -c feature/xxx # новая ветка

git switch main # вернуться в main

git merge feature/xxx --no-ff # слить ветку

git branch -d feature/xxx # удалить ветку

Когда что-то пошло не так

git log --oneline --graph # посмотреть историю

git diff # что изменилось

git restore . # отменить все изменения

git stash # спрятать изменения

git reset HEAD~1 # отменить последний коммит

Диагностика

git status -s # краткий статус

git log --oneline -10 # последние 10 коммитов

git check-ignore -v .env # почему файл игнорируется

git ls-files # все отслеживаемые файлы

git remote -v # удалённые репозитории
