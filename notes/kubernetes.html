// 01Что такое Kubernetes и зачем

Kubernetes (K8s) — оркестратор контейнеров. Описываете «как должно быть» — он сам поддерживает это состояние.

Что даёт

  • Self-healing — упавшие контейнеры перезапускаются автоматически.
  • Horizontal scaling — увеличить replicas одной командой / по метрике.
  • Service discovery — внутренний DNS, балансировка.
  • Rolling updates — обновление без downtime, откат.
  • Resource management — bin-packing подов на ноды.
  • Декларативность — git как источник истины.

Когда не Kubernetes

  • Маленький стартап с 1–3 сервисами — Docker Compose + один VPS дешевле и проще.
  • Stateful монолит на 1 хосте — overkill.
  • Команда без операционных навыков — K8s сам по себе — целая профессия.
// Контекст 2026
Базовый K8s сейчас встречается практически в каждой второй DevOps-вакансии. Не обязательно уметь поднять кластер с нуля — но читать манифесты, дебажить pods, писать Helm-чарты требуют от джуниоров.

// 02Архитектура: control plane и nodes

Control plane

  • kube-apiserver — единственная точка входа. REST/JSON, валидирует и пишет в etcd.
  • etcd — распределённое KV-хранилище. Источник истины кластера. Raft. Бэкап etcd = бэкап кластера.
  • kube-scheduler — решает, на какую ноду посадить новый Pod.
  • kube-controller-manager — крутит controller'ы (Deployment, Job, NodeController...).
  • cloud-controller-manager — интеграция с облаком (LoadBalancer, IP, volumes).

Worker node

  • kubelet — агент, общается с apiserver, запускает контейнеры через CRI.
  • kube-proxy — настраивает iptables/IPVS для Service.
  • Container runtime — containerd (default), CRI-O. Docker dockershim удалён с 1.24.

Поток создания Pod

  1. kubectl apply → apiserver валидирует и пишет в etcd (status: Pending).
  2. Scheduler видит Pod без node, выбирает ноду по фильтрам/скорам, обновляет nodeName.
  3. kubelet нужной ноды видит «свой» Pod, тянет образ через CRI, запускает контейнеры.
  4. kubelet репортит status обратно в apiserver.

// 03Объекты, манифесты, declarative

Всё в K8s — объект (Pod, Service, ConfigMap...). Описывается YAML или JSON. Структура:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
  namespace: prod
  labels:
    app: web
    env: prod
  annotations:
    description: "Public frontend"
spec:
  ...
status:                    # заполняет кластер, не вы
  ...

Декларативность

Вы пишете желаемое (spec). Controller-loop ищет дельту между spec и реальностью и сводит её к нулю.

Namespace

Логическая изоляция: ресурсы группируются (kube-system, prod, staging). У одного имени Pod может быть только в одном ns.

Labels vs Annotations

  • Labels — для селекторов (selector.matchLabels, kubectl get -l). Короткие, идентификация.
  • Annotations — для метаданных (changelog, URL дашборда). Не используются для отбора.

Императив vs декларатив

# Императив (быстро, для экспериментов)
kubectl create deployment web --image=nginx
kubectl scale deployment web --replicas=3
kubectl set image deployment/web nginx=nginx:1.27

# Декларатив (правильно, в git)
kubectl apply -f web.yaml
kubectl apply -f manifests/
kubectl diff -f web.yaml         # увидеть, что поменяется

// 04Pod — атомарная единица

Pod — один или несколько контейнеров на одной ноде с общими namespace (network, IPC) и volume.

apiVersion: v1
kind: Pod
metadata: { name: web, labels: { app: web } }
spec:
  containers:
    - name: app
      image: nginx:1.27
      ports: [ { containerPort: 80 } ]
      env:
        - name: ENV
          value: prod
      resources:
        requests: { cpu: 100m, memory: 128Mi }
        limits:   { cpu: 500m, memory: 256Mi }
  initContainers:
    - name: wait-db
      image: busybox
      command: ["sh","-c","until nc -z db 5432; do sleep 1; done"]
  restartPolicy: Always         # default; Pod-level

Sidecar pattern

Несколько контейнеров в одном Pod: основной + вспомогательный (envoy proxy, log-shipper, cron-job).

Лифесайкл

  • Pending — Pod создан, ждёт планировщика/образ.
  • Running — хотя бы один контейнер запущен.
  • Succeeded — все контейнеры завершились с exit 0 (Jobs).
  • Failed — хотя бы один контейнер упал и не будет рестартиться.
  • Unknown — kubelet не отвечает.

restartPolicy

  • Always — для долгоживущих сервисов (default).
  • OnFailure — для Job (рестарт при exit != 0).
  • Never — для одноразовых.
// Важно
Голый Pod в проде не используется. Если нода упадёт, Pod не воскреснет. Вместо этого — Deployment / StatefulSet / DaemonSet, они управляют Pod-ами через ReplicaSet.

Deployment

apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 3
  selector:
    matchLabels: { app: web }
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 25%, maxUnavailable: 25% }
  revisionHistoryLimit: 10
  template:
    metadata:
      labels: { app: web }
    spec:
      containers:
        - name: app
          image: ghcr.io/org/app:v1.2.3
          ports: [ { containerPort: 8080 } ]

kubectl rollout status deploy/web
kubectl rollout history deploy/web
kubectl rollout undo deploy/web
kubectl rollout undo deploy/web --to-revision=4
kubectl scale deploy/web --replicas=5

StatefulSet

Каждая реплика имеет стабильное имя (db-0, db-1) и свой PVC (data-db-0). Используется с headless Service (clusterIP=None) для отдельного DNS на каждый Pod.

Job / CronJob

apiVersion: batch/v1
kind: CronJob
metadata: { name: backup }
spec:
  schedule: "0 2 * * *"
  concurrencyPolicy: Forbid
  successfulJobsHistoryLimit: 3
  failedJobsHistoryLimit: 1
  jobTemplate:
    spec:
      backoffLimit: 3
      template:
        spec:
          restartPolicy: OnFailure
          containers:
            - name: backup
              image: backup:latest

// 06Services и kube-proxy

Service — стабильный endpoint поверх множества Pod (которые могут пересоздаваться).

Тип ServiceЧто делает
ClusterIPВнутренний IP, доступен только из кластера (default)
NodePortОткрывает порт на всех нодах (30000–32767)
LoadBalancerВ облаке — провижинит внешний LB
ExternalNameDNS CNAME на внешний сервис
Headless (clusterIP: None)Без виртуального IP, DNS возвращает IP всех Pod
 
apiVersion: v1
kind: Service
metadata: { name: web }
spec:
  type: ClusterIP
  selector: { app: web }
  ports:
    - port: 80          # порт самого Service
      targetPort: 8080  # порт в Pod
      protocol: TCP

DNS

CoreDNS внутри кластера: web.default.svc.cluster.local резолвится в ClusterIP. Внутри одного namespace — короткое имя web.

kube-proxy

Раскладывает правила iptables (или IPVS, eBPF в Cilium) на каждой ноде. ClusterIP — виртуальный, реальный трафик балансируется через DNAT.

Endpoints / EndpointSlice

Список реальных IP Pod'ов, которые матчат selector. kube-proxy подписан на изменения.

// Не путать
Pod падает → kubelet рестартит на той же ноде, но IP сохраняется. Pod удалён и пересоздан → новый IP. Service IP стабильный, потому что виртуальный.

// 07Ingress и Gateway API

Ingress — слой 7 (HTTP/HTTPS) выше Service. Один external IP на много hostname/path → Service.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: app
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts: [app.example.com]
      secretName: app-tls
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service: { name: web, port: { number: 80 } }
          - path: /api
            pathType: Prefix
            backend:
              service: { name: api, port: { number: 80 } }

Ingress controllers

  • ingress-nginx — самый популярный, Nginx-based.
  • Traefik — современный, file-watcher.
  • HAProxy ingress.
  • Cloud-нативные: AWS ALB, GCP GLB, AKS App Routing.

Gateway API (новое)

Преемник Ingress: более expressive, role-based (Gateway, HTTPRoute, TLSRoute). С 1.29 стабильно. Если стартуете с нуля и поддерживает контроллер — лучше использовать Gateway API.

cert-manager

Автоматически выпускает Let's Encrypt-сертификаты по аннотации на Ingress. Стандарт для prod.

// 08ConfigMap, Secret, env

ConfigMap

apiVersion: v1
kind: ConfigMap
metadata: { name: app-config }
data:
  LOG_LEVEL: info
  feature_flags: "new_ui,beta_api"
  app.conf: |
    server.port=8080
    server.timeout=30s

Secret

Тот же ConfigMap, но base64 encoded (это не шифрование!). Шифрование at-rest настраивается в etcd.

apiVersion: v1
kind: Secret
metadata: { name: db-cred }
type: Opaque
data:
  password: czNjcjN0       # base64 of "s3cr3t"

Использование в Pod

spec:
  containers:
    - name: app
      image: app
      envFrom:
        - configMapRef: { name: app-config }
        - secretRef: { name: db-cred }
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef: { name: db-cred, key: password }
      volumeMounts:
        - name: config
          mountPath: /etc/app
          readOnly: true
  volumes:
    - name: config
      configMap:
        name: app-config
        items: [ { key: app.conf, path: app.conf } ]

Внешние секреты

Для prod храните секреты вне кластера: External Secrets Operator синхронизирует Vault/AWS Secrets Manager/1Password в K8s Secret. Sealed Secrets (Bitnami) — зашифрованные секреты в git.

// Внимание
Secret в etcd хранится в base64 — это не шифрование. Любой с доступом к etcd видит plain text. Включайте EncryptionConfiguration + храните ключ KMS вне кластера.

// 09Volumes, PV, PVC, StorageClass

Уровни абстракции

  • PersistentVolume (PV) — конкретный кусок хранилища (диск EBS, том в Ceph). Создаётся админом или динамически.
  • PersistentVolumeClaim (PVC) — «хочу 10 GB SSD». Пишет приложение / разработчик.
  • StorageClass — рецепт, как провижинить PV (тип диска, провайдер).
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: data }
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: gp3
  resources:
    requests: { storage: 10Gi }

Access modes

  • ReadWriteOnce (RWO) — одна нода может монтировать. Block-device.
  • ReadWriteMany (RWX) — много нод. Только сетевые ФС (NFS, CephFS).
  • ReadOnlyMany (ROX) — много нод, только чтение.
  • ReadWriteOncePod (RWOP) — один Pod на весь кластер.

volumeClaimTemplates в StatefulSet

spec:
  serviceName: db
  replicas: 3
  template: ...
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: [ReadWriteOnce]
        storageClassName: gp3
        resources: { requests: { storage: 50Gi } }

Для каждой реплики создаётся свой PVC (data-db-0, data-db-1, data-db-2).

Прочие volume types

  • emptyDir — пустая папка, живёт пока Pod жив. Scratch space.
  • hostPath — путь на ноде. Только для системных DaemonSet, в обычном prod избегайте.
  • projected — комбо ConfigMap + Secret + downwardAPI в одном mount.

// 10Resources, requests, limits, QoS

Requests vs Limits

  • requests — гарантированный ресурс. Scheduler ищет ноду с этим запасом.
  • limits — потолок. Превышение CPU → throttling; превышение memory → OOMKill.
resources:
  requests:
    cpu: 100m            # 0.1 CPU
    memory: 128Mi
  limits:
    cpu: 500m
    memory: 256Mi

Единицы

  • CPU: 1 = 1 ядро, 100m = 0.1 ядра (milliCPU).
  • Memory: 128Mi = 128 × 2²⁰ байт. Не путайте с M (10⁶).

QoS классы

КлассУсловиеЧто значит
Guaranteedrequests == limits для всех container'овПоследним убивается при пресс. на ноду
Burstablerequests < limits или только часть заданаСреднее
BestEffortНи requests, ни limitsПервым убивается при OOM

LimitRange / ResourceQuota

На уровне namespace:

  • LimitRange — дефолтные/min/max requests-limits для контейнеров.
  • ResourceQuota — суммарный лимит на namespace (например, max 10 CPU и 20 GiB RAM).

HPA — Horizontal Pod Autoscaler

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata: { name: web }
spec:
  scaleTargetRef: { apiVersion: apps/v1, kind: Deployment, name: web }
  minReplicas: 2
  maxReplicas: 10
  metrics:
    - type: Resource
      resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 70 }

Нужен metrics-server. Для custom-метрик (RPS, queue size) — Prometheus Adapter / KEDA.

// Правило большого пальца
Всегда задавайте requests. Без них scheduler не знает, что положить на ноду — будут плохие сюрпризы. limits на memory ставьте; на CPU — спорно (CPU throttling часто бьёт по latency).

// 11Probes: liveness, readiness, startup

ProbeНа что отвечаетЧто делает kubelet
liveness«Контейнер жив?»Если падает — рестартует контейнер
readiness«Готов принимать трафик?»Если падает — убирает Pod из endpoints Service
startup«Запустился?»Откладывает liveness/readiness, пока не пройдёт
 
livenessProbe:
  httpGet: { path: /healthz, port: 8080 }
  initialDelaySeconds: 10
  periodSeconds: 10
  timeoutSeconds: 1
  failureThreshold: 3
readinessProbe:
  httpGet: { path: /ready, port: 8080 }
  periodSeconds: 5
  failureThreshold: 2
startupProbe:
  httpGet: { path: /healthz, port: 8080 }
  failureThreshold: 30
  periodSeconds: 10              # итого до 5 минут на старт

Принцип

  • liveness — чек чего-то проще всего, без зависимостей. «Процесс не deadlocked».
  • readiness — чек зависимостей (БД, кэш). Когда падает — трафик уходит.
  • startup — для медленно стартующих приложений (Java, миграции БД).
// Антипаттерн
Делать liveness=readiness=тот же тяжёлый /health, который проверяет всё (БД, S3, кэш). Если БД мигнула — kubelet рестартанёт все Pod'ы каскадно, обвалив сервис целиком. Разделяйте: liveness — про процесс, readiness — про зависимости.

// 12Scheduling: nodeSelector, affinity, taints

nodeSelector — простое

spec:
  nodeSelector:
    disktype: ssd
    gpu: "true"

Affinity — гибкое

spec:
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
          - matchExpressions:
              - { key: zone, operator: In, values: [eu-west-1a, eu-west-1b] }
    podAntiAffinity:                        # разнести реплики по нодам
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          podAffinityTerm:
            labelSelector: { matchLabels: { app: web } }
            topologyKey: kubernetes.io/hostname

Taints и Tolerations

Taint — «отметка» на ноде. Pod без соответствующего toleration не сядет туда.

# Пометить ноду
kubectl taint nodes node-1 dedicated=db:NoSchedule

# Pod, который может туда сесть
spec:
  tolerations:
    - key: dedicated
      operator: Equal
      value: db
      effect: NoSchedule

Effect: NoSchedule (не планировать), PreferNoSchedule (избегать), NoExecute (выселить уже работающие).

PriorityClass и PDB

  • PriorityClass — приоритет Pod при нехватке ресурсов (важные выживают, низкие — выселяются).
  • PodDisruptionBudget — гарантия «минимум 2 Pod web всегда онлайн» при добровольных эвакуациях (drain ноды, rollout).
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata: { name: web-pdb }
spec:
  minAvailable: 2
  selector:
    matchLabels: { app: web }

// 13RBAC, ServiceAccount, NetworkPolicy

RBAC

  • Role / ClusterRole — разрешения (verbs: get, list, create, update, delete) на ресурсы.
  • RoleBinding / ClusterRoleBinding — даёт Role субъекту (User/Group/ServiceAccount).
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata: { namespace: dev, name: pod-reader }
rules:
  - apiGroups: [""]
    resources: [pods, pods/log]
    verbs: [get, list, watch]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata: { namespace: dev, name: dev-can-read-pods }
subjects:
  - kind: User
    name: alice
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io

ServiceAccount

Identity для процесса внутри Pod. По умолчанию — default в каждом namespace (минимальные права). Для приложений, которые ходят в API кластера — отдельный SA с точными правами.

spec:
  serviceAccountName: app-sa
  automountServiceAccountToken: false      # явно отключить, если API не нужен

NetworkPolicy

Файрвол на уровне Pod. По умолчанию — всё со всем. NetworkPolicy включает «default deny»:

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: deny-all-ingress, namespace: prod }
spec:
  podSelector: {}
  policyTypes: [Ingress]
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-web-to-db, namespace: prod }
spec:
  podSelector:
    matchLabels: { app: db }
  policyTypes: [Ingress]
  ingress:
    - from:
        - podSelector: { matchLabels: { app: web } }
      ports:
        - protocol: TCP
          port: 5432

Работает только если CNI поддерживает: Cilium, Calico, Weave. Flannel — нет.

// 14Helm и Kustomize

Helm — пакетный менеджер

  • Chart — упакованный набор шаблонов K8s-манифестов + values.yaml.
  • Шаблоны на Go templates.
  • Release — установленный экземпляр chart с конкретными values.
  • Версионирование, откат, hooks (pre-install, post-upgrade).
# Структура chart
my-app/
├── Chart.yaml          # метаданные
├── values.yaml         # дефолтные values
├── templates/
│   ├── deployment.yaml
│   ├── service.yaml
│   └── _helpers.tpl    # подключаемые шаблоны
└── charts/             # subcharts (deps)

helm install web ./my-app -f values-prod.yaml -n prod
helm upgrade --install web ./my-app -f values-prod.yaml -n prod
helm rollback web 1
helm uninstall web
helm list -A
helm template ./my-app -f values-prod.yaml | kubectl diff -f -

helm repo add bitnami https://charts.bitnami.com/bitnami
helm install pg bitnami/postgresql -f pg-values.yaml

Kustomize — без шаблонов, оверлеи

base/
├── kustomization.yaml
├── deployment.yaml
└── service.yaml
overlays/
├── prod/
│   ├── kustomization.yaml
│   ├── replicas-patch.yaml
│   └── ingress.yaml
└── staging/...

# overlays/prod/kustomization.yaml
resources: [../../base]
patches:
  - path: replicas-patch.yaml
images:
  - name: app
    newTag: v1.2.3

# Применить
kubectl apply -k overlays/prod
kubectl kustomize overlays/prod  | kubectl diff -f -

Helm vs Kustomize

  • Helm — для общих компонентов (БД, Prometheus). Гибкие values, sharing через repo.
  • Kustomize — для своих приложений с overlay-моделью. Никаких шаблонов, плоский YAML — легче читать в PR.
  • Часто комбинируют: Helm для общих чартов, Kustomize для управления окружениями.

// 15Operators и CRD

CRD — Custom Resource Definition

Расширяет API кластера новыми kind'ами: Certificate, Workflow, KafkaTopic. После apply'а CRD появляется новый kubectl get certificates.

Operator

Controller-loop, который следит за custom resource и приводит реальность в соответствие. Паттерн «expert as code»:

  • cert-manager — выпуск TLS-сертификатов.
  • Prometheus Operator — управление Prometheus/Alertmanager через CR (ServiceMonitor, PrometheusRule).
  • PostgreSQL/MySQL operators (Zalando, Crunchy, Percona) — БД с автомейнтенансом.
  • ArgoCD / FluxCD — GitOps.
  • Strimzi — Kafka.

GitOps (ArgoCD)

Главная идея: git — источник истины. ArgoCD следит за репозиторием и применяет манифесты в кластер; дрейф между git и кластером — детектится и подсвечивается.

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata: { name: web, namespace: argocd }
spec:
  project: default
  source:
    repoURL: https://github.com/org/k8s-manifests
    path: apps/web/overlays/prod
    targetRevision: main
  destination:
    server: https://kubernetes.default.svc
    namespace: prod
  syncPolicy:
    automated: { prune: true, selfHeal: true }

// 16Диагностика и kubectl recipes

Базовые recipes

kubectl get pods -A
kubectl get pods -n prod -o wide
kubectl get pods -l app=web --watch
kubectl describe pod web-xyz
kubectl logs web-xyz -c app --tail=100 -f
kubectl logs web-xyz --previous              # после рестарта
kubectl exec -it web-xyz -c app -- sh
kubectl port-forward svc/web 8080:80
kubectl cp web-xyz:/var/log/app.log ./app.log
kubectl top pod -n prod
kubectl top nodes
kubectl events --types=Warning --sort-by=lastTimestamp

Диагностика «Pod не стартует»

  1. kubectl describe pod X → секция Events.
  2. Pending: scheduling fail → проверьте resources, taints, affinity, PVC.
  3. ImagePullBackOff: тег не существует / imagePullSecret отсутствует.
  4. CrashLoopBackOff: kubectl logs X --previous.
  5. OOMKilled: kubectl describe pod → State.Terminated.Reason. Поднять limits.
  6. ErrImagePull / ImagePullErr: проблема с registry.
  7. Init container не проходит: kubectl logs X -c init-name.

Диагностика «не приходит трафик»

  1. Service: kubectl get endpoints web — есть IP?
  2. Если нет — selector не совпадает с labels Pod, или Pod не Ready.
  3. Из другого Pod: kubectl run -it --rm test --image=curlimages/curl -- sh → curl web:80.
  4. Ingress: kubectl describe ingress → events; kubectl logs -n ingress-nginx.
  5. NetworkPolicy: проверьте, что нет deny от/к этому Pod.
  6. DNS: kubectl exec -it X -- nslookup web.

Полезные инструменты

  • k9s — TUI для кластера.
  • stern — multi-pod logs tail.
  • kubectx / kubens — переключение контекстов и namespaces.
  • kubectl-tree — иерархия ресурсов.
  • kubectl debug — ephemeral container (1.25+) для распухших prod-Pod'ов.

// 17Prod чек-лист

Pod / Deployment

  • requests и memory limit заданы.
  • liveness, readiness, startup probes.
  • image с конкретным тегом (не latest), лучше digest.
  • securityContext: runAsNonRoot: true, readOnlyRootFilesystem: true.
  • imagePullPolicy: IfNotPresent (default для именованного тега).
  • Anti-affinity для разнесения реплик по нодам.
  • PodDisruptionBudget.
  • terminationGracePeriodSeconds + graceful shutdown в приложении.

Безопасность

  • RBAC по принципу least privilege.
  • NetworkPolicy «default deny» + точечные allow.
  • Pod Security Standards (restricted в проде).
  • Сканирование образов (Trivy в CI).
  • Шифрование etcd at-rest.
  • Audit log включён.
  • Секреты не в манифестах git (External Secrets / SealedSecrets).

Observability

  • Prometheus + node_exporter + kube-state-metrics + cAdvisor.
  • Loki + Promtail для логов.
  • Алерты на: under-replicated deployments, OOMKilled, CrashLoopBackOff, PVC fill, node NotReady, certificate expiry.
  • Дашборды: cluster overview, namespace usage, top pods.

Resilience

  • Бэкап etcd (snapshot ежедневно, тестируйте restore).
  • Velero для бэкапа объектов K8s + PVC.
  • Multi-AZ ноды, anti-affinity по zone.
  • Cluster autoscaler + Karpenter.
  • Регулярные kube-апгрейды (отстаёте максимум на 1 minor).
