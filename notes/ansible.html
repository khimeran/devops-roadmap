// 01Что такое Ansible и зачем

Ansible — система управления конфигурациями. Один YAML описывает желаемое состояние N серверов; ansible-playbook приводит их к этому состоянию.

Главное отличие от Bash-скриптов — идемпотентность: запустите playbook 1 или 100 раз — итог одинаковый. Модули проверяют текущее состояние и меняют только то, что отличается.

Что делает в реальной жизни

  • Подготавливает голую VM (пакеты, пользователи, sudoers, SSH-ключи).
  • Деплоит приложения (pull кода, перезапуск systemd, миграции БД).
  • Раскатывает конфиги (nginx, postgres, prometheus) с шаблонами Jinja2.
  • Дёргает API: AWS, Hetzner, Yandex Cloud, Proxmox.
  • Оркестрирует upgrade rolling-стратегией (5 серверов за раз).

Когда не Ansible

  • Provisioning инфры — лучше Terraform (state, plan).
  • Контейнеры — Kubernetes/Helm.
  • Имиджи — Packer (Ansible как provisioner внутри Packer — норм).
  • Сотни тысяч хостов — push-модель упирается в SSH-пул; SaltStack/Puppet масштабируются лучше.

// 02Архитектура: agentless, push-модель

Ansible не требует агентов. На целевом хосте нужны только:

  • SSH-доступ (Linux/macOS) или WinRM/PSRP (Windows).
  • Python 3 — для исполнения модулей.

Как выполняется task

  1. Control node читает playbook и inventory.
  2. Открывает SSH-сессию (с ControlPersist для скорости).
  3. Сериализует модуль в Python-скрипт.
  4. Копирует в ~/.ansible/tmp/ на целевом хосте.
  5. Запускает с нужными аргументами, забирает JSON-результат.
  6. Удаляет временные файлы.

// 04Inventory: hosts, groups

Inventory — список хостов и групп. Поддерживает форматы INI и YAML. YAML — современный стандарт.

# inventories/production/hosts.yml
all:
  vars:
    ansible_user: deploy
    ansible_ssh_private_key_file: ~/.ssh/id_ed25519
  children:
    webservers:
      hosts:
        web-01:
          ansible_host: 10.0.1.10
        web-02:
          ansible_host: 10.0.1.11
    databases:
      hosts:
        db-01:
          ansible_host: 10.0.2.10
          postgres_version: "16"
    production:
      children: [webservers, databases]

Patterns: как выбрать хосты

ansible all -m ping
ansible webservers -m ping
ansible 'webservers:&production' -m ping     # пересечение
ansible 'webservers:!web-01' -m ping         # исключение
ansible 'web-*' -m ping                      # glob
ansible '~web-[0-9]+' -m ping               # regex

Динамический inventory

Inventory может быть скриптом, который возвращает JSON. Плагины умеют тянуть хосты из AWS/GCP/Hetzner Cloud/Proxmox/Terraform state:

# inventories/aws_ec2.yml
plugin: amazon.aws.aws_ec2
regions: [us-east-1]
filters:
  tag:Environment: production
keyed_groups:
  - prefix: tag
    key: tags.Role
// Совет
Никогда не редактируйте inventory руками для cloud-хостов — это рассинхрон с реальностью. Используйте dynamic inventory или генерируйте из Terraform output.

// 05Ad-hoc команды

Запуск одного модуля без playbook — для проверок и быстрых операций:

# Проверка живости
ansible all -m ping

# Произвольная shell-команда
ansible webservers -m shell -a "uptime"
ansible webservers -m command -a "df -h"          # безопаснее, без shell

# Установка пакета
ansible webservers -m apt -a "name=nginx state=present update_cache=yes" --become

# Копирование файла
ansible webservers -m copy -a "src=./nginx.conf dest=/etc/nginx/nginx.conf" --become

# Перезапуск сервиса
ansible webservers -m service -a "name=nginx state=restarted" --become

# Сбор фактов
ansible web-01 -m setup
ansible web-01 -m setup -a "filter=ansible_distribution*"

shell vs command vs raw

  • command — без shell. Нет pipes, redirects, переменных. Безопаснее.
  • shell — есть pipes и переменные. Использовать только когда нужно.
  • raw — без Python на target. Для бутстрапа (установка Python на голую BSD/embedded).
// Антипаттерн
ansible all -m shell -a "apt install nginx" — не используйте shell для того, что есть в нативных модулях (apt, service, user). Нативный модуль идемпотентен, shell — нет.

// 06Playbooks и структура

Playbook — список plays. Play — сценарий применения tasks к группе хостов.

# playbooks/site.yml
- name: Configure webservers
  hosts: webservers
  become: true
  gather_facts: true
  vars:
    nginx_port: 80

  pre_tasks:
    - name: Update apt cache
      apt: { update_cache: yes, cache_valid_time: 3600 }

  roles:
    - common
    - nginx

  tasks:
    - name: Deploy application
      git:
        repo: "https://github.com/org/app.git"
        dest: /opt/app
        version: main
      notify: restart app

  post_tasks:
    - name: Verify health
      uri:
        url: "http://localhost:{{ nginx_port }}/health"
        status_code: 200

  handlers:
    - name: restart app
      service: { name: app, state: restarted }

Порядок выполнения внутри play

  1. gather_facts (если не false).
  2. pre_tasks.
  3. roles: их tasks/main.yml.
  4. tasks.
  5. post_tasks.
  6. handlers — в конце play, если были notify.

Запуск

ansible-playbook site.yml
ansible-playbook site.yml --check                # dry-run
ansible-playbook site.yml --diff                 # показать изменения
ansible-playbook site.yml -l web-01              # только этот хост
ansible-playbook site.yml --tags nginx           # только эти tag
ansible-playbook site.yml --start-at-task "Deploy"
ansible-playbook site.yml -vvv                   # verbose

// 07Tasks и модули

Task — одна операция: запуск модуля с параметрами.

- name: Create user
  ansible.builtin.user:
    name: deploy
    groups: sudo
    shell: /bin/bash
    state: present
  become: true
  tags: [users]
  register: user_result

FQCN (Fully Qualified Collection Name)

Современный стиль: ansible.builtin.apt вместо apt. Это явно указывает collection и избегает конфликтов имён. Все новые playbook'ы пишите с FQCN.

register и debug

- name: Get kernel version
  command: uname -r
  register: kernel
  changed_when: false

- debug:
    var: kernel.stdout

- debug:
    msg: "Kernel is {{ kernel.stdout }}"

// 08Handlers и notify

Handler — task, который запускается только если уведомлён, и только один раз за play (в конце).

tasks:
  - name: Render nginx config
    template:
      src: nginx.conf.j2
      dest: /etc/nginx/nginx.conf
    notify: reload nginx

  - name: Render sites-enabled
    template:
      src: site.j2
      dest: /etc/nginx/sites-enabled/{{ item }}.conf
    loop: [api, web]
    notify: reload nginx

handlers:
  - name: reload nginx
    service:
      name: nginx
      state: reloaded

Если оба template-таска поменяли файлы — nginx reload произойдёт один раз. Если не поменяли — handler не сработает.

listen — несколько handlers на один сигнал

tasks:
  - template: { src: app.conf.j2, dest: /etc/app/conf }
    notify: "app config changed"

handlers:
  - name: restart app
    service: { name: app, state: restarted }
    listen: "app config changed"
  - name: flush cache
    command: appctl cache:flush
    listen: "app config changed"

force_handlers

По умолчанию handlers не запускаются, если в play произошла ошибка. force_handlers: true или --force-handlers заставляет запустить handlers всех успешных tasks.

// Best practice
Имя handler пишите в нижнем регистре и описательно (reload nginx, не RestartNginx) — оно используется как идентификатор в notify:.

// 09Variables: уровни и приоритет

Переменные можно объявить в 22 местах. Приоритет (низший → высший):

  1. role defaults (roles/X/defaults/main.yml)
  2. inventory file vars
  3. inventory group_vars/all
  4. playbook group_vars/all
  5. inventory group_vars/*
  6. playbook group_vars/*
  7. inventory host_vars/*
  8. playbook host_vars/*
  9. host facts (gathered)
  10. play vars
  11. play vars_files
  12. role vars (roles/X/vars/main.yml)
  13. block vars
  14. task vars
  15. set_fact
  16. extra vars (-e) — побеждают всех

Практически важно знать

  • defaults — для роли всегда заводите defaults, иначе пользователи будут падать.
  • group_vars / host_vars — основной канал для пользовательских значений.
  • vars в роли — для константных значений, не предполагающих переопределения.
  • set_fact — runtime-вычисление, остаётся до конца play.
# group_vars/webservers.yml
nginx_port: 8080
nginx_workers: "{{ ansible_processor_vcpus | default(2) }}"

Lazy evaluation

Все Jinja2-выражения вычисляются в момент использования, а не объявления. Поэтому можно ссылаться на переменные, которые будут определены позже:

app_url: "http://{{ inventory_hostname }}:{{ nginx_port }}"

// 10Jinja2: шаблоны и фильтры

Ansible использует Jinja2 для рендеринга шаблонов (модуль template) и для выражений в YAML ({{ }}, {% %}).

Шаблон: templates/nginx.conf.j2

worker_processes {{ nginx_workers }};

events {
    worker_connections {{ nginx_worker_connections | default(1024) }};
}

http {
    {% for upstream in upstreams %}
    upstream {{ upstream.name }} {
        {% for backend in upstream.backends %}
        server {{ backend.host }}:{{ backend.port }};
        {% endfor %}
    }
    {% endfor %}
}

// 11Facts и gather_facts

Facts — переменные, собранные с целевого хоста: дистрибутив, ядро, сеть, диски, CPU, память. Доступны как ansible_*.

- debug:
    msg: |
      OS: {{ ansible_distribution }} {{ ansible_distribution_version }}
      Kernel: {{ ansible_kernel }}
      IP: {{ ansible_default_ipv4.address }}
      CPU: {{ ansible_processor_vcpus }} cores
      RAM: {{ (ansible_memtotal_mb / 1024) | round(1) }} GB

Часто используемые facts

  • ansible_distribution, ansible_distribution_major_version
  • ansible_os_family (Debian / RedHat / Suse)
  • ansible_architecture (x86_64, aarch64)
  • ansible_default_ipv4.address / ansible_all_ipv4_addresses
  • ansible_hostname / ansible_fqdn / inventory_hostname
  • ansible_processor_vcpus, ansible_memtotal_mb
  • ansible_mounts (список диск-mount'ов)
  • ansible_date_time.iso8601

Кэширование facts

# ansible.cfg
[defaults]
fact_caching = jsonfile
fact_caching_connection = /tmp/ansible_facts
fact_caching_timeout = 7200

Полезно, если запускаете playbook на 100+ хостов часто. Альтернативный backend — redis.

Custom facts

Положите исполняемый файл в /etc/ansible/facts.d/myapp.fact на target — он попадёт в ansible_local.myapp.

// 12Условия, циклы, blocks

when

- name: Install nginx on Debian
  apt: { name: nginx, state: present }
  when: ansible_os_family == "Debian"

- name: Skip on staging
  command: ./run_prod_only.sh
  when:
    - environment == "production"
    - inventory_hostname not in groups['maintenance']

loop

- name: Install several packages
  apt: { name: "{{ item }}", state: present }
  loop: [nginx, git, curl, vim]

- name: Create users
  user:
    name: "{{ item.name }}"
    groups: "{{ item.groups | default('users') }}"
  loop:
    - { name: alice, groups: sudo }
    - { name: bob }

- name: With index
  debug: { msg: "{{ ansible_loop.index }}: {{ item }}" }
  loop: [a, b, c]
  loop_control:
    extended: true

block / rescue / always

- block:
    - name: Try risky operation
      command: ./migrate.sh
  rescue:
    - name: Rollback on failure
      command: ./rollback.sh
    - fail:
        msg: "Migration failed and rolled back"
  always:
    - name: Clean tmp
      file: { path: /tmp/migration, state: absent }

changed_when / failed_when

Иногда command/shell возвращают 0, но что-то реально поменялось — или наоборот. Управляйте семантикой явно:

- shell: my-tool status
  register: r
  changed_when: "'pending'" in r.stdout
  failed_when: r.rc != 0 and "'no changes'" not in r.stderr

// 13Roles: структура и переиспользование

Role — стандартизированная упаковка tasks/handlers/templates/vars для одной «штуки» (nginx, postgres, kubernetes-node).

roles/nginx/
├── defaults/main.yml      # дефолты, низкий приоритет
├── vars/main.yml          # константы роли
├── tasks/main.yml         # точка входа
├── handlers/main.yml
├── templates/
│   └── nginx.conf.j2
├── files/
│   └── ssl-cert.pem
├── meta/main.yml          # зависимости, поддерж. ОС
└── README.md

tasks/main.yml — точка входа

- name: Install nginx
  apt: { name: nginx, state: present }

- name: Configure
  template:
    src: nginx.conf.j2
    dest: /etc/nginx/nginx.conf
  notify: reload nginx

- name: Ensure service
  service: { name: nginx, state: started, enabled: true }

Использование

- hosts: webservers
  become: true
  roles:
    - common
    - role: nginx
      vars:
        nginx_port: 8080
    - { role: monitoring, tags: monitoring }

  tasks:
    - name: Include role mid-play
      import_role:                  # статический include
        name: backup
    - include_role:                 # динамический
        name: deploy
        tasks_from: rollback.yml

import_role vs include_role

  • import_role — резолвится в начале play, нельзя использовать переменные времени выполнения.
  • include_role — резолвится в runtime, можно в loop, with conditional на runtime-vars.
// Принцип
Одна роль = одна вещь. Если в роли больше 10 файлов в tasks/ — пора разделить. Hi-cohesion, low-coupling работает и в Ansible.

// 14Ansible Vault — секреты

Vault шифрует файлы AES-256 паролем. В git коммитятся уже зашифрованные.

# Создать зашифрованный файл
ansible-vault create group_vars/production/secrets.yml

# Редактировать
ansible-vault edit group_vars/production/secrets.yml

# Зашифровать существующий
ansible-vault encrypt group_vars/production/secrets.yml

# Расшифровать (обычно не нужно)
ansible-vault decrypt group_vars/production/secrets.yml

# Сменить пароль
ansible-vault rekey group_vars/production/secrets.yml

# Зашифровать одно значение (inline)
ansible-vault encrypt_string 's3cr3t' --name 'db_password'

Запуск с vault

# Запрос пароля
ansible-playbook site.yml --ask-vault-pass

# Пароль из файла
ansible-playbook site.yml --vault-password-file ~/.vault_pass

# Скрипт-провайдер пароля (можно тянуть из 1Password/Vault)
ansible-playbook site.yml --vault-password-file ./vault_pass.sh

Несколько vault'ов (vault-id)

ansible-vault encrypt --vault-id prod@~/.vp_prod file.yml
ansible-playbook site.yml \
  --vault-id prod@~/.vp_prod \
  --vault-id dev@~/.vp_dev
// Альтернативы
Vault удобен для маленьких команд. Для больших — HashiCorp Vault / AWS Secrets Manager / SOPS (Mozilla). Подключаются через lookup-плагины: {{ lookup('community.hashi_vault.hashi_vault', 'secret=...') }}.

// 15Collections и Galaxy

С Ansible 2.10 модули вынесены в collections. Базовое (ansible.builtin.*) поставляется с дистрибутивом; всё остальное ставится отдельно.

# Установить collection
ansible-galaxy collection install community.docker
ansible-galaxy collection install community.postgresql

# Из requirements.yml
ansible-galaxy collection install -r requirements.yml

# requirements.yml
collections:
  - name: community.docker
    version: ">=3.4.0"
  - name: community.postgresql
  - name: ansible.posix

roles:
  - name: geerlingguy.docker
    version: "7.0.0"
  - src: https://github.com/org/role-x.git
    scm: git
    version: main
    name: role_x

Использование

- community.docker.docker_container:
    name: app
    image: nginx:1.27
    state: started
    ports: ["80:80"]

В тяжёлой инфре имеет смысл написать свою collection: общие роли + плагины + модули, версионированные и переиспользуемые между проектами.

// 16Идемпотентность и check mode

Идемпотентность = многократный запуск ≡ однократному. Это главное обещание Ansible.

Гарантирует или нарушает?

  • Гарантирует: нативные модули (apt, file, service, user, template).
  • Нарушает: command, shell, script — нужно явно указать creates:, removes: или changed_when:.
# Идемпотентность через creates
- name: Compile dependencies
  command: ./configure && make
  args:
    chdir: /opt/src
    creates: /opt/src/.compiled        # не запускать, если файл есть

# Через changed_when
- shell: my-tool reload
  register: r
  changed_when: "'reloaded'" in r.stdout

Check mode (dry-run)

ansible-playbook site.yml --check --diff

Показывает, что было бы изменено. Часть модулей не поддерживает check mode полностью (например, command) — они скипаются с warning.

idempotency tests в CI

Идиоматическая проверка: второй прогон playbook должен показать 0 changed. Часто запихивают в Molecule (тест-фреймворк для ролей).

// Правило
Если playbook на втором прогоне показывает changed: N>0 — где-то идемпотентность сломана. Чаще всего это shell-команда без changed_when или template, который рендерит timestamp.

Ограничение хостов

# Один хост
ansible-playbook site.yml -l web-01

# Несколько групп
ansible-playbook site.yml -l 'webservers:databases'

# Исключить
ansible-playbook site.yml -l 'all:!staging'

Tags

- name: Install
  apt: { name: nginx }
  tags: [install, nginx]

- name: Configure
  template: { src: x.j2, dest: /etc/nginx.conf }
  tags: [config, nginx]

# Запуск
ansible-playbook site.yml --tags config
ansible-playbook site.yml --skip-tags install
ansible-playbook site.yml --list-tags

serial — rolling update

- hosts: webservers
  serial: 2                # по 2 хоста за раз
  max_fail_percentage: 10  # если >10% упало — стоп всего play
  tasks:
    - name: Take out of LB
      command: lb-remove {{ inventory_hostname }}
      delegate_to: lb-01
    - name: Deploy
      ...
    - name: Health check
      uri: { url: "http://{{ inventory_hostname }}/health" }
    - name: Put back to LB
      command: lb-add {{ inventory_hostname }}
      delegate_to: lb-01

serial можно задавать как список: [1, 5, "30%"] — сначала 1 хост (канарейка), потом 5, потом 30% от оставшихся.

strategy

  • linear (default) — все хосты делают task 1, потом все task 2.
  • free — каждый хост летит вперёд независимо. Быстрее, но handlers и порядок странные.
  • host_pinned — обработка хоста до конца перед следующим (используется с serial).

delegate_to / run_once

- name: Run only on one host
  command: ./global-action.sh
  run_once: true
  delegate_to: localhost

// 18Best practices и cheatsheet

10 правил продакшен-Ansible

  1. Inventory отдельный для prod/staging. Никогда не передавайте окружение через extra-vars.
  2. Vault для секретов всегда; пароль — не в репо.
  3. Имя у каждого task. Без имени task — кошмар для логов.
  4. FQCN модулей (ansible.builtin.apt), не короткие имена.
  5. Идемпотентность: запускайте playbook дважды в CI — второй раз 0 changed.
  6. Не используйте shell там, где есть нативный модуль.
  7. Tags на каждом task: позволяет точечно прогонять части playbook.
  8. Roles — для всего больше 5 tasks.
  9. Linter: ansible-lint + yamllint в pre-commit.
  10. Тесты ролей в Molecule (минимум — create + converge + idempotence).
