// 01Что такое Docker и зачем он нужен

Docker — платформа для запуска приложений в контейнерах. Контейнер — это изолированный процесс на хосте, упакованный вместе со всеми зависимостями (библиотеки, конфиги, бинарь). Тот же образ запускается одинаково на ноутбуке разработчика, в CI и в проде.

Контейнер vs Виртуальная Машина

ПараметрVMКонтейнер
ЯдроСвоё (полная ОС)Общее ядро хоста
РазмерГБМБ–десятки МБ
СтартМинутыМиллисекунды
ИзоляцияПолная (гипервизор)Namespaces + cgroups
Накладные расходыВысокиеМинимальные

Что под капотом

  • Linux namespaces — изоляция: PID (свои процессы), NET (своя сеть), MNT (свои маунты), UTS (своё имя хоста), IPC, USER.
  • cgroups — ограничения ресурсов: CPU, RAM, IO, сеть.
  • OverlayFS — слоистая ФС: образ только для чтения + writable-слой контейнера сверху.

Зачем DevOps

  • «Работает у меня» — больше не оправдание. Один артефакт (образ) для всех окружений.
  • Стандарт упаковки. Kubernetes, ECS, Cloud Run — всё запускает контейнеры.
  • Микросервисы. 10 сервисов = 10 образов = 10 независимых деплоев.
  • CI как контейнеры. Каждый шаг GitHub Actions / GitLab CI запускается в контейнере.

// 02Архитектура: daemon, client, registry

Компоненты

  • Docker daemon (dockerd) — фоновый процесс, который реально создаёт контейнеры, образы, сети. Слушает на Unix-сокете /var/run/docker.sock.
  • Docker CLI (docker) — клиент, отправляющий REST-запросы демону.
  • containerd — низкоуровневый runtime, которым пользуется daemon. Управляет жизненным циклом контейнеров.
  • runc — ещё ниже: непосредственно запускает процесс с правильными namespaces/cgroups.
  • Registry — хранилище образов (Docker Hub, GHCR, GitLab Registry, ECR, приватный Harbor).

Жизненный цикл

# Образ → контейнер → процесс
docker pull nginx:1.27          # 1. вытянули образ из registry
docker create --name web nginx:1.27   # 2. создали контейнер (готов, но не запущен)
docker start web              # 3. запустили
docker stop web               # 4. SIGTERM, через 10с SIGKILL
docker rm web                 # 5. удалили

# run = create + start одной командой
docker run --name web -d nginx:1.27

Образ vs контейнер

  • Образ — снапшот файловой системы + метаданные (команда запуска, env, expose). Неизменяемый, состоит из слоёв.
  • Контейнер — работающий экземпляр образа. Имеет writable-слой поверх образа.
  • Из одного образа можно поднять N контейнеров.

// 03Базовые команды CLI

Запуск

docker run -d \                       # -d = detached, в фоне
  --name nginx \                       # имя контейнера
  -p 8080:80 \                         # хост:контейнер
  -v $(pwd)/html:/usr/share/nginx/html \  # bind mount
  -e ENV=prod \                        # переменная окружения
  --restart unless-stopped \           # перезапуск
  --memory 256m --cpus 0.5 \           # лимиты
  nginx:1.27

Управление

docker ps                       # запущенные
docker ps -a                    # все, включая остановленные
docker stop <id|name>
docker start <id|name>
docker restart <id|name>
docker rm -f <id|name>          # -f = принудительно
docker kill <id|name>            # SIGKILL немедленно

Заглянуть внутрь

docker logs -f --tail 100 <name>
docker exec -it <name> bash         # шелл внутри
docker exec <name> ps aux           # одна команда
docker inspect <name>               # JSON со всем
docker stats                        # live CPU/mem/io
docker top <name>                   # процессы в контейнере
docker port <name>                  # маппинги портов

Образы

docker images                       # локальные
docker pull alpine:3.20
docker rmi alpine:3.20
docker tag myapp:latest registry.example.com/myapp:v1.2
docker push registry.example.com/myapp:v1.2
docker history nginx:1.27              # слои образа

Очистка

docker system df                     # сколько занято
docker system prune -a --volumes      # удалить ВСЁ неиспользуемое
docker image prune
docker container prune
docker volume prune
// Ловушка
docker system prune -a удаляет все образы, которые сейчас не используются контейнером. После релиза в проде можно потерять старую версию — будь аккуратен с -a на live-серверах.

// 04Dockerfile — все инструкции

Dockerfile — рецепт сборки образа. Каждая инструкция создаёт слой. Слои кешируются — это критично для скорости пересборки.

FROM — базовый образ

FROM python:3.12-slim          # slim = минимальный debian
FROM alpine:3.20                # 5MB, musl libc
FROM scratch                    # пусто, только бинарь — для Go
FROM ubuntu:24.04 AS builder    # именованный stage для multi-stage

RUN — выполнить во время сборки

# Плохо: 3 слоя, мусор apt-cache в образе
RUN apt-get update
RUN apt-get install -y curl jq
RUN rm -rf /var/lib/apt/lists/*

# Хорошо: один слой, чисто
RUN apt-get update \
 && apt-get install -y --no-install-recommends curl jq \
 && rm -rf /var/lib/apt/lists/*

COPY vs ADD

COPY requirements.txt /app/    # дефолт — простое копирование
COPY --chown=app:app . /app
COPY --from=builder /out/app /usr/bin/app   # из другого stage

ADD archive.tar.gz /tmp/         # ADD умеет распаковывать и качать URL
# Правило: всегда COPY, ADD только если нужна распаковка tar

CMD vs ENTRYPOINT

CMDENTRYPOINT
Что задаётДефолтные аргументыСам бинарь
Переопределениеdocker run img <new-cmd>--entrypoint
Аргументы при runЗаменяют CMDДобавляются к ENTRYPOINT
# Шаблон: ENTRYPOINT — программа, CMD — её дефолтные аргументы
ENTRYPOINT ["python", "app.py"]
CMD ["--port", "8080"]

# docker run img            → python app.py --port 8080
# docker run img --port 9000 → python app.py --port 9000

Exec form (массив, JSON) — без shell. Shell form (CMD echo hi) — через /bin/sh -c. В проде используй exec form, иначе сигналы (SIGTERM) не доходят до приложения.

Остальные инструкции

WORKDIR /app                # рабочий каталог (создаёт если нет)
ENV NODE_ENV=production      # переменная окружения
ARG VERSION=1.0              # аргумент сборки --build-arg VERSION=...
EXPOSE 8080                  # документация — какой порт слушает (не публикует!)
VOLUME ["/data"]              # точка маунта (анонимный volume)
USER app:app                 # под каким UID запускать процесс
LABEL org.opencontainers.image.source=https://github.com/me/app
HEALTHCHECK --interval=10s --timeout=3s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
STOPSIGNAL SIGTERM

Минимальный production Dockerfile (Python)

FROM python:3.12-slim
WORKDIR /app
ENV PYTHONDONTWRITEBYTECODE=1 PYTHONUNBUFFERED=1

RUN useradd -r -u 1000 app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY --chown=app:app . .
USER app

EXPOSE 8080
HEALTHCHECK CMD curl -f http://localhost:8080/health || exit 1
ENTRYPOINT ["python"]
CMD ["app.py"]

// 05.dockerignore, ARG vs ENV

.dockerignore — без него образ будет огромным

При docker build весь контекст (каталог сборки) отправляется демону. Без .dockerignore туда уедут node_modules, .git, IDE-папки и логи. Это медленно и небезопасно (секреты).

# .dockerignore — синтаксис как .gitignore
.git
.gitignore
.env*
node_modules
__pycache__
*.pyc
.venv
.idea
.vscode
dist
build
*.log
README.md

ARG vs ENV — разница, на которой ломаются

ARGENV
Когда задаётсяПри сборке (--build-arg)В Dockerfile или -e
Доступен в RUNДаДа
В runtime контейнераНетДа
Виден в docker inspectНетДа
ARG APP_VERSION=dev
ENV APP_VERSION=$APP_VERSION   # «прокинуть» ARG в ENV если нужно в runtime
RUN echo "building $APP_VERSION"

# Сборка:
docker build --build-arg APP_VERSION=1.2.3 -t myapp .
// Опасность
Никогда не передавай секреты через --build-arg — они остаются в истории слоёв образа (docker history). Используй --secret (BuildKit, см. секцию 16).

// 06Слои и Layer cache

Каждая инструкция в Dockerfile создаёт слой — это снапшот diff'а файловой системы. Слои кешируются Docker'ом, и при повторной сборке используются заново, если их содержимое и предшественники не изменились.

Правило: меняющееся — в конец

# ПЛОХО — любое изменение кода ломает кеш установки зависимостей
FROM python:3.12-slim
COPY . /app
RUN pip install -r /app/requirements.txt   # пересборка при каждом коде

# ХОРОШО — зависимости в кеше, пока requirements.txt не менялся
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .                                     # код — в самом конце

Что инвалидирует кеш

  • Изменение содержимого файлов в COPY/ADD.
  • Изменение текста инструкции (даже пробел).
  • Любая инструкция выше — ломает кеш для всех ниже.
  • Смена ARG, на которые ссылается RUN.

Размер образа — простые правила

  • Базовый образ: -slim или alpine вместо полного.
  • Объединяй RUN: apt update && apt install && rm -rf /var/lib/apt/lists/*.
  • Не оставляй кэш пакетных менеджеров: --no-cache-dir (pip), --no-install-recommends (apt), --no-cache (apk).
  • Multi-stage build — самый сильный приём (см. секцию 7).
  • Не клади в образ исходники тестов, документацию, .git.

Анализ слоёв

docker history myapp:latest        # слои с размерами
docker image inspect myapp:latest    # детали

# Полезный сторонний инструмент: dive
# https://github.com/wagoodman/dive
dive myapp:latest

// 07Multi-stage builds и distroless

Идея: один Dockerfile, несколько FROM. В первом stage'е собираем — там есть компиляторы, dev-зависимости. В финальный stage копируем только готовый артефакт. В итоге образ становится в десятки раз меньше.

Пример: Go

# Stage 1: сборка
FROM golang:1.23-alpine AS builder
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -ldflags="-s -w" -o /out/app ./cmd/app

# Stage 2: только бинарь
FROM gcr.io/distroless/static-debian12
COPY --from=builder /out/app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]

Итог: образ ~15МБ вместо ~900МБ.

Пример: Node.js

FROM node:20-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev

FROM node:20-alpine AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
ENV NODE_ENV=production
COPY --from=deps /app/node_modules ./node_modules
COPY --from=build /app/dist ./dist
USER node
CMD ["node", "dist/server.js"]

Distroless / scratch / alpine — что выбрать

БазаРазмерКогда
scratch0 МБПолностью статический Go-бинарь
distroless/static~2 МБСтатический бинарь + CA-сертификаты
distroless/base~20 МББинарь с glibc
alpine~5 МБКогда нужен shell, BusyBox utils, apk
debian-slim~75 МББольшая совместимость, glibc
// Alpine vs glibc
Alpine использует musl libc — некоторые Python/Node-пакеты с native-extensions ломаются. Если видишь странные ошибки сегфолтов в Alpine — попробуй -slim.

// 08Volumes vs Bind mounts vs tmpfs

Контейнеры эфемерны — данные внутри теряются при docker rm. Чтобы данные пережили контейнер — нужно прокинуть с хоста или из named volume.

ТипГде хранитсяКогда
Named volume/var/lib/docker/volumes/<name>Данные приложения (БД, кеш)
Bind mountЛюбой путь хостаИсходники в dev, конфиги
tmpfsRAMСекреты, временные файлы

Named volume

docker volume create pgdata
docker run -d --name pg \
  -v pgdata:/var/lib/postgresql/data \
  postgres:16

docker volume ls
docker volume inspect pgdata
docker volume rm pgdata

Bind mount

# Для разработки — код хоста виден контейнеру live
docker run -d --name app \
  -v $(pwd):/app \
  -w /app \
  node:20-alpine npm run dev

# С флагом :ro контейнер не сможет писать
docker run -v /etc/nginx.conf:/etc/nginx/nginx.conf:ro nginx

tmpfs

docker run --tmpfs /tmp:size=64m,mode=1777 alpine
# /tmp в памяти, исчезнет при остановке. Полезно для секретов и кэшей.

Синтаксис --mount (современный)

docker run \
  --mount type=volume,source=pgdata,target=/var/lib/postgresql/data \
  --mount type=bind,source=$(pwd)/conf,target=/etc/conf,readonly \
  --mount type=tmpfs,target=/tmp,tmpfs-size=64m \
  postgres:16
// Гочча с permissions
Bind mount наследует UID/GID с хоста. Если контейнер запущен от USER 1000, а файлы на хосте принадлежат root — будет permission denied. Решения: chown на хосте, --user $(id -u):$(id -g) или init-скрипт в entrypoint.

// 09Docker networking

Драйверы сетей

ДрайверКогда использовать
bridgeДефолт. Контейнеры на одном хосте общаются друг с другом
hostКонтейнер использует сеть хоста напрямую (нет изоляции, нет NAT)
noneБез сети вообще — только loopback
overlayСеть между хостами в Swarm/Kubernetes
macvlanКонтейнер получает MAC и IP физической сети

Дефолтный bridge vs custom bridge

  • Дефолтный docker0 — все контейнеры без указания --network. Резолвинг по имени НЕ работает, только по IP.
  • Custom bridge — создаёшь сам, имена контейнеров автоматически резолвятся через встроенный DNS Docker'а. Всегда используй custom.
docker network create app-net

docker run -d --network app-net --name db postgres:16
docker run -d --network app-net --name web \
  -e DATABASE_URL="postgres://postgres@db:5432/app" \   # db резолвится в IP контейнера
  myapp

docker network ls
docker network inspect app-net
docker network connect app-net <container>       # добавить контейнер
docker network disconnect app-net <container>

Публикация портов

-p 8080:80           # все интерфейсы хоста
-p 127.0.0.1:8080:80 # только localhost (безопаснее!)
-P                   # опубликовать ВСЕ EXPOSE на случайные порты
// Безопасность
-p 8080:80 биндит на 0.0.0.0 — порт открыт всему миру. На dev-сервере с публичным IP это часто означает БД, торчащую наружу. Используй -p 127.0.0.1:8080:80 или фаервол.

// 10Docker Compose углублённо

Compose описывает мульти-контейнерное приложение в одном YAML. Это «package manager» твоего dev-окружения и удобный способ декларативно поднять стек на одном хосте.

Полный пример

# compose.yml (имя файла docker-compose.yml тоже работает)
services:
  web:
    build:
      context: .
      dockerfile: Dockerfile
      args:
        VERSION: ${VERSION:-dev}
    image: myapp:${VERSION:-dev}
    ports:
      - "127.0.0.1:8080:8080"
    environment:
      DATABASE_URL: postgres://app:secret@db:5432/app
      REDIS_URL: redis://cache:6379/0
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8080/health"]
      interval: 10s
      timeout: 3s
      retries: 3

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: app
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: app
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app"]
      interval: 5s

  cache:
    image: redis:7-alpine
    command: redis-server --maxmemory 256mb --maxmemory-policy allkeys-lru

volumes:
  pgdata:

Команды

docker compose up -d              # поднять в фоне
docker compose up --build         # пересобрать образы
docker compose down               # остановить и удалить
docker compose down -v            # + удалить volumes
docker compose ps
docker compose logs -f web
docker compose exec web bash
docker compose restart web
docker compose run --rm web pytest  # одноразовый запуск
docker compose pull                 # обновить образы из registry

Профили — необязательные сервисы

services:
  pgadmin:
    image: dpage/pgadmin4
    profiles: [debug]

# Запустится только если явно попросишь:
docker compose --profile debug up

Несколько compose-файлов

# compose.yml — базовое
# compose.dev.yml — оверрайды для dev
# compose.prod.yml — оверрайды для prod
docker compose -f compose.yml -f compose.dev.yml up

depends_on с healthcheck

Просто depends_on: [db] означает «db стартовала первой», но НЕ «db готова принимать запросы». Postgres стартует за 2 секунды, а потом ещё 5 секунд готовится. Чтобы дождаться готовности — комбинируй с healthcheck + condition: service_healthy.

// Compose v2
Новая команда — docker compose (без дефиса). Старый docker-compose (через дефис) — это python-версия v1, устарела. Файл называй compose.yml — это новый канон.

// 11Healthchecks и restart policies

HEALTHCHECK

# В Dockerfile
HEALTHCHECK --interval=10s \
            --timeout=3s \
            --start-period=20s \
            --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1
  • --interval — как часто проверять.
  • --timeout — сколько ждать ответа.
  • --start-period — grace-period после старта, провалы не считаются.
  • --retries — сколько провалов = unhealthy.

Статус: starting → healthy / unhealthy. Виден в docker ps и используется depends_on: condition: service_healthy.

Restart policies

--restart no              # дефолт — не перезапускать
--restart on-failure      # только при ненулевом exit
--restart on-failure:3    # не более 3 раз
--restart always          # всегда
--restart unless-stopped  # всегда, но НЕ после явного docker stop — рекомендуется в проде

Лимиты ресурсов

docker run \
  --memory 512m \         # hard limit, OOMKill при превышении
  --memory-reservation 256m \  # soft limit
  --cpus 1.5 \            # 1.5 ядра
  --pids-limit 100 \      # max процессов
  myapp

// 12Secrets и переменные окружения

Передача env

docker run -e DB_URL="postgres://..." myapp
docker run --env-file .env myapp           # из файла

# В compose
services:
  web:
    environment:
      DB_URL: postgres://app:${DB_PASS}@db:5432/app
    env_file:
      - .env

Что НЕЛЬЗЯ делать

  • ❌ Класть секреты в Dockerfile через ENV — они в слоях образа навсегда.
  • ❌ Передавать через --build-arg — остаются в docker history.
  • ❌ Коммитить .env с продовскими паролями.

BuildKit secret — для секретов в сборке

# Dockerfile
RUN --mount=type=secret,id=npmrc,target=/root/.npmrc \
    npm install

# Сборка
DOCKER_BUILDKIT=1 docker build \
  --secret id=npmrc,src=$HOME/.npmrc \
  -t myapp .

Docker secrets (Swarm) / Compose secrets

services:
  web:
    image: myapp
    secrets:
      - db_password

secrets:
  db_password:
    file: ./secrets/db_password.txt
# Внутри контейнера: /run/secrets/db_password

В Kubernetes секреты живут отдельно (объект Secret), монтируются как файлы или env. В Docker compose поддержка ограничена, но базовый use-case покрывает.

// 13Registry — push, pull, теги

Имя образа: registry/namespace/repo:tag

nginx:1.27                                # дефолт = docker.io/library/nginx:1.27
ghcr.io/kirill/myapp:v1.2                 # GitHub Container Registry
registry.gitlab.com/group/proj/web:latest # GitLab
123456.dkr.ecr.us-east-1.amazonaws.com/myapp:latest  # AWS ECR

Логин и push

docker login ghcr.io -u kirill -p $GITHUB_TOKEN
docker tag myapp:latest ghcr.io/kirill/myapp:v1.2
docker push ghcr.io/kirill/myapp:v1.2

Стратегии тегирования

  • SemVer: v1.2.3, v1.2, v1 — для библиотек/публичных образов.
  • Git SHA: sha-a1b2c3d — уникально, всегда можно соотнести с коммитом.
  • latest в проде — антипаттерн. Невозможно понять что развёрнуто.
  • Иммутабельность: один тег = один образ. Не пересобирай под старый тег.

// 14Сканирование уязвимостей

Образ может содержать сотни CVE — от устаревшего OpenSSL до зависимостей приложения. Сканеры читают манифест образа и сравнивают с базами уязвимостей.

Trivy — самый популярный

brew install trivy
apt install trivy

trivy image nginx:1.27                       # все уязвимости
trivy image --severity HIGH,CRITICAL myapp    # только серьёзные
trivy image --ignore-unfixed myapp            # без фиксов — игнорируем
trivy image --exit-code 1 --severity CRITICAL myapp  # упасть в CI

trivy fs .                                    # сканировать репозиторий (requirements.txt, package.json и т.д.)
trivy config Dockerfile                       # проверка Dockerfile на ошибки

Альтернативы

  • Grype — от Anchore, быстрый.
  • docker scout — встроен в Docker Desktop (docker scout cves myimg).
  • Snyk — коммерческий, хорош для SCA.
  • Скан в registry: GHCR / GitLab / Harbor умеют сканировать при push.

Как уменьшить площадь атаки

  • Минимальная база (distroless, alpine, slim).
  • Не запускать от root: USER 1000.
  • Регулярно пересобирать образы — даже если код не менялся, базовые образы получают патчи.
  • Закрепляй версии: nginx:1.27.0 вместо nginx:latest.
  • --read-only для root FS контейнера + tmpfs для нужных мест записи.

// 15Диагностика: OOMKilled, exit codes, logs

Exit codes

КодЗначение
0Успех
1Ошибка приложения
125Ошибка самого docker (например, не нашёл образ)
126CMD не может быть выполнен (нет прав, не исполняемый)
127Команда не найдена
137SIGKILL — OOM-killer убил из-за памяти
139SIGSEGV — segfault
143SIGTERM — нормальный docker stop
docker inspect <name> --format '{{.State.ExitCode}} {{.State.Error}}'
docker inspect <name> --format '{{.State.OOMKilled}}'

OOMKilled — что делать

  1. docker inspect → State.OOMKilled=true?
  2. Смотрим --memory лимит и реальное потребление: docker stats <name>.
  3. Если приложение жрёт больше лимита — поднимай лимит или ищи утечку.
  4. Для JVM/Node — настрой heap size с учётом лимита контейнера (-Xmx, NODE_OPTIONS=--max-old-space-size).
  5. dmesg | grep -i killed на хосте покажет действия OOM-killer.

Логи

docker logs -f --tail 200 <name>
docker logs --since 10m --until 5m <name>
docker logs --timestamps <name>

# Где лежат на хосте (json-file driver):
cat /var/lib/docker/containers/<id>/<id>-json.log

Совет: настрой log-driver и ротацию в /etc/docker/daemon.json — иначе один разговорчивый контейнер забивает диск:

{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3"
  }
}

Полезные приёмы

docker top <name>                          # процессы внутри

// 16BuildKit и buildx

BuildKit — современный движок сборки Docker. Параллелит независимые шаги, поддерживает кэш-маунты, secret-маунты, multi-arch. В Docker Desktop включён по умолчанию.

Кэш-маунты для зависимостей

# syntax=docker/dockerfile:1.7
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt
COPY . .

Кэш pip сохраняется между сборками — даже если кеш слоя инвалидирован.

buildx — multi-arch сборка

docker buildx create --use --name multi
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t ghcr.io/kirill/app:v1 \
  --push .

Удалённый кеш в registry

docker buildx build \
  --cache-from type=registry,ref=ghcr.io/kirill/app:buildcache \
  --cache-to   type=registry,ref=ghcr.io/kirill/app:buildcache,mode=max \
  -t ghcr.io/kirill/app:v1 \
  --push .

Используется в CI чтобы переиспользовать слои между запусками джобы (где локальный docker-кеш чистый).

// 17Best practices в продакшне

Dockerfile

  • Закрепляй версию базового образа: nginx:1.27.0, не nginx.
  • Сортируй инструкции от редко меняющихся к часто меняющимся.
  • Один RUN apt-get update && apt-get install && rm -rf /var/lib/apt/lists/*.
  • USER — не root. Создай юзера в Dockerfile.
  • Multi-stage — даже для языков без компиляции (нужно собрать ассеты, например).
  • HEALTHCHECK обязателен — иначе orchestrator не знает что приложение упало.
  • STOPSIGNAL — если приложение не понимает SIGTERM, укажи SIGINT.
  • .dockerignore исключает контекст: .git, node_modules, .env.
  • OCI labels: org.opencontainers.image.source — ссылка на репо.

Runtime

  • Всегда задавай лимиты: --memory, --cpus.
  • --read-only + tmpfs для нужных директорий записи.
  • --cap-drop=ALL и добавляй обратно только нужные capabilities.
  • --security-opt no-new-privileges.
  • Не маунть /var/run/docker.sock внутрь контейнера без острой необходимости (это даёт root на хосте).
  • Используй пользователя без UID 0: USER 1000.
  • restart: unless-stopped — не always.

CI/CD

  • Кэшируй слои через buildx --cache-from/--cache-to.
  • Тегируй git SHA + semver, никогда не пушь только latest.
  • Сканируй trivy image --exit-code 1 --severity CRITICAL в pipeline.
  • Подписывай образы (cosign).

12-factor

  • Конфигурация — через env, не файлами в образе.
  • Логи — в stdout/stderr, оркестратор сам разберётся.
  • Stateless — состояние в БД/Redis/Volume, не в локальной FS контейнера.
  • Один процесс на контейнер. Нет supervisor'а.
