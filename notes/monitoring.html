// 01Наблюдаемость: metrics, logs, traces

Три кита observability — три типа сигналов, каждый отвечает на свой вопрос:

СигналНа что отвечаетТипСтек (de-facto)
MetricsЧто происходит сейчас? Тренды?Числа во времениPrometheus → Grafana
LogsЧто именно произошло в строке X?Текст с timestampLoki / ELK / Vector
TracesПочему один запрос медленный?Цепочка spansTempo / Jaeger / OTel

Monitoring vs Observability

  • Monitoring — реагируешь на известные проблемы (CPU > 90%).
  • Observability — можешь ответить на новые, не предусмотренные заранее вопросы (почему именно эти 3% запросов из этого региона тупят?).

Четыре золотых сигнала (Google SRE)

  1. Latency — время ответа (отдельно успешных и упавших).
  2. Traffic — нагрузка (RPS).
  3. Errors — частота ошибок.
  4. Saturation — насколько «полная» система (CPU/RAM/IO).
// Принцип USE для ресурсов
На каждый ресурс смотрите Utilization, Saturation, Errors. Брендан Грегг.

// 02Prometheus: архитектура

Prometheus — TSDB (time-series DB) + scraper + alerting engine в одном бинаре. Pull-модель: сам ходит за метриками на endpoints.

Компоненты экосистемы

  • prometheus-server — scrape + storage + PromQL.
  • node_exporter — метрики хоста.
  • application — отдаёт /metrics сама (через клиентскую библиотеку).
  • pushgateway — для коротких job (cron).
  • alertmanager — отдельный сервис: routing, silences, нотификации.
  • grafana — UI поверх данных.

Поток данных

app:/metrics ──┐
node_exporter ─┼─► Prometheus ──► PromQL ──► Grafana
cAdvisor ──────┘       │
                       └─► rules ──► Alertmanager ──► Slack/Email/PagerDuty

Storage

Локальный TSDB на диске. Дефолтный retention — 15 дней. Для долгого хранения и федерации — Thanos / Cortex / Mimir (offload в S3, multi-tenant, downsampling).

ПараметрЗначение
--storage.tsdb.retention.time15d (default)
--storage.tsdb.path./data
--web.listen-address:9090
--web.enable-lifecyclePOST /-/reload без перезапуска
--web.enable-admin-apidelete series API

// 03Модель данных и labels

Каждая time-series — комбинация metric_name{label1="v1",label2="v2"}. Уникальная пара labels = отдельный ряд.

http_requests_total{method="GET", handler="/api/users", code="200"} 4521
http_requests_total{method="POST", handler="/api/users", code="201"} 89

4 типа метрик

ТипСемантикаПример
CounterРастёт монотонно, сбрасывается при рестартеhttp_requests_total
GaugeПроизвольное значение, растёт/падаетmemory_usage_bytes, queue_size
HistogramBucket'ы + sum + count → квантили вычисляютсяhttp_request_duration_seconds
SummaryКвантили считаются на клиенте, нельзя агрегироватьТо же, но реже используется

Соглашения по именованию

  • Единицы в конце: _bytes, _seconds, _total для counter.
  • Snake_case.
  • Префикс по подсистеме: node_, http_, db_.
  • Не кладите в label высоко-кардинальные значения (user_id, request_id) — взорвёт TSDB.
// Кардинальность убивает
Каждая уникальная комбинация labels — отдельный ряд. user_id в label на 1М юзеров = 1М рядов на каждую метрику. Через час Prometheus падает. Правило: labels — это размерность для агрегации (метод, код, инстанс), не идентификатор сущности.

// 04scrape_configs и discovery

# prometheus.yml
global:
  scrape_interval: 15s
  evaluation_interval: 15s
  external_labels:
    cluster: production

scrape_configs:
  - job_name: prometheus
    static_configs:
      - targets: ["localhost:9090"]

  - job_name: node
    static_configs:
      - targets: ["web-01:9100", "db-01:9100"]
        labels: { env: prod }

  - job_name: kubernetes-pods
    kubernetes_sd_configs:
      - role: pod
    relabel_configs:
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: "true"
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_port]
        target_label: __address__
        replacement: "$1:$2"

alerting:
  alertmanagers:
    - static_configs:
        - targets: ["alertmanager:9093"]

rule_files:
  - "rules/*.yml"

Service discovery

  • static_configs — для мелких/стабильных setup.
  • file_sd_configs — JSON-файл, обновляемый внешним процессом.
  • kubernetes_sd_configs — pods/services/endpoints/nodes/ingresses.
  • consul_sd / dns_sd / ec2_sd — динамические targets.

Relabeling — основа гибкости

Прежде чем targets попадут в TSDB, Prometheus прогоняет их через relabel_configs (фильтрация и перепись labels) и metric_relabel_configs (после scrape — фильтрация по имени метрики).

# Пример: убрать высоко-кардинальную метрику
metric_relabel_configs:
  - source_labels: [__name__]
    regex: "go_gc_duration_seconds.*"
    action: drop

Свой exporter

Используйте клиентскую библиотеку (Go/Python/Java/Rust) и отдавайте /metrics на встроенном порту:

reqs = Counter('app_requests_total', 'Total requests', ['method', 'code'])

start_http_server(8000)

// 06PromQL: основы

PromQL — язык запросов. Возвращает 4 типа результата: instant vector, range vector, scalar, string.

Селекторы

# Instant: значение каждого ряда сейчас
node_cpu_seconds_total

# Фильтр по label
node_cpu_seconds_total{mode="idle"}
node_cpu_seconds_total{mode!="idle"}
node_cpu_seconds_total{instance=~"web-.*"}
node_cpu_seconds_total{job=~"node|cadvisor"}

# Range: значения за окно
node_cpu_seconds_total[5m]

# Сдвиг во времени
node_cpu_seconds_total offset 1h

Арифметика и сравнения

# Свободное место в %
node_filesystem_avail_bytes / node_filesystem_size_bytes * 100

# Фильтр (бинарный оператор с bool — возвращает 0/1)
node_filesystem_avail_bytes / node_filesystem_size_bytes < 0.1

Группировка при бинарных операциях

# ignoring — игнорировать определённые labels
sum(rate(http_errors_total[5m])) by (service) /
sum(rate(http_requests_total[5m])) by (service)

# on() — сопоставлять только по этим labels
metric_a * on(instance) group_left(role) metric_b

// 07PromQL: rate, histogram, aggregations

rate / irate / increase

  • rate(x[5m]) — средняя скорость роста counter за 5 минут (per-second).
  • irate(x[5m]) — мгновенная (последние 2 точки). Дёрганный — для дашбордов avoid.
  • increase(x[5m]) — = rate * время. Сколько «штук» за 5 минут.
# RPS за 5 минут
sum(rate(http_requests_total[5m])) by (service)

# Ошибки в % за 5 минут
sum(rate(http_requests_total{code=~"5.."}[5m])) by (service)
  /
sum(rate(http_requests_total[5m])) by (service)

# CPU usage % (1 - idle)
100 * (1 - avg by (instance) (rate(node_cpu_seconds_total{mode="idle"}[5m])))

histogram_quantile

Histogram отдаёт bucket'ы (le="0.005", le="0.01"...). Квантили считаются на запросе:

# p95 latency
histogram_quantile(0.95,
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service))

Aggregation operators

sum, avg, min, max, count, stddev, quantile, topk, bottomk, group. Управление группировкой: by / without:

sum by (job) (rate(http_requests_total[5m]))
avg without (cpu) (rate(node_cpu_seconds_total[5m]))
topk(5, sum by (instance) (rate(node_cpu_seconds_total[5m])))

Полезные функции

  • deriv(), predict_linear(x[1h], 4*3600) — экстраполяция (диск кончится через ?).
  • changes(x[1h]) — сколько раз gauge менялся.
  • resets(x[1h]) — сколько раз counter сбрасывался.
  • absent(metric) / absent_over_time — алерт «метрики нет».
  • label_replace, label_join — манипуляция labels.
  • vector(0) — выдать пустоту как 0 (полезно в OR).
// Правило большого пальца
Окно в rate(x[5m]) должно быть ≥ 4× scrape_interval. При scrape 15s — окно минимум 1m, лучше 5m. Иначе rate теряет точность при пропущенных scrape'ах.

// 08Recording и alerting rules

Recording rules — кэширование тяжёлых запросов

# rules/recording.yml
groups:
  - name: http
    interval: 30s
    rules:
      - record: job:http_requests:rate5m
        expr: sum by (job) (rate(http_requests_total[5m]))

      - record: job:http_errors:rate5m
        expr: sum by (job) (rate(http_requests_total{code=~"5.."}[5m]))

Префиксы: level:metric:operation (Google convention).

Alerting rules

groups:
  - name: critical
    rules:
      - alert: HighErrorRate
        expr: job:http_errors:rate5m / job:http_requests:rate5m > 0.05
        for: 10m
        labels:
          severity: critical
          team: backend
        annotations:
          summary: "High error rate on {{ $labels.job }}"
          description: "{{ $value | humanizePercentage }} errors in last 5m"
          runbook: "https://wiki/runbooks/high-error-rate"

Жизненный цикл алерта

  • inactive — выражение false.
  • pending — expr=true, но for: ещё не истекло.
  • firing — отправляется в Alertmanager.

for: 10m — критично, чтобы не флапать по одиночному всплеску.

// 09Alertmanager: routing, silences

Отдельный сервис, получает алерты от Prometheus, занимается:

  • Группировкой — несколько одинаковых алертов = одно уведомление.
  • Routing — куда отправлять (Slack/Telegram/Email/PagerDuty/WebHook).
  • Suppression / Inhibit — при critical скрывать warning.
  • Silences — временное «молчание» по фильтру.
# alertmanager.yml
route:
  group_by: [alertname, cluster, service]
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  receiver: slack-default
  routes:
    - matchers: ['severity="critical"']
      receiver: pagerduty
      continue: true
    - matchers: ['team="data"']
      receiver: slack-data

inhibit_rules:
  - source_matchers: ['severity="critical"']
    target_matchers: ['severity="warning"']
    equal: [alertname, cluster]

receivers:
  - name: slack-default
    slack_configs:
      - api_url: "https://hooks.slack.com/..."
        channel: "#alerts"
        title: "{{ .CommonLabels.alertname }}"
        text: "{{ range .Alerts }}{{ .Annotations.summary }}\n{{ end }}"

  - name: pagerduty
    pagerduty_configs:
      - service_key: "..."
// Шаблоны
Алерты используют Go templates. {{ $labels.X }}, {{ $value }}, {{ .Annotations.summary }}. Часто пишут отдельный template-файл и подключают через templates:.

// 10Grafana: дашборды, переменные

UI поверх Prometheus, Loki, InfluxDB, MySQL и т.д. Дашборд — JSON, можно версионировать в git.

Datasource

Provisioning через файлы /etc/grafana/provisioning/datasources/*.yml:

apiVersion: 1
datasources:
  - name: Prometheus
    type: prometheus
    url: http://prometheus:9090
    isDefault: true
  - name: Loki
    type: loki
    url: http://loki:3100

Переменные дашборда

  • Query — список из метрики: label_values(node_cpu_seconds_total, instance).
  • Custom — статичный список.
  • Interval — для функций (используйте $__rate_interval!).
  • Datasource — выбор источника прямо в дашборде.
  • Multi-value + Include All — типичная связка.

В PromQL: rate(http_requests_total{instance=~"$instance"}[$__rate_interval]).

Типы панелей

  • Time series — основной.
  • Stat — одно большое число.
  • Gauge / Bar gauge — для % и порогов.
  • Table — для top-N.
  • Heatmap — для histogram-распределений.
  • Logs — для Loki.
  • Node graph — для трассировок.

// 11Loki + Promtail: логи

Loki — "Prometheus для логов". Хранит логи без полнотекстовой индексации; индексируются только labels. Дёшево, быстро на горячих labels.

Архитектура

app stdout ──► Promtail / Vector / Fluent Bit ──► Loki ──► Grafana
                       │
                       └─ читает /var/log/, контейнеры, journald

Promtail config (упрощённо)

server:
  http_listen_port: 9080

clients:
  - url: http://loki:3100/loki/api/v1/push

scrape_configs:
  - job_name: docker
    docker_sd_configs:
      - host: unix:///var/run/docker.sock
    relabel_configs:
      - source_labels: [__meta_docker_container_name]
        target_label: container

  - job_name: system
    static_configs:
      - targets: [localhost]
        labels:
          job: varlogs
          __path__: /var/log/*.log
// Кардинальность v.2
В labels Loki — те же правила, что у Prometheus: не суйте request_id, trace_id, user_id в labels. Они должны быть внутри сообщения лога, оттуда LogQL вытащит через | json или | regexp.

// 12LogQL

LogQL = «PromQL для логов». Селектор labels + фильтры + парсинг.

# Все логи nginx
{job="nginx"}

# С фильтром по подстроке
{job="nginx"} |= "ERROR"
{job="nginx"} != "healthcheck"
{job="nginx"} |~ "5\\d\\d"             # regex

# Парсинг JSON
{app="api"} | json | level="error" | line_format "{{.msg}}"

# logfmt
{app="api"} | logfmt | duration > 500ms

# Метрика из логов
sum by (path) (rate({app="api"} |~ "GET" [5m]))

Range функции (как в PromQL)

# Скорость 5xx
sum(rate({app="api"} |~ "status=5" [5m]))

# Топ ошибок
topk(10,
  sum by (path) (count_over_time({app="api"} |= "ERROR" [1h])))

На основе LogQL можно делать alerting rules (через Loki ruler) — например, «больше 100 ERROR за 5m — алерт».

// 13Tempo / OpenTelemetry: трассировки

Trace = цепочка spans. Каждый span — кусок работы (HTTP-запрос, SQL-вызов, gRPC-call), с trace_id и parent_id.

OpenTelemetry (OTel)

  • Стандарт CNCF. Заменяет проприетарные SDK Jaeger/Zipkin.
  • OTel SDK в приложении инструментирует код.
  • OTel Collector принимает (gRPC/HTTP), обрабатывает, отправляет в backend.
  • Backend — Tempo (Grafana) / Jaeger / Zipkin / коммерческие SaaS.

Tempo

Дешёвое хранилище traces в S3. Связан с логами через trace_id: в Grafana кликаешь на span → улетаешь в Loki с фильтром по trace_id. Это и есть correlation.

Auto vs manual инструментация

  • Auto — opentelemetry-instrumentation-* агенты автоматически оборачивают HTTP, БД, Redis, gRPC, Kafka и т.п. Минимум кода.
  • Manual — tracer.start_as_current_span("checkout") в бизнес-логике для важных операций.
// Sampling
Сохранять 100% traces — дорого. Стандарт: head-based (1–5% случайно) или tail-based (на коллекторе — оставлять только медленные/ошибочные). 1% — нормальная default.

// 14Zabbix: когда и зачем

Zabbix — старая, тяжёлая, но широко распространённая в РФ система. Push+pull, агент на хосте, БД (MySQL/Postgres), своя web-UI.

АлертыВстроенные triggersAlertmanager

В рос. компаниях с парком физических серверов + сетевым железом — Zabbix часто живёт уже 10 лет. Учите минимум: что такое item, trigger, template, host group, как настроить мониторинг через агент и SNMP.

Zabbix → Prometheus конверсия

Иногда параллельно живут оба: Zabbix для инфры, Prometheus для приложений и k8s. Это нормально на переходный период.

// 15SLI / SLO / Error budget

Без этого мониторинг превращается в шум. Sit down с продактом и сформулируйте, что значит «работает».

  • SLI (Service Level Indicator) — измеримая характеристика. Пример: доля запросов с latency < 300ms и кодом 2xx за период.
  • SLO (Service Level Objective) — цель: 99.9% запросов за месяц.
  • SLA (Service Level Agreement) — то же, но в договоре, с штрафами.
  • Error budget — допустимая доля «плохих» событий: 100% − 99.9% = 0.1% (~43 минуты/месяц).

SLO-based alerting

Алертить не на «CPU 90%», а на сжигание бюджета: «за последний час сжато > 5% месячного бюджета» → серьёзная проблема. Метод Multi-window multi-burn-rate (SRE Workbook).

# Пример алерта на быстрое сжигание бюджета
- alert: SLOBurnRateHigh
  expr: |
    (
      sum(rate(http_requests_total{code=~"5.."}[1h])) by (service)
      / sum(rate(http_requests_total[1h])) by (service)
    ) > (14.4 * 0.001)            # 14.4×budget rate для 99.9%
  for: 2m
  labels: { severity: critical }
// Принцип
Алерт должен значить «пора звонить человеку прямо сейчас». Если за неделю > 5 алертов оказались false positive — это не алерты, а спам. Чините правила, не игнорируйте.

// 17Антипаттерны и подводные камни

  1. Высоко-кардинальные labels — user_id / request_id / IP. TSDB взорвётся за часы.
  2. rate() на gauge — бессмысленно. rate() только для counter.
  3. Слишком короткий range в rate — rate(x[1m]) при scrape_interval=30s — половина запросов вернёт пусто.
  4. Алерт без for: — флапает на одиночные всплески.
  5. Алерт «CPU > 90%» — не симптом, а причина. Алертить нужно на симптом (latency, errors).
  6. Слишком много алертов — теряется доверие, появляется alert fatigue.
  7. Дашборд без legend и единиц — что значит линия 0.0042?
  8. Хранение всего навечно — без downsampling Thanos/Mimir растёт линейно. 15d retention default — норма.
  9. Нет smoke-теста для Prometheus — самой Prometheus тоже мониторьте (через второй Prometheus или Pushgateway heartbeat).
  10. Игнор runbook — каждый алерт должен иметь annotations.runbook со ссылкой на инструкцию что делать.

// 18Best practices и cheatsheet

Порядок внедрения с нуля

  1. Поднять Prometheus + Grafana + node_exporter.
  2. Импортировать дашборд 1860 (Node Exporter Full).
  3. Добавить cAdvisor (если есть Docker).
  4. Подключить blackbox_exporter для внешнего health-check.
  5. Инструментировать приложения (RED-метрики: Rate, Errors, Duration).
  6. Alertmanager + первые 3–5 алертов (latency, error rate, no metrics).
  7. Loki + Promtail.
  8. SLO для основных сервисов.

Минимальный набор алертов для VM

- alert: InstanceDown
  expr: up == 0
  for: 5m
  labels: { severity: critical }
  annotations: { summary: "{{ $labels.instance }} is down" }

- alert: HighDiskUsage
  expr: |
    100 * (1 - node_filesystem_avail_bytes{fstype!~"tmpfs|overlay"}
              / node_filesystem_size_bytes) > 85
  for: 10m
  labels: { severity: warning }

- alert: DiskWillFillIn4h
  expr: predict_linear(node_filesystem_avail_bytes[1h], 4*3600) < 0
  for: 10m
  labels: { severity: critical }

- alert: HighCPU
  expr: 100 * (1 - avg by (instance)
              (rate(node_cpu_seconds_total{mode="idle"}[5m]))) > 90
  for: 15m

- alert: OOMKillRecent
  expr: rate(node_vmstat_oom_kill[5m]) > 0
  for: 1m

RED-метрики для каждого endpoint

# Rate
sum by (path) (rate(http_requests_total[5m]))

# Errors
sum by (path) (rate(http_requests_total{code=~"5.."}[5m]))

# Duration p95
histogram_quantile(0.95,
  sum by (le, path) (rate(http_request_duration_seconds_bucket[5m])))
