// 01Что такое IaC и Terraform

Infrastructure as Code — описание инфраструктуры в текстовых файлах, версионирование в git, применение через CLI/CI. Цель: воспроизводимость и аудит.

Что описываем

  • VM, диски, сети, балансировщики в облаке.
  • Managed-сервисы: RDS, S3, GKE.
  • DNS-записи, TLS-сертификаты.
  • K8s-объекты (через provider kubernetes / helm).
  • Учётки и роли (IAM, GitHub teams, PagerDuty schedules).

Где Terraform

Terraform — provisioning уровня инфраструктуры. Что НЕ Terraform:

  • Конфигурация ОС внутри VM → Ansible/Puppet/cloud-init.
  • Деплой приложений → Helm / Argo / k8s manifests.
  • Имиджи VM → Packer.

Модель работы

  1. Описываете желаемое состояние в .tf файлах (HCL).
  2. Terraform читает state (что сейчас).
  3. plan: показывает дельту (что добавит/изменит/удалит).
  4. apply: вызывает API провайдера, обновляет state.
// Декларативность ≠ идемпотентность
В Ansible идемпотентность — на уровне модуля. В Terraform всё идемпотентно по умолчанию: state описывает что есть, plan показывает дельту. Повторный apply без изменений → no changes.

// 02Terraform vs OpenTofu

В августе 2023 HashiCorp сменила лицензию Terraform с MPLv2 на BSL — не OSS. Сообщество форкнуло проект как OpenTofu под Linux Foundation. На 2025–2026 это два почти идентичных продукта.

TerraformOpenTofu
ЛицензияBSL (с 1.6)MPLv2 (true OSS)
Бинарьterraformtofu
HCL и языкИдентичныИдентичны
ПровайдерыRegistry HashiCorpRegistry OpenTofu + совместимы с HC
УникальноеStacks, новые фичи Terraform CloudState encryption, dynamic providers, провайдер-iteration ранее
Кто выбираетЭнтерпрайз с HC поддержкойOSS-команды, нейтральная позиция

Большинство существующего кода работает в обоих. Учитесь на любом — навыки переносятся 1-в-1. В этом конспекте — terraform, но tofu подставляется на все команды.

Файлы конфигурации

Внутри одной директории Terraform склеивает все *.tf файлы. Имена не важны (обычно main.tf, variables.tf, outputs.tf, providers.tf, versions.tf).

// 04HCL: язык

HashiCorp Configuration Language. Декларативный, с поддержкой выражений, циклов, функций. Синтаксис похож на JSON, но с комментариями и более читаемый.

# Комментарий (или //)
/* блочный
   комментарий */

resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name        = "web-server"
    Environment = var.env
  }
}

Типы данных

  • Примитивы: string, number, bool.
  • Коллекции: list(T), set(T), map(T).
  • Структуры: object({a=string, b=number}), tuple([string, number]).
  • any — динамический.
  • null — отсутствие значения.

local.map_squared = {for k, v in var.m : k => v * v}

// 05Основные блоки

terraform — мета-конфиг

terraform {
  required_version = ">= 1.6"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.40"
    }
  }
  backend "s3" { ... }
}

provider — подключение к API

provider "aws" {
  region  = "us-east-1"
  profile = "prod"

  default_tags {
    tags = {
      ManagedBy = "terraform"
      Repo      = "infra"
    }
  }
}

# Несколько инстансов одного провайдера
provider "aws" {
  alias  = "eu"
  region = "eu-west-1"
}

resource "aws_s3_bucket" "eu" {
  provider = aws.eu
  bucket   = "my-eu-bucket"
}

resource — создаваемая сущность

resource <TYPE> <NAME> {
  attribute = value
  block {
    nested_attribute = value
  }
}

TYPE — определяется провайдером (aws_instance, yandex_compute_instance, kubernetes_namespace). NAME — локальный идентификатор внутри модуля.

Все основные блоки

  • terraform { ... } — мета-конфиг.
  • provider "X" { ... } — конфигурация провайдера.
  • resource "T" "N" { ... } — создаваемый ресурс.
  • data "T" "N" { ... } — чтение из внешнего источника.
  • variable "N" { ... } — входной параметр.
  • output "N" { ... } — выходное значение.
  • locals { ... } — локальные значения.
  • module "N" { ... } — подключение модуля.
  • import { ... } — декларативный импорт (1.5+).
  • moved { ... } — переименование без recreate.
  • removed { ... } — удаление из state без destroy (1.7+).
  • check { ... } — runtime-проверки (1.5+).

// 06Workflow: init / plan / apply

terraform init           # скачать провайдеров, настроить backend
terraform fmt -recursive # отформатировать .tf
terraform validate       # проверить синтаксис и типы
terraform plan -out=plan.tfplan
terraform apply plan.tfplan
terraform destroy

init подробнее

  • Скачивает указанные провайдеры в .terraform/providers/.
  • Создаёт .terraform.lock.hcl — фиксирует точные версии (как package-lock.json).
  • Конфигурирует backend.
  • Скачивает модули.

Флаги: -upgrade (обновить версии), -reconfigure (новый backend), -migrate-state (перенести state).

plan

Сравнивает: ваш код ↔ state ↔ реальность (через refresh). Выдаёт список изменений: + create, ~ update, - destroy, -/+ replace (force new).

Иногда видите «known after apply» — значение зависит от того, что вернёт API после создания.

terraform plan -target=module.vpc       # только этот ресурс
terraform plan -refresh=false           # без обращения к API
terraform plan -var="env=prod"
terraform plan -var-file=prod.tfvars

apply

  • С планом: terraform apply plan.tfplan — применит ровно то, что было в плане.
  • Без плана: terraform apply → ещё раз plan + интерактивный подтверждение.
  • -auto-approve — без подтверждения (для CI).
  • -parallelism=10 — параллельные операции (default).
// Правило
В CI всегда plan -out=plan.tfplan + код-ревью + apply plan.tfplan. Никогда apply -auto-approve без сохранённого плана — между plan и apply мир может измениться.

// 07State: что это и где хранить

State — JSON-файл (terraform.tfstate), который хранит маппинг: ресурс в коде ↔ реальный объект в облаке. Без state Terraform не знает, что он уже создал.

Зачем state

  • Маппинг код → ID реальных ресурсов.
  • Метаданные (зависимости, sensitive значения).
  • Кэш для перформанса больших инфр.

Локальный state — почему плохо

  • Лежит на ноутбуке → второй человек не может работать.
  • Нет блокировки → два apply параллельно ломают state.
  • Секреты в plain text внутри state.
  • Случайное rm -rf .terraform — катастрофа.

Remote backend — норма prod

S3 + DynamoDB-lock — classic. Альтернативы: GCS, Azure Storage, Terraform Cloud, Yandex Object Storage, локальный consul/etcd.

// Безопасность state
State содержит значения секретов в plain text (пароль RDS, токен). Поэтому:
  • Backend с шифрованием at-rest (S3 SSE-KMS).
  • Жёсткие IAM-политики на bucket.
  • Не коммитьте .tfstate в git никогда.
  • OpenTofu 1.7+ умеет state encryption на уровне самого Terraform.

// 08Remote backends и locking

S3 + DynamoDB

terraform {
  backend "s3" {
    bucket         = "my-tf-state"
    key            = "envs/prod/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "tf-locks"
    kms_key_id     = "alias/terraform-state"
  }
}

DynamoDB-таблица должна иметь LockID primary key (string). На время apply Terraform пишет туда блокировку.

Bootstrap проблема

Чтобы создать S3-bucket для state, нужен Terraform. Чтобы запустить Terraform, нужен S3. Решение:

  1. Первый раз — локальный state, создаём bucket и DynamoDB вручную (или мини-Terraform с локальным state).
  2. Затем добавляем backend "s3" и делаем terraform init -migrate-state — Terraform мигрирует локальный state в S3.

Backend конфиг частичный

# backend.tf
terraform {
  backend "s3" {}      # значения задаются при init
}

# Запуск
terraform init \
  -backend-config="bucket=my-tf-state" \
  -backend-config="key=envs/prod/state.tfstate" \
  -backend-config="region=us-east-1" \
  -backend-config="dynamodb_table=tf-locks"

Альтернативные backend'ы

  • Terraform Cloud / Enterprise — UI + RBAC + run pipeline.
  • Spacelift / env0 / Scalr — third-party «Terraform Cloud».
  • Atlantis — open-source автомейкер для PR'ов в git.
  • GCS / AzureRM — для соответствующих облаков.

// 09Variables, locals, outputs

variable — вход

variable "env" {
  description = "Окружение: dev / staging / prod"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.env)
    error_message = "env должен быть dev/staging/prod."
  }
}

variable "db_password" {
  type      = string
  sensitive = true                  # замаскировано в логах
}

variable "instance_specs" {
  type = object({
    cpu     = number
    memory  = number
    disk_gb = number
  })
  default = { cpu = 2, memory = 4096, disk_gb = 50 }
}

Источники variable (по приоритету)

  1. -var и -var-file в CLI.
  2. *.auto.tfvars / *.auto.tfvars.json (автозагрузка).
  3. terraform.tfvars / terraform.tfvars.json (автозагрузка).
  4. ENV TF_VAR_name.
  5. default в декларации.

locals — DRY-выражения

locals {
  name_prefix = "${var.project}-${var.env}"
  common_tags = {
    Project     = var.project
    Environment = var.env
    Owner       = "platform"
  }
  is_prod = var.env == "prod"
}

resource "aws_s3_bucket" "app" {
  bucket = "${local.name_prefix}-app"
  tags   = local.common_tags
}

output — экспорт

output "vpc_id" {
  value       = aws_vpc.main.id
  description = "VPC ID для соседних стэков"
}

output "db_password" {
  value     = aws_db_instance.main.password
  sensitive = true
}

# Чтение output'ов
terraform output
terraform output vpc_id
terraform output -json | jq ".vpc_id.value"

// 10data sources и dependency

Data source — readonly запрос к API. Не создаёт ресурсов; используется для импорта существующих значений.

data "aws_ami" "ubuntu" {
  most_recent = true
  owners      = ["099720109477"]
  filter {
    name   = "name"
    values = ["ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*"]
  }
}

resource "aws_instance" "web" {
  ami = data.aws_ami.ubuntu.id
  ...
}

remote_state — обмен между стэками

data "terraform_remote_state" "network" {
  backend = "s3"
  config = {
    bucket = "my-tf-state"
    key    = "envs/prod/network/terraform.tfstate"
    region = "us-east-1"
  }
}

resource "aws_instance" "web" {
  subnet_id = data.terraform_remote_state.network.outputs.subnet_id
}

Зависимости

Terraform строит DAG автоматически по ссылкам (resource.X.id). Иногда нужна явная зависимость:

resource "null_resource" "trigger_after_db" {
  depends_on = [aws_db_instance.main]
  triggers = {
    db_endpoint = aws_db_instance.main.endpoint
  }
}

// 11Modules

Модуль — папка с .tf файлами + variables + outputs. Переиспользуется через module "X" { source = "..." }.

Структура своего модуля

modules/vpc/
├── main.tf
├── variables.tf       # входы
├── outputs.tf         # выходы
├── versions.tf        # required_providers, required_version
└── README.md

Использование

module "vpc" {
  source = "../../modules/vpc"

  cidr_block = "10.0.0.0/16"
  env        = var.env
}

module "db" {
  source = "../../modules/rds"

  vpc_id        = module.vpc.vpc_id
  subnet_ids    = module.vpc.private_subnet_ids
  instance_size = "db.t3.medium"
}

Источники модулей

  • Локальный путь: ./modules/vpc.
  • Git: git::https://github.com/org/repo.git//modules/vpc?ref=v1.2.0.
  • Terraform Registry: terraform-aws-modules/vpc/aws + version = "5.0.0".
  • S3 / GCS — для приватных репозиториев.

Best practices

  • Версионируйте модули семвером, ссылайтесь по тегу.
  • Тестируйте модули отдельно (Terratest, terraform test).
  • README с примером usage + описанием inputs/outputs (или terraform-docs auto-gen).
  • Один модуль = одна логическая сущность (VPC, EKS-кластер, RDS).
  • Не плодите тонкие обёртки на каждый ресурс — это анти-паттерн.

Terraform Registry — готовые модули

Для AWS/Azure/GCP стандарт — модули terraform-aws-modules/* (Anton Babenko). Покрывают 80% типовых сценариев: VPC, EKS, RDS, ALB, S3.

// 12Meta-arguments: count, for_each, depends_on

count — массив одинаковых

resource "aws_instance" "worker" {
  count         = 3
  ami           = data.aws_ami.ubuntu.id
  instance_type = "t3.small"
  tags = { Name = "worker-${count.index}" }
}

# Ссылка: aws_instance.worker[0].id, aws_instance.worker[*].id

Проблема count: если удалить элемент из середины (count=3 → count=2), index сдвигается, всё после — пересоздаётся. Используйте только когда порядок не важен.

for_each — словарь / set

variable "users" {
  type = map(object({ role = string }))
  default = {
    alice = { role = "admin" }
    bob   = { role = "viewer" }
  }
}

resource "aws_iam_user" "users" {
  for_each = var.users
  name     = each.key
  tags     = { role = each.value.role }
}

# Ссылка: aws_iam_user.users["alice"].arn

Удаление bob не задевает alice. Используйте for_each вместо count по умолчанию.

lifecycle

resource "aws_db_instance" "main" {
  ...
  lifecycle {
    prevent_destroy = true                # нельзя дропнуть apply'ем
    create_before_destroy = true          # сначала создать новый
    ignore_changes = [tags["LastModified"]]   # не реагировать на изменение
    replace_triggered_by = [aws_security_group.db]
  }
}

dynamic — динамические nested блоки

resource "aws_security_group" "app" {
  name = "app-sg"

  dynamic "ingress" {
    for_each = var.ingress_rules
    content {
      from_port   = ingress.value.from_port
      to_port     = ingress.value.to_port
      protocol    = "tcp"
      cidr_blocks = ingress.value.cidrs
    }
  }
}

// 14Workspaces и multi-env

Workspaces (встроенный механизм)

terraform workspace list
terraform workspace new staging
terraform workspace select prod
terraform workspace show

Workspace = отдельный state file. В коде доступен через terraform.workspace.

// Когда workspaces — плохо
Workspaces подходят для лёгких вариаций (test fixtures, ephemeral сред). Для prod / staging с разными backend, регионами, ролями — workspaces плохо. Рекомендуется: отдельная директория на окружение + отдельный backend.

Multi-env через директории (рекомендация)

envs/
├── prod/
│   ├── main.tf       # module "infra" { source = "../../modules/infra"; env = "prod" }
│   └── backend.tf
└── staging/
    ├── main.tf
    └── backend.tf
modules/
└── infra/            # вся логика тут

Плюсы:

  • Свой backend / bucket / regions / IAM roles.
  • Можно временно сломать staging, prod не задеть.
  • Видно в git, что относится к какому env.
  • Параметры через terraform.tfvars в каждой env-папке.

Terragrunt

Обёртка над Terraform от Gruntwork. Убирает дублирование между environments через DRY-конструкции. Альтернатива — стандартные TF features (1.6+ terraform.tfvars + share modules).

// 15Работа со state напрямую

terraform state list                       # все ресурсы
terraform state show aws_instance.web
terraform state rm aws_instance.web        # убрать из state (не удаляет из облака!)
terraform state mv aws_instance.web aws_instance.api
terraform state pull > backup.tfstate
terraform state push fixed.tfstate
terraform state replace-provider hashicorp/aws registry.opentofu.org/hashicorp/aws

import — добавить существующий ресурс в state

# Старый способ (CLI)
terraform import aws_instance.web i-0abc1234

# Новый декларативный (1.5+)
import {
  to = aws_instance.web
  id = "i-0abc1234"
}
# После плана и applyа можно сгенерировать code:
# terraform plan -generate-config-out=generated.tf

moved — переименование без recreate

moved {
  from = aws_instance.web
  to   = module.compute.aws_instance.web
}

Применили — Terraform переписал state, реальный ресурс не трогался.

removed — удалить из state, не из облака

removed {
  from = aws_instance.legacy
  lifecycle { destroy = false }
}

// 17Terraform в CI/CD

Базовый workflow для PR

  1. На pull_request: fmt -check, validate, tflint, checkov, plan.
  2. Постим plan в комментарий PR.
  3. Ревью человеком.
  4. На merge в main: apply сохранённого плана (или нового, если безопасно).

GitHub Actions — пример

name: terraform
on:
  pull_request:
    paths: ["envs/prod/**", "modules/**"]
  push:
    branches: [main]

permissions:
  id-token: write
  contents: read
  pull-requests: write

jobs:
  plan:
    runs-on: ubuntu-latest
    defaults: { run: { working-directory: envs/prod } }
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123:role/tf-plan
          aws-region: us-east-1
      - uses: hashicorp/setup-terraform@v3
        with: { terraform_version: 1.9.0 }
      - run: terraform init
      - run: terraform fmt -check -recursive
      - run: terraform validate
      - id: plan
        run: terraform plan -out=plan.tfplan -no-color | tee plan.txt
      - uses: actions/upload-artifact@v4
        with: { name: tfplan, path: envs/prod/plan.tfplan }
      - if: github.event_name == 'pull_request'
        uses: actions/github-script@v7
        with:
          script: |
            const plan = require('fs').readFileSync('envs/prod/plan.txt', 'utf8');
            const body = '```\n' + plan.slice(-60000) + '\n```';
            github.rest.issues.createComment({...context.repo, issue_number: context.issue.number, body});

  apply:
    needs: plan
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production            # required-reviewer approval
    defaults: { run: { working-directory: envs/prod } }
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123:role/tf-apply
          aws-region: us-east-1
      - uses: hashicorp/setup-terraform@v3
      - run: terraform init
      - uses: actions/download-artifact@v4
        with: { name: tfplan }
      - run: terraform apply -auto-approve plan.tfplan

Atlantis vs GitHub Actions

  • Atlantis — самостоятельный сервер, общается с GitHub через webhooks. PR-комментарии atlantis plan / atlantis apply. Шарит state-lock между разработчиками.
  • Actions — родное для GitHub, проще запустить. Для больших монорепо лучше Atlantis.
// Безопасность CI
Используйте OIDC к AWS/GCP вместо долгоживущих ключей. Plan-роль с минимумом прав (describe*, list*), apply-роль с полными правами в отдельном environment с approval. Никогда не давайте PR-плану доступ к apply.

// 18Cheatsheet и best practices

10 правил продакшен-Terraform

  1. Backend remote + state encryption + DynamoDB lock.
  2. Версии (provider, terraform) запинены в required_* и lock-файле.
  3. Один stack ≠ одна вселенная. Разбивайте на network / data / compute / observability.
  4. Multi-env через отдельные директории, не workspaces.
  5. for_each вместо count по умолчанию.
  6. Sensitive variables помечены sensitive = true.
  7. plan обязателен в CI; apply — только сохранённого плана.
  8. tflint + checkov + tfsec в pre-commit и CI.
  9. Не редактируйте state вручную в редких исключениях (см. moved/removed/import).
  10. Документация модулей автогенерируется (terraform-docs).
