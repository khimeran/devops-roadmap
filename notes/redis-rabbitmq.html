// 01Зачем DevOps это знать

Redis и RabbitMQ — два «скучных» инфра-сервиса, которые встречаются почти в каждом backend-стэке. От DevOps ждут не разработки на них, а:

  • Развернуть и настроить prod-ready инсталляцию (HA, бэкапы, мониторинг).
  • Понимать, что показывают метрики и логи; ловить деградации заранее.
  • Знать, когда какой инструмент уместен — отбиваться от запросов «нам нужен Redis Cluster для очереди».
  • Чинить инциденты: split-brain, OOM, потеря сообщений, дедлоки.

Roadmap по модулю

  1. Redis: модель данных и persistence (день 1–4).
  2. Redis HA: master-replica, Sentinel, Cluster (день 5–7).
  3. RabbitMQ: модель и reliability (день 8–11).
  4. RabbitMQ HA: кластер, Quorum queues (день 12–13).
  5. Проект: связка app → Rabbit → worker → Redis cache (день 14).

// 02Redis: что это

Redis — in-memory key-value store. Однопоточный (один event-loop) — конкурентность достигается асинхронностью и репликами на разных портах.

Типичные применения

  • Cache — кэширование результатов запросов / API (TTL + LRU eviction).
  • Session store — сессии веб-приложений.
  • Rate limiting — счётчики на ключ INCR + EXPIRE.
  • Distributed lock — SET key value NX PX + Redlock.
  • Pub/Sub — простые событийные шины.
  • Очереди — через списки (LPUSH/BRPOP) или Streams.
  • Leaderboard / счётчики — Sorted Sets.
  • Геопоиск — модуль Geo (GEOADD/GEOSEARCH).

Архитектура процесса

  • Один процесс redis-server, один event-loop.
  • Слушает по умолчанию на 127.0.0.1:6379.
  • RESP — текстовый протокол (telnet-friendly).
  • Дочерние процессы для тяжёлых задач: snapshot RDB, AOF rewrite.
// Лицензия
С 2024 Redis Inc. перевела сервер на RSALv2/SSPLv1 — несовместимая с OSS. Сообщество форкнуло код как Valkey (Linux Foundation). API совместимы — клиенты, команды, протокол идентичны. Если выбираете сейчас — берите Valkey 7.2/8.0.

TTL и срок жизни ключа

SET cache:user:42 "{...}" EX 300       # истекает через 300с
EXPIRE cache:user:42 60
TTL cache:user:42                       # сколько осталось
PERSIST cache:user:42                   # снять TTL

Транзакции и Lua

# MULTI/EXEC — атомарная пачка команд
MULTI
INCR counter
EXPIRE counter 60
EXEC

# Lua — атомарный скрипт
EVAL "return redis.call('INCR', KEYS[1])" 1 counter

В Redis нет «отката» транзакции — она просто атомарна. Если одна команда в MULTI синтаксически ломаная — весь блок отбрасывается перед EXEC. Если падает в runtime — остальные выполняются.

// 04Persistence: RDB и AOF

Redis в памяти, но умеет сохранять состояние на диск. Два механизма, можно использовать оба одновременно.

# redis.conf — типичная prod-конфигурация
save 900 1                       # RDB: snapshot если 1+ изменение за 900с
save 300 10
save 60 10000

appendonly yes
appendfsync everysec             # fsync раз в секунду — баланс скорость/durability
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

Гибридный режим (default с 7.x)

AOF начинается с RDB-snapshot'a, дальше дописываются команды. Быстрое восстановление + минимальная потеря.

BGSAVE и fork

RDB-snapshot создаётся через fork() — дочерний процесс пишет, родительский продолжает обслуживать. Memory overhead: копирование страниц при записи (COW). Если приложение пишет интенсивно — RAM может удвоиться. Учитывайте при sizing.

// Внимание
appendfsync always — fsync на каждую запись, durability максимум, но throughput падает в 10×. everysec — стандарт. no — почти как без AOF.

// 05Eviction policy и maxmemory

Когда Redis упирается в maxmemory, он начинает выселять ключи по выбранной политике.

maxmemory 4gb
maxmemory-policy allkeys-lru

Политики

ПолитикаЛогика
noevictionНе выселять, write-команды возвращают ошибку
allkeys-lruLRU среди всех ключей
allkeys-lfuLFU (по частоте обращения)
allkeys-randomСлучайный
volatile-lruLRU только среди ключей с TTL
volatile-lfuLFU только среди с TTL
volatile-ttlБлиже к expire
volatile-randomСлучайный среди с TTL

Выбор политики

  • Cache → allkeys-lru / allkeys-lfu.
  • Storage (Redis как БД) → noeviction. Иначе теряете данные.
  • Mixed (часть кэш, часть данные) → volatile-lru: cache-ключи ставим с TTL, важные — без.
// Совет
Дефолт — noeviction. Это значит, что под нагрузкой и без maxmemory вы получите OOM на хосте, а не вежливые ошибки от Redis. Всегда задавайте maxmemory = 70–80% от лимита процесса.

// 06Pub/Sub и Streams

Pub/Sub (легаси)

# Подписчик
SUBSCRIBE chat.room1

# Издатель
PUBLISH chat.room1 "hello"

Без persistence — если подписчика нет в момент публикации, сообщение теряется. Не использовать как «очередь».

Streams (5.0+) — нормальная очередь

# Добавить
XADD events * type signup user 42
XADD events * type purchase user 42 amount 99.99

# Прочитать с какого-то ID
XREAD COUNT 10 STREAMS events 0
XREAD BLOCK 5000 STREAMS events "$"     # ждать новые до 5s

# Consumer groups (как Kafka)
XGROUP CREATE events workers "$" MKSTREAM
XREADGROUP GROUP workers worker-1 COUNT 1 BLOCK 0 STREAMS events >
XACK events workers <id>

Streams — серьёзная альтернатива RabbitMQ для очередей с гарантиями at-least-once и consumer groups. Поддерживают capped retention (XADD MAXLEN ~ 1000000).

Keyspace notifications

Можно подписаться на события Redis (set, del, expire) через Pub/Sub:

CONFIG SET notify-keyspace-events "Ex"     # expired events
SUBSCRIBE __keyevent@0__:expired

Удобно для делайт-очередей: SET с TTL → expired event → обработка.

// 07Репликация и Sentinel

Master-replica

Асинхронная репликация. Реплики read-only, видят все команды master через RESP-поток. Несколько уровней (replica of replica) — поддерживается.

# replica.conf
replicaof master-host 6379
replica-read-only yes
masterauth "<password>"

Reconnect: реплика старается сделать partial resync через offset; если не получается — полный sync через RDB.

Sentinel — high availability

Sentinel — отдельный процесс. Минимум 3 инстанса (нечётное для кворума). Что делает:

  • Health-check master и replicas.
  • Failover: если master недоступен дольше down-after-milliseconds, выбирает наиболее «свежую» реплику и промоутит в master.
  • Обновляет конфиги клиентов: клиенты подключаются к Sentinel, не к Redis напрямую — Sentinel отдаёт адрес текущего master.
  • Уведомляет (через скрипты).
# sentinel.conf
sentinel monitor mymaster 10.0.0.10 6379 2     # quorum=2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1

Подключение клиента

# python
from redis.sentinel import Sentinel
s = Sentinel([("sentinel-1", 26379), ("sentinel-2", 26379)], socket_timeout=0.5)
master = s.master_for("mymaster")
slave = s.slave_for("mymaster")
// Когда Sentinel
Подходит для случаев, где данные помещаются в один инстанс, но нужен failover. Если данных больше — нужен Redis Cluster.

// 08Redis Cluster

Шардирование на уровне сервера. 16384 hash slots распределяются по узлам; ключ попадает в слот через CRC16(key) mod 16384.

Топология

  • Минимум 3 master-узла + по 1 реплике (итого 6).
  • Каждый master отвечает за ~5461 слот.
  • Реплика следит за master, делает failover.
  • Узлы общаются gossip-протоколом по «cluster bus» (порт port + 10000).

Создание кластера

redis-cli --cluster create \
  10.0.0.1:6379 10.0.0.2:6379 10.0.0.3:6379 \
  10.0.0.4:6379 10.0.0.5:6379 10.0.0.6:6379 \
  --cluster-replicas 1

redis-cli --cluster check 10.0.0.1:6379
redis-cli --cluster reshard 10.0.0.1:6379

Hash tags — мульти-ключевые операции

В кластере multi-key команды работают только если ключи в одном слоте. Hash tag — фигурные скобки:

MSET {user:42}:name "Alex" {user:42}:email "a@x.com"
# оба ключа хэшируются по "user:42" → один слот

Ограничения

  • Транзакции (MULTI/EXEC) и Lua-скрипты — только в пределах одного слота.
  • Команды над несколькими БД (SELECT 1) недоступны — только DB 0.
  • Pub/Sub global (с 7.0 есть sharded pub/sub — SSUBSCRIBE).
// Не путать
Redis Cluster ≠ просто несколько Redis-инстансов. Это конкретный режим с слотами, протоколом и автоматическим redirect (MOVED, ASK). Клиент должен поддерживать cluster-mode.

// 09Безопасность и ACL

Базовый чек-лист prod

  • bind 10.0.0.10 — никогда не слушать на 0.0.0.0 в публичной сети.
  • protected-mode yes.
  • requirepass или ACL.
  • TLS (tls-port, tls-cert-file...).
  • Firewall: только нужные порты от нужных IP.
  • Отключить опасные команды (FLUSHALL, CONFIG, KEYS, DEBUG) через ACL или rename-command.

ACL (6.0+)

# Создать пользователя для приложения
ACL SETUSER app on >StrongPwd! ~cache:* +@read +@write -@dangerous

# Только чтение
ACL SETUSER reporter on >Pwd ~* +@read

# Список
ACL LIST
ACL WHOAMI
ACL LOG

Категории команд: @read, @write, @admin, @dangerous, @slow. Гибче, чем один пароль на всё.

// Реальный кейс
Redis на 0.0.0.0 без пароля в публичной сети — атакующий за 10 минут запишет cron-payload через CONFIG SET dir /var/spool/cron + SAVE. Десятки тысяч инстансов так компрометированы. Никогда не оставляйте Redis наружу.

// 10Мониторинг Redis

Что смотреть

  • Память: used_memory, used_memory_rss, mem_fragmentation_ratio.
  • Throughput: instantaneous_ops_per_sec.
  • Hit ratio: keyspace_hits / (hits + misses).
  • Reject'ы: rejected_connections, connected_clients vs maxclients.
  • Eviction: evicted_keys — растёт → не хватает памяти.
  • Replication lag: master_repl_offset - slave_repl_offset.
  • Latency: redis-cli --latency, slowlog.

// 11RabbitMQ: что это

RabbitMQ — message broker, реализующий AMQP 0-9-1 (плюс STOMP, MQTT через плагины). Эрланговая VM (как и Erlang/Elixir) — отсюда репутация «работает годами без проблем».

Зачем брокер

  • Decoupling — producer не знает о consumer'ах.
  • Buffer — спайки нагрузки гасятся очередью.
  • Routing — сообщение одновременно нескольким consumer'ам.
  • Reliable delivery — гарантии доставки (at-least-once, ordered).
  • Async — фоновые задачи без блокировки HTTP-запроса.

Когда RabbitMQ, а не Kafka

  • Сложный routing (fanout, topic-based).
  • Per-message acks, retries, dead-letter — это в Rabbit нативно.
  • Низкие задержки и средние объёмы (10k–50k msg/sec).
  • Не нужно «реплеить» сообщения через дни/недели.

Kafka выигрывает на потоковых сценариях с TB-объёмами и replay (event sourcing, analytics pipeline).

Topic patterns

  • * — ровно одно слово.
  • # — ноль или больше слов.

Routing key orders.eu.created матчит binding'ам orders.*.created, orders.#, *.eu.*.

Атрибуты exchange / queue

  • durable — переживёт рестарт брокера.
  • auto-delete — удалится, когда отвалится последний consumer/binding.
  • exclusive — только для одной connection.
  • x-message-ttl, x-max-length, x-dead-letter-exchange — аргументы queue.

Dead Letter Exchange

Сообщения, которые: rejected без requeue, истекли TTL, или превысили max-length — могут уходить в DLX. Оттуда — в специальную очередь для разбора:

queue.declare "orders" durable=true
  x-dead-letter-exchange = "orders.dlx"
  x-message-ttl = 60000

// 14Acknowledgements и persistence

Без правильных acks Rabbit теряет сообщения тихо. Три уровня надёжности:

Сторона consumer

  • auto-ack — Rabbit удаляет сообщение сразу после доставки. Crash клиента = потеря.
  • manual ack (default правильный) — клиент вызывает basic.ack после успешной обработки. Если соединение оборвалось — сообщение возвращается в очередь и доставляется другому consumer'у.
  • basic.nack / basic.reject — с флагом requeue=true/false.
# pika (python)
def on_message(ch, method, props, body):
    try:
        process(body)
        ch.basic_ack(delivery_tag=method.delivery_tag)
    except Exception:
        ch.basic_nack(delivery_tag=method.delivery_tag, requeue=False)

Сторона producer

  • Publisher confirms — после publish Rabbit отправляет basic.ack producer'у, что сообщение принято и (если queue durable) лежит на диске. Без этого publish — «fire and forget».
  • Mandatory flag — если routing невозможен, Rabbit возвращает basic.return, иначе сообщение тихо отбрасывается.

Persistence

  • Queue durable=true — переживёт рестарт сервера.
  • Message persistent (delivery_mode=2) — пишется на диск.
  • Оба обязательны вместе. Только durable queue без persistent message теряет данные при рестарте.

Prefetch / QoS

# Каждый consumer получает максимум 10 unacked сообщений
ch.basic_qos(prefetch_count=10)

Без prefetch один consumer схватит всю очередь и не сможет обработать. Норма для длинных задач — prefetch=1; для лёгких — 10–100.

// Recipe для надёжности
durable queue + persistent message + manual ack + publisher confirms + DLX для отказов = at-least-once без потерь. Идемпотентность обработчика — на стороне consumer (одно и то же сообщение может прийти дважды).

// 15Кластеризация и Quorum queues

Кластер

Минимум 3 узла (кворум на failover). Делятся метаданными (exchanges, queues, bindings, users). Сообщения по умолчанию не реплицируются — живут только на «home» node.

# Подключение узла к кластеру
rabbitmqctl stop_app
rabbitmqctl reset
rabbitmqctl join_cluster rabbit@node1
rabbitmqctl start_app
rabbitmqctl cluster_status

# Создание quorum queue
queue.declare "orders" durable=true
  x-queue-type = "quorum"

Streams (3.9+)

Append-only лог, как Kafka, в одном Rabbit. Replay, time-based offset, partitioning. Альтернатива для event-sourcing сценариев.

// Внимание (4.x)
Начиная с RabbitMQ 4.0, classic mirrored queues удалены. Если апгрейдитесь со старых версий — мигрируйте на quorum queues заранее.

// 16Мониторинг RabbitMQ

Management UI

Включить плагин: rabbitmq-plugins enable rabbitmq_management. Доступен на :15672. Видны connections, channels, queues, exchanges, графики rate.

Prometheus plugin

rabbitmq-plugins enable rabbitmq_prometheus
# /metrics на :15692

Ключевые метрики

  • queue_messages_ready — лежит и ждёт consumer'а.
  • queue_messages_unacknowledged — отдано consumer'у, ack ещё нет.
  • publish/deliver/ack rate — throughput.
  • rabbitmq_disk_free vs disk_free_limit — при достижении лимита publishers блокируются.
  • rabbitmq_resident_memory_used vs vm_memory_high_watermark.
  • file descriptors used / limit.
  • channels per connection — если >100, что-то не так.

CLI

rabbitmqctl status
rabbitmqctl list_queues name messages consumers
rabbitmqctl list_consumers
rabbitmqctl list_connections
rabbitmqctl list_channels
rabbitmqctl list_users
rabbitmqctl list_policies

rabbitmq-diagnostics check_running
rabbitmq-diagnostics check_local_alarms
rabbitmq-diagnostics check_port_connectivity
rabbitmq-diagnostics observer

Алерты-минимум

  • queue_messages_ready > X в течение N минут (очередь растёт быстрее обработки).
  • consumers == 0 для важной queue.
  • memory или disk alarm активен.
  • connections > 80% от лимита.
  • channels > 1000 (вероятно, leak в клиенте).

Антипаттерны

  • Redis как primary БД для финансовых данных — слишком слабые гарантии durability (даже AOF).
  • RabbitMQ для event sourcing с replay через дни — не для этого, переходите на Kafka.
  • Kafka как cache — overkill, лучше Redis.
  • Один Redis на cache + очередь + хранилище сессий — разделяйте инстансы. Memory pressure на одном из применений положит остальные.

10 правил продакшена

  1. Redis — задавайте maxmemory и явную maxmemory-policy.
  2. Redis — не KEYS в проде, только SCAN.
  3. Redis — мониторьте evicted_keys и hit ratio.
  4. Redis HA — Sentinel для standalone, Cluster для большого датасета.
  5. RabbitMQ — quorum queues для всего важного.
  6. RabbitMQ — manual ack + publisher confirms + DLX.
  7. RabbitMQ — prefetch_count явный, не дефолт.
  8. RabbitMQ — не пересоздавайте queue/binding на каждый запуск приложения; декларируйте идемпотентно с одинаковыми аргументами.
  9. Оба — TLS + auth, никогда не оставляйте наружу с дефолтными паролями.
  10. Оба — мониторинг через нативные Prometheus exporters + алерты на ключевые метрики.
