// 01Что такое Git и зачем он нужен

Git — это распределённая система контроля версий (DVCS), созданная Линусом Торвальдсом в 2005 году для разработки ядра Linux. Git хранит снимки (snapshots) состояния файлов проекта в специальном каталоге .git, и позволяет в любой момент откатиться к любой версии, переключиться между параллельными линиями разработки и слить изменения нескольких авторов.

Почему DevOps-инженеру это критично

  • Infrastructure as Code — Ansible playbooks, Terraform манифесты, Helm чарты живут в Git. Каждое изменение инфраструктуры — это коммит.
  • CI/CD — пайплайны срабатывают на git-события (push, tag, PR).
  • Аудит и rollback — кто что изменил и когда. Сломал прод — откатил.
  • Совместная работа — десятки человек редактируют один репозиторий без конфликтов.

Ключевые понятия одной строкой

  • Репозиторий (repo) — папка с проектом + история в .git/.
  • Коммит — снимок состояния всех файлов + автор + сообщение + ссылка на родителя.
  • Ветка (branch) — подвижный указатель на коммит. main, feature/x.
  • HEAD — указатель «где я сейчас». Обычно совпадает с веткой.
  • Remote — внешняя копия репозитория (на GitHub, GitLab, Gitea).
// Важно понимать
Git хранит снимки, а не изменения. Каждый коммит — это полный снимок состояния. Дельты вычисляются на лету при diff. Это принципиальное отличие от SVN/CVS и причина, по которой ветки в Git дёшевы.

// 03Базовый цикл: init, add, commit

Создать репозиторий

# Новый проект с нуля
mkdir my-project && cd my-project
git init                       # создаёт .git/

# Или клонировать существующий
git clone git@github.com:user/repo.git
git clone https://github.com/user/repo.git
git clone --depth 1 <url>     # shallow clone — только последний коммит

Базовый цикл работы

# 1. Посмотреть что изменилось
git status                     # краткий обзор
git status -s                  # совсем коротко: M, A, ??

# 2. Добавить файлы в индекс (staging area)
git add file.txt               # один файл
git add src/                   # всю папку
git add .                      # всё в текущей директории
git add -A                     # всё, включая удалённые файлы
git add -p                     # интерактивно по хункам — золотой стандарт

# 3. Закоммитить
git commit -m "feat: add user login"
git commit                     # откроет редактор для длинного сообщения
git commit -am "fix"           # add + commit для уже отслеживаемых файлов

Хорошее сообщение коммита

Стандарт Conventional Commits:

<type>(<scope>): <short summary>

<detailed body, why not what>

<footer: BREAKING CHANGE, Closes #123>

Типы: feat, fix, docs, style, refactor, test, chore, ci, build, perf.

Пример:

feat(auth): add password reset via email

Send a one-time token to user's email; token is valid for 15 min.
Closes #142
// Правило 50/72
Первая строка — до 50 символов, тело — до 72 символов в строке. Так нормально читается в git log, GitHub diff, email-уведомлениях.

Исправить последний коммит

# Изменить сообщение последнего коммита
git commit --amend -m "new message"

# Добавить забытый файл в последний коммит
git add forgotten.txt
git commit --amend --no-edit

# ВАЖНО: amend меняет хэш коммита. Не делай amend после push в общую ветку!

// 04Три состояния файлов

Это самая важная ментальная модель Git. Файл всегда находится в одном из трёх «миров»:

СостояниеГде живётЧто значит
Working treeНа дискеТо, что ты видишь и редактируешь
Index / Staging.git/indexПодготовлено к коммиту (git add)
Repository.git/objectsЗакоммичено навсегда (git commit)

Команды переводят файл из одного состояния в другое:

working tree  ──[git add]──>  index  ──[git commit]──>  repository
     ^                          |                            |
     └──[git checkout/restore]──┘                            |
     └──────[git reset --hard / git restore --source]────────┘

Просмотр изменений по состояниям

git diff              # working tree vs index — что НЕ добавлено
git diff --staged     # index vs HEAD — что добавлено, но не закоммичено
git diff HEAD         # working tree + index vs HEAD — всё несохранённое
git diff main..feat   # все изменения между ветками

Отмена изменений

# Убрать файл из staging (не трогая working tree)
git restore --staged file.txt
git reset HEAD file.txt          # старый синтаксис, делает то же

# Откатить изменения в working tree до состояния HEAD
git restore file.txt
git checkout -- file.txt         # старый синтаксис

# Откатить всё: и индекс, и working tree
git reset --hard HEAD            # ОСТОРОЖНО — несохранённые изменения потеряны
// Подводный камень
git reset --hard и git restore без --staged удаляют изменения в working tree без подтверждения и без возможности восстановить (если файл не был закоммичен). Перед хардкорными операциями делай git stash.

// 05История: log, diff, show, blame

git log — навигация по истории

git log                         # полная история текущей ветки
git log --oneline               # одна строка на коммит
git log --graph --all           # граф всех веток
git log --oneline --graph --decorate --all     # идеально (алиас lg)

git log -n 5                    # последние 5 коммитов
git log --author="Kirill"      # по автору
git log --since="2 weeks ago"
git log --grep="login"         # поиск по сообщению
git log -p                      # с diff каждого коммита
git log --stat                  # с статистикой изменений
git log -- path/to/file         # история конкретного файла
git log -S "foo()"            # коммиты, добавляющие/удаляющие "foo()"

git show — детали одного коммита

git show HEAD                   # последний коммит + diff
git show a3f1b2                 # по короткому хэшу
git show HEAD~2                 # коммит 2 шага назад от HEAD
git show HEAD:path/file.txt     # содержимое файла на момент HEAD

git blame — кто и когда написал строку

git blame README.md             # построчно: коммит + автор + строка
git blame -L 10,20 README.md    # только строки 10–20
git blame --since="1 month" file
// tig вместо git log
tig — TUI-обёртка над git. tig покажет интерактивный лог, нажми Enter на коммите — увидишь diff. tig blame file — интерактивный blame. Установка: dnf install tig.

// 06Ветки: branch, checkout, switch

Ветка — это просто файл в .git/refs/heads/ с хэшем коммита внутри. Создание ветки — операция O(1), занимает 41 байт. Ветви в Git не «дорогие», как в SVN — пользуйся ими щедро.

Жизненный цикл ветки

# Посмотреть ветки
git branch                      # локальные
git branch -a                   # + удалённые
git branch -v                   # + последний коммит

# Создать ветку
git branch feature/login        # создать, но остаться на текущей
git switch -c feature/login     # создать И переключиться (modern)
git checkout -b feature/login   # то же, старый синтаксис

# Переключаться
git switch main                 # переключиться на main
git switch -                    # вернуться на предыдущую ветку
git checkout main               # старый синтаксис

# Переименовать
git branch -m old-name new-name

# Удалить
git branch -d feature/login     # безопасно (только если слита)
git branch -D feature/login     # принудительно (даже если несли́та)
// switch vs checkout
switch и restore появились в Git 2.23 (2019) специально, чтобы разделить ответственности checkout. checkout делал слишком много — и менял ветки, и восстанавливал файлы. Современный код использует switch для веток и restore для файлов.

Соглашения по именам веток

  • main или master — основная стабильная ветка
  • develop — интеграционная ветка (GitFlow)
  • feature/<name> — новая фича
  • fix/<issue-id> — багфикс
  • hotfix/<name> — срочное исправление прода
  • release/v1.2 — подготовка релиза

// 07Слияние веток и стратегии merge

Базовое слияние

# Я хочу влить feature/login в main
git switch main
git pull                        # свежий main
git merge feature/login         # слить
git push                        # отправить

Три типа merge

1. Fast-forward (FF)

Если на main нет новых коммитов с момента ответвления — Git просто двигает указатель main вперёд. Никакого merge-коммита.

До:    A---B (main, feature)
После: A---B---C---D (main, feature)

2. 3-way merge

Если в обеих ветках есть свои коммиты — Git ищет общего предка и создаёт merge-коммит с двумя родителями.

До:    A---B---C (main)
            \
             D---E (feature)

После: A---B---C---M (main)
            \     /
             D---E (feature)

3. Squash merge

Все коммиты ветки склеиваются в один новый коммит на main. История ветки теряется, зато main остаётся чистым.

git merge --squash feature/login
git commit -m "feat: add login (squashed)"

Флаги управления стратегией

git merge --ff-only feature      # слить только если FF возможен (отказать иначе)
git merge --no-ff feature        # всегда создавать merge-коммит, даже если FF возможен
git merge --squash feature       # склеить в один коммит
git merge --abort                # откатить начатый merge
СтратегияКогда применятьМинусы
FFМелкие правки, чистая линияТеряет факт существования ветки
--no-ffВ command teams — видно границы фичГраф становится «лохматым»
--squashГрязная история ветки, нужен один логичный коммитТеряется детальная история

// 08Конфликты слияния

Конфликт возникает, когда один и тот же участок файла изменён в обеих сливаемых ветках. Git не знает, какую версию выбрать, и просит человека решить.

Как выглядит конфликт

<<<<<<< HEAD
String greeting = "Hello, World!";
=======
String greeting = "Привет, мир!";
>>>>>>> feature/i18n
  • <<<<<<< HEAD — версия текущей ветки
  • ======= — разделитель
  • >>>>>>> feature/i18n — версия вливаемой ветки

Алгоритм решения конфликта

# 1. Запустить merge, получить ошибку
git merge feature/i18n
# CONFLICT (content): Merge conflict in main.java

# 2. Посмотреть, какие файлы в конфликте
git status                       # они помечены как "both modified"

# 3. Открыть каждый файл и руками решить — оставить нужный код
#    Убрать все маркеры <<<, ===, >>>
vim main.java

# 4. Добавить разрешённый файл в индекс
git add main.java

# 5. Завершить merge
git commit                       # откроется редактор с готовым сообщением

# Если всё пошло не так — отменить
git merge --abort

Полезные инструменты

git mergetool                    # открыть графический разрешатель (meld, vimdiff, kdiff3)
git config --global merge.tool meld
git checkout --ours file.txt     # принять ВЕРСИЮ ТЕКУЩЕЙ ветки целиком
git checkout --theirs file.txt   # принять ВЕРСИЮ ВЛИВАЕМОЙ ветки целиком
git diff --name-only --diff-filter=U   # список файлов в конфликте
// Совет
Чтобы видеть в конфликте ещё и общего предка (3-way diff): git config --global merge.conflictstyle diff3. Это критично помогает понять, кто что менял, и часто конфликт решается тривиально.

// 09.gitignore и .gitattributes

.gitignore

Список паттернов файлов, которые Git должен игнорировать (не отслеживать). Лежит в корне репо или в любой подпапке.

# Конкретный файл
secret.env

# Все файлы с расширением
*.log
*.pyc

# Папка целиком
node_modules/
__pycache__/
.venv/

# Все .log на любой глубине
**/*.log

# Игнорировать всё в build/, кроме .keep
build/*
!build/.keep

# Игнорировать файл, отрицание
*.txt
!important.txt
// Гача
.gitignore работает только для неотслеживаемых файлов. Если файл уже был закоммичен — добавление его в gitignore не поможет. Нужно: git rm --cached file (удалить из индекса, но оставить на диске), потом коммит.

Готовые шаблоны

На github.com/github/gitignore — шаблоны для всех языков (Python, Node, Go, Java) и IDE (JetBrains, VSCode). Просто скопируй подходящий.

.gitattributes

Управляет тем, как Git обрабатывает определённые файлы: line endings, diff, merge, экспорт.

# Все текстовые — LF на любых платформах
* text=auto eol=lf

# Бинарные — не пытаться делать diff
*.png binary
*.pdf binary

# Исключить из git archive
.gitignore export-ignore
tests/ export-ignore

# LFS для больших файлов
*.psd filter=lfs diff=lfs merge=lfs -text

// 10Удалённые репозитории: push, pull, fetch

Управление remotes

git remote -v                              # посмотреть remote'ы (обычно origin)
git remote add origin git@github.com:user/repo.git
git remote add upstream git@github.com:original/repo.git   # для форков
git remote rename origin gh
git remote remove origin
git remote set-url origin git@gitlab.com:user/repo.git     # сменить URL

fetch vs pull

КомандаЧто делаетМеняет ли working tree
git fetchСкачивает изменения с remote в origin/mainНет
git pullfetch + merge origin/main в текущую веткуДа
git pull --rebasefetch + rebase поверх origin/mainДа
 
git fetch                       # просто скачать новости, не сливать
git fetch --all                 # со всех remote'ов
git fetch --prune               # удалить локальные ссылки на удалённые ветки

git pull                        # синхронизировать текущую ветку
git pull origin main            # явно с какого remote/ветки
git pull --rebase               # через rebase вместо merge — чище история

push

git push                                       # отправить текущую ветку
git push origin main                           # явно
git push -u origin feature/login               # первый push новой ветки (--set-upstream)
git push --tags                                # отправить и теги
git push origin --delete feature/old           # удалить ветку на remote
git push --force-with-lease                    # БЕЗОПАСНЫЙ force push (откажет, если кто-то ещё пушил)
git push --force                               # опасно! Перезаписывает чужие коммиты
// Никогда
Не делай git push --force в shared ветки (main, develop). Используй --force-with-lease для своих feature-веток после rebase. --force-with-lease откажется пушить, если remote изменился — это защищает чужие коммиты.

// 11GitHub: SSH-ключи и workflow

Генерация SSH-ключа

# Современный ed25519 (быстрее и безопаснее RSA)
ssh-keygen -t ed25519 -C "kirill@example.com"
# Enter, Enter (без пароля) или с паролем

# Файлы:
#   ~/.ssh/id_ed25519       — приватный (НИКОМУ не показывать!)
#   ~/.ssh/id_ed25519.pub   — публичный (его в GitHub)

# Скопировать публичный ключ
cat ~/.ssh/id_ed25519.pub
# или (Linux): xclip -sel clip < ~/.ssh/id_ed25519.pub
# или (Mac):   pbcopy < ~/.ssh/id_ed25519.pub

Добавить ключ в GitHub: Settings → SSH and GPG keys → New SSH key → вставить содержимое .pub.

# Проверить, что SSH работает
ssh -T git@github.com
# > Hi username! You've successfully authenticated...

~/.ssh/config — несколько ключей

Если у тебя несколько аккаунтов (work, personal):

Host github-personal
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_personal

Host github-work
  HostName github.com
  User git
  IdentityFile ~/.ssh/id_ed25519_work

Тогда: git clone git@github-work:company/repo.git

Типичный workflow (GitHub Flow)

  1. Создать ветку от main: git switch -c feature/login
  2. Коммиты в ветку
  3. Запушить: git push -u origin feature/login
  4. Открыть Pull Request в веб-интерфейсе
  5. Code review, обсуждение, правки
  6. Squash-merge или merge в main
  7. Локально удалить ветку: git switch main && git pull && git branch -d feature/login

// 12Rebase vs merge

Rebase переписывает историю ветки так, будто она ответвилась от свежего состояния целевой ветки. Это даёт линейную историю без merge-коммитов.

Графически

До rebase:
A---B---C (main)
    \
     D---E (feature)

После git rebase main (находясь на feature):
A---B---C (main)
        \
         D'---E' (feature)
# D' и E' — НОВЫЕ коммиты с новыми хэшами!

Команды

# Обычный rebase: переехать на свежий main
git switch feature/login
git fetch origin
git rebase origin/main

# Конфликты — как при merge
# Разрешить, git add, потом:
git rebase --continue
# Или отказаться
git rebase --abort

# Interactive rebase — переписать историю ветки
git rebase -i HEAD~5             # последние 5 коммитов
git rebase -i origin/main        # все коммиты от точки ветвления

Что можно делать в interactive rebase

pick   a3f1b2 feat: add login
squash b4c2d3 fix typo            # склеить с предыдущим
reword c5d3e4 refactor structure  # поменять сообщение
edit   d6e4f5 add tests           # остановиться и дать редактировать
drop   e7f5g6 wip                 # удалить коммит
# Можно менять порядок строк — Git применит в новом порядке

Когда что выбирать

СценарийЛучшеПочему
Влить feature в mainmerge --no-ffВидно, что была ветка
Обновить feature-ветку свежим mainrebaseЛинейная история без бесконечных merge'ев
Подчистить локальные коммиты перед PRrebase -iСквошнуть «fix typo» в нормальный коммит
Уже отправлено в shared веткуmergeRebase ломает чужую историю
// Золотое правило rebase
Никогда не делай rebase для коммитов, которые уже отправлены и доступны другим людям. Rebase меняет хэши коммитов; чужие клоны будут думать, что у них есть ваши коммиты, а на самом деле — нет. Получится хаос.

// 13Cherry-pick, revert, reset

git cherry-pick — взять один коммит

Применяет указанный коммит из другой ветки на текущую. Создаёт новый коммит с новым хэшем (но тем же содержанием и сообщением).

# Применить коммит a3f1b2 на текущую ветку
git cherry-pick a3f1b2

# Несколько коммитов
git cherry-pick a3f1b2 c5d3e4

# Диапазон (НЕ включая первый!)
git cherry-pick a3f1b2..c5d3e4

# Без создания коммита — оставить изменения в индексе
git cherry-pick -n a3f1b2

# Конфликт — решить и continue
git cherry-pick --continue
git cherry-pick --abort

Применение: срочно перенести багфикс из main в release/v1.0 без слияния всех остальных изменений.

git revert — отменить коммит безопасно

Создаёт новый коммит, который обратит изменения указанного коммита. История не переписывается.

# Отменить последний коммит
git revert HEAD

# Отменить конкретный коммит
git revert a3f1b2

# Отменить несколько без создания N коммитов
git revert --no-commit HEAD~3..HEAD
git commit -m "revert: rollback feature X"

# Revert merge-коммита (надо указать -m 1 — какого родителя считать "правильным")
git revert -m 1 <merge-commit>

git reset — переписать историю

Двигает указатель ветки на указанный коммит. Опасно для общих веток.

ФлагHEADIndexWorking tree
--softдвигаетне трогаетне трогает
--mixed (default)двигаетсбрасываетне трогает
--hardдвигаетсбрасываетсбрасывает
 
# Откатить последние 3 коммита, но оставить изменения в working tree
git reset --soft HEAD~3

# Откатить — изменения в working tree, но не staged
git reset HEAD~3

# Полный откат — потерять всё
git reset --hard HEAD~3          # ОПАСНО

# Если уже запушено — придётся force push (только в своей ветке!)
git push --force-with-lease
КомандаМеняет историюКогда применять
revertНет (создаёт новый коммит)Отменить в shared ветке
resetДаТолько в локальной/своей ветке

// 14Stash и tags

git stash — отложить изменения

«Прячет» текущие незакоммиченные изменения в стек, оставляя working tree чистым.

# Спрятать
git stash                           # только tracked файлы
git stash -u                        # + untracked
git stash -a                        # + игнорируемые
git stash push -m "work in progress on login"

# Посмотреть, что лежит
git stash list
# stash@{0}: WIP on feature/login: ...
# stash@{1}: On main: work in progress on login

git stash show stash@{0}            # краткий diff
git stash show -p stash@{0}         # полный diff

# Вернуть
git stash pop                       # вернуть последний и удалить из стека
git stash apply stash@{1}           # вернуть, НЕ удаляя из стека

# Удалить
git stash drop stash@{0}
git stash clear                     # удалить все

Применение: внезапно нужно переключиться на другую ветку для срочного фикса, а текущие изменения ещё не готовы для коммита.

git tag — метки версий

# Лёгкий тег (просто указатель)
git tag v1.0.0

# Аннотированный тег (с автором, датой, сообщением — для релизов!)
git tag -a v1.0.0 -m "Release 1.0.0"

# Тег на конкретный коммит
git tag -a v0.9.0 a3f1b2 -m "Beta release"

# Просмотр
git tag                          # все теги
git tag -l "v1.*"             # фильтр
git show v1.0.0

# Отправить теги на remote (по умолчанию push их НЕ шлёт!)
git push origin v1.0.0           # один тег
git push --tags                  # все теги

# Удалить
git tag -d v1.0.0                # локально
git push origin :refs/tags/v1.0.0  # на remote
// SemVer
Стандарт версионирования MAJOR.MINOR.PATCH: v1.4.2. MAJOR — breaking changes, MINOR — новая функциональность совместимо, PATCH — багфиксы. Подробно: semver.org/lang/ru.

// 15Reflog и восстановление потерянных коммитов

git reflog — журнал всех движений HEAD за последние ~90 дней. Это спасательный круг, когда кажется, что коммиты потеряны.

git reflog
# a3f1b2 HEAD@{0}: commit: feat: add login
# b4c2d3 HEAD@{1}: reset: moving to HEAD~3   ← вот тут я "потерял" коммиты
# c5d3e4 HEAD@{2}: commit: refactor

Восстановить потерянный коммит

# Найти хэш потерянного коммита через reflog
git reflog

# Создать ветку прямо на нём
git branch recovery c5d3e4

# Или сразу переключиться
git checkout c5d3e4
git switch -c recovery

Восстановить ветку, удалённую через branch -D

git reflog                       # найти последний коммит удалённой ветки
git branch <ветка> <хэш>         # создать заново
// Главное
Git очень редко по-настоящему теряет данные. Даже после reset --hard и branch -D коммиты живут в .git/objects/ ещё ~90 дней (до запуска git gc). Если что-то «потерялось» — git reflog.

// 16Pull Request workflow

PR (на GitHub) или MR (на GitLab) — это не команда Git, а функция платформы: «прошу влить мою ветку в основную». Включает в себя обсуждение, code review, автотесты в CI.

Полный цикл PR

  1. Sync с main: 
    git switch main
git pull
  2. Ветка под задачу: 
    git switch -c feature/INGEST-142-import-csv
  3. Работаем, коммитим часто: 
    git add -p
git commit -m "feat(ingest): parse CSV header"
git commit -m "feat(ingest): validate row types"
git commit -m "test(ingest): cover edge cases"
  4. Перед PR — подчистить и обновиться: 
    git rebase -i origin/main        # сжать "fix typo" коммиты
git rebase origin/main           # подтянуть свежий main
  5. Push и открыть PR: 
    git push -u origin feature/INGEST-142-import-csv
gh pr create --title "feat(ingest): import CSV" --body "Closes INGEST-142"
  6. Code review: ревьюеры пишут комментарии. Отвечаешь, правишь. 
    git add fixed-file
git commit --amend                 # если правка тривиальная
# или новый коммит "review fixes" — на вкус команды
git push --force-with-lease
  7. CI зелёный + апрувы → squash merge.
  8. Чистим: 
    git switch main && git pull
git branch -d feature/INGEST-142-import-csv
git fetch --prune                 # удалить локальные origin/feature/...

GitHub CLI (gh)

gh pr create                    # создать PR из текущей ветки
gh pr list                      # все открытые PR
gh pr view 42                   # детали PR
gh pr checkout 42               # локально переключиться на чужой PR
gh pr review 42 --approve
gh pr merge 42 --squash

Слияние и история

git revert <hash>                       # отменить безопасно (новый коммит)
git reflog                              # найти потерянное
git stash / git stash pop               # отложить / вернуть

Теги

git tag -d <tag> / git push origin :refs/tags/<tag>
