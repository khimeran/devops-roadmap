// 01Что такое Bash и зачем DevOps

Bash (Bourne Again SHell) — командная оболочка и язык скриптов, дефолтный shell в большинстве Linux-дистрибутивов. Bash интерпретирует команды интерактивно (когда ты пишешь их в терминал) и выполняет скрипты — текстовые файлы с последовательностью команд.

Почему это критично для DevOps

  • Entry point контейнеров. Docker ENTRYPOINT и CMD часто — bash-скрипты, готовящие окружение перед стартом приложения.
  • CI/CD pipelines. Каждый шаг GitHub Actions / GitLab CI — это shell-команда. Без bash ты ограничен готовыми actions и не можешь склеить что-то нетривиальное.
  • Ansible. Модули shell / command — это bash. Custom facts, raw-задачи, диагностика на хосте — всё bash.
  • Эксплуатация. Бэкапы, ротация логов, мониторинг, деплой, healthcheck'и — это десятки коротких скриптов в /usr/local/bin.
  • Глубже понять Linux. Каждый скрипт — это файлы, процессы, переменные окружения, права. Bash — это «клей» между утилитами.

Bash vs sh vs zsh

ОболочкаГде встречаетсяОсобенности

bashUbuntu, Debian, CentOS, macOS (старый)Стандарт DevOps. Массивы, [[ ]], process substitution

// Важно
Если первая строка скрипта #!/bin/sh, скрипт исполнится оболочкой sh, даже если ты разрабатывал его в bash. На Alpine это /bin/sh → busybox ash, где половина bash-синтаксиса не работает. Для bash всегда пиши #!/usr/bin/env bash.

// 02Shebang, права, запуск скрипта

Структура скрипта

#!/usr/bin/env bash
# script.sh — описание что делает скрипт
# Автор: kirill@devops

set -euo pipefail   # строгий режим, см. секцию 10

echo "Hello, $USER"

Shebang — первая строка

Это «магическая» строка #!path, говорящая ядру какой интерпретатор запускать. Без неё ядро попытается выполнить файл как ELF-бинарь и упадёт.

  • #!/bin/bash — старый стиль. Работает, но в некоторых системах (BSD, минимальный busybox) bash лежит не в /bin.
  • #!/usr/bin/env bash — современный и портативный. env найдёт bash в PATH.
  • #!/bin/sh — POSIX-совместимый shell. Используй когда скрипт должен работать в Alpine / контейнерах без bash.

Права на исполнение

chmod +x script.sh        # дать всем право исполнять
chmod 755 script.sh       # rwxr-xr-x — владелец полные, остальные читать/исполнять
ls -l script.sh
# -rwxr-xr-x  1 kirill kirill  256 may 16 12:00 script.sh

Способы запуска

# 1. С shebang и +x — обычный запуск
./script.sh

# 2. Явно через интерпретатор (shebang и +x не нужны)
bash script.sh

# 3. source — выполнить в ТЕКУЩЕМ shell (переменные останутся после)
source script.sh
. script.sh                  # синоним

# 4. Если скрипт лежит в каталоге из PATH — просто по имени
script.sh
// Ловушка
./script.sh запускает скрипт в дочернем процессе, поэтому переменные/cd внутри него не повлияют на твою сессию. Если нужно повлиять на текущий shell (например, активировать venv) — используй source.

// 03Переменные и типы данных

Объявление

name="kirill"           # ПРОБЕЛОВ ВОКРУГ = БЫТЬ НЕ ДОЛЖНО!
age=29
greeting="Hello, $name"

echo "$name"            # kirill
echo "${name}_admin"     # kirill_admin (фигурные скобки для границ)

Bash не типизирован — но есть declare

Любая переменная в bash — это строка. Но можно подсказать bash трактовать её иначе:

declare -i  count=10         # integer — целое число
declare -r  PI=3.14           # readonly — не перезаписать
declare -a  arr               # indexed array
declare -A  map               # associative array (Bash 4+)
declare -x  EXPORTED="value"  # экспорт в окружение, = export

Окружение и export

FOO="bar"                  # локальная переменная shell, дочерним процессам НЕ видна
export FOO="bar"           # теперь видна всем дочерним процессам
env | grep FOO            # посмотреть все экспортированные
unset FOO                   # удалить

Массивы

# Indexed array
fruits=("apple" "banana" "cherry")
echo "${fruits[0]}"         # apple
echo "${fruits[@]}"         # все элементы
echo "${#fruits[@]}"        # длина (3)
fruits+=("date")             # добавить
unset "fruits[1]"           # удалить элемент

# Associative array (как dict)
declare -A server
server[host]="prod-01"
server[ip]="10.0.0.5"
echo "${server[host]}"       # prod-01
echo "${!server[@]}"         # все ключи: host ip

Подстановка с дефолтами и проверками

echo "${VAR:-default}"       # если VAR пуст — вернуть "default"
echo "${VAR:=default}"       # если VAR пуст — присвоить "default"
echo "${VAR:?обязательная переменная}"  # упасть с ошибкой если пусто
echo "${VAR:+есть значение}"  # вернуть строку ТОЛЬКО если VAR не пуст

# Манипуляции со строкой
path="/var/log/nginx.log"
echo "${path##*/}"           # nginx.log  (после последнего /)
echo "${path%.*}"            # /var/log/nginx  (до последней .)
echo "${path/log/LOG}"        # замена первого совпадения
echo "${path//log/LOG}"       # замена всех совпадений
echo "${#path}"              # длина строки
// Совет
Используй "${VAR:?описание}" в начале скрипта чтобы гарантировать что важные переменные заданы. Падать рано — лучше, чем работать с пустым значением и грохнуть продакшен.

// 04Кавычки и подстановка

Кавычки — самая частая ошибка новичков. Правило: всегда заключай переменные в двойные кавычки, кроме случаев когда осознанно хочешь обратное.

Двойные кавычки "..."

name="Kirill Eagle"
echo "Привет, $name"        # Привет, Kirill Eagle
echo "Сегодня $(date +%F)"   # Подстановка команды работает

Сохраняют переменные, command substitution, экранирование. Это дефолт.

Одинарные кавычки '...'

echo '$name стоит $100'      # дословно: $name стоит $100

Литерал. Никаких подстановок. Используй для регулярок, awk-программ, jq-фильтров.

Без кавычек — опасно

file="my report.txt"
rm $file        # BUG! Удалит файлы 'my' и 'report.txt' (word splitting)
rm "$file"      # OK: один файл "my report.txt"

Command substitution

now=$(date +%Y-%m-%d)
echo "Бэкап: backup-$now.tar.gz"

# Старый синтаксис (не используй):
now=`date +%Y-%m-%d`         # работает, но плохо вкладывается

Arithmetic substitution

echo $((2 + 3 * 4))           # 14
count=5
total=$((count * 2 + 1))     # 11

# Внутри (( )) переменные без $ — это удобно
if (( count > 3 )); then
  echo "много"
fi

Brace expansion и pathname expansion

echo file{1,2,3}.txt          # file1.txt file2.txt file3.txt
echo {01..05}                 # 01 02 03 04 05
echo {a..e}                   # a b c d e

ls *.log                     # pathname expansion (glob)
ls log_2025-??-??.txt        # ? = один символ

// 05Условия: if / elif / else

Базовый синтаксис

if [[ "$USER" == "root" ]]; then
  echo "вы root"
elif [[ "$USER" == "kirill" ]]; then
  echo "привет, шеф"
else
  echo "кто ты вообще"
fi

[ vs [[ vs (( — какую скобку выбрать

КонструкцияДля чегоЗаметки
[ ... ]POSIX test, классикаСтаро, требует кавычек на переменные
[[ ... ]]Bash-расширение для строк/файловДефолт в bash. Безопаснее, поддержка regex и логических операторов
(( ... ))АрифметикаСравнения чисел: < > == != <= >=

Сравнение строк

if [[ "$a" == "$b" ]]; ...   # равны
if [[ "$a" != "$b" ]]; ...   # не равны
if [[ -z "$a" ]]; ...        # пустая строка
if [[ -n "$a" ]]; ...        # НЕ пустая
if [[ "$a" == prod* ]]; ...   # wildcard
if [[ "$a" =~ ^[0-9]+$ ]]; ...# regex (только с [[ ]])

Сравнение чисел

if (( count > 10 )); ...       # современный
if [[ "$count" -gt 10 ]]; ...   # аналог через -gt

# Операторы -gt -lt -ge -le -eq -ne

Проверка файлов

if [[ -f "/etc/passwd" ]]; ...    # обычный файл существует
if [[ -d "/var/log" ]]; ...       # каталог
if [[ -e "/path" ]]; ...          # существует любой тип
if [[ -r "$file" ]]; ...          # читаемый
if [[ -w "$file" ]]; ...          # писeмый
if [[ -x "$file" ]]; ...          # исполняемый
if [[ -s "$file" ]]; ...          # непустой
if [[ "$a" -nt "$b" ]]; ...      # a новее b

Логические операторы

if [[ -f "$f" && -r "$f" ]]; ...
if [[ "$env" == prod || "$env" == staging ]]; ...
if ! [[ -d "$dir" ]]; ...      # отрицание

Краткая форма

mkdir -p logs && echo "ok"   # && — если предыдущее вернуло 0
test -f file.txt || touch file.txt  # || — если вернуло ненулевой
// Ловушка
[ $a == $b ] без кавычек упадёт если $a пустой или содержит пробелы. Внутри [[ ]] это безопаснее, но кавычки всё равно ставь — это спасёт от глобинга.

// 06Конструкция case

Когда условий 3+ и они проверяют одну и ту же переменную — case читается намного лучше чем if/elif.

case "$1" in
  start)
    echo "запускаем сервис"
    ;;
  stop)
    echo "останавливаем"
    ;;
  restart|reload)
    echo "перезапуск"
    ;;
  status)
    echo "статус"
    ;;
  *)
    echo "usage: $0 {start|stop|restart|status}"
    exit 1
    ;;
esac
  • ;; — конец ветки (как break в C).
  • | — несколько паттернов в одной ветке.
  • * — fallback (как default).
  • Поддерживается glob: prod-*, ?.txt, [0-9]*.

Реальный пример — обработка окружения

env="${1:-dev}"
case "$env" in
  prod*)    api="https://api.example.com" ;;
  stag*)    api="https://staging.example.com" ;;
  dev|local) api="http://localhost:8080" ;;
  *)        echo "Неизвестное окружение: $env"; exit 1 ;;
esac
echo "API: $api"

// 07Циклы: for, while, until

for — по списку

for name in alice bob charlie; do
  echo "Hello, $name"
done

# По массиву
servers=("web-01" "web-02" "db-01")
for srv in "${servers[@]}"; do
  ssh "$srv" uptime
done

# По файлам
for f in /var/log/*.log; do
  echo "Размер $f: $(stat -c%s "$f") байт"
done

for — C-style

for ((i=0; i<5; i++)); do
  echo "итерация $i"
done

seq и диапазоны

for i in $(seq 1 5); do echo "$i"; done
for i in {1..5}; do echo "$i"; done      # brace expansion — быстрее
for i in {0..20..2}; do echo "$i"; done   # с шагом 2

while — пока условие истинно

count=0
while (( count < 5 )); do
  echo "count=$count"
  ((count++))
done

# Бесконечный цикл (выход через break / сигнал)
while true; do
  curl -s healthcheck.url >/dev/null || echo "DOWN"
  sleep 10
done

while read — построчное чтение файла

# Шаблон, который встретишь сотни раз
while IFS= read -r line; do
  echo "строка: $line"
done < /etc/hosts

# С разделителем (CSV)
while IFS=, read -r host ip port; do
  echo "$host -> $ip:$port"
done < servers.csv

until — пока условие ложно

# Подождать пока сервис не поднимется
until curl -sf http://localhost:8080/health; do
  echo "ждём..."
  sleep 2
done
echo "готово"

break и continue

for srv in "${servers[@]}"; do
  ping -c1 "$srv" &>/dev/null || continue   # не отвечает — следующий
  if [[ "$srv" == "web-99" ]]; then
    break                                  # нашли — выходим
  fi
done
// Ловушка
for f in $(ls *.log) сломается на именах с пробелами. Используй for f in *.log — bash сам подставит файлы безопасно. Или find ... -print0 | xargs -0 для рекурсии.

Объявление и вызов

greet() {

}

# Способ 2 — bash-стиль
function greet {

Аргументы функции

Внутри функции $1 $2 ... $@ $# — это аргументы функции, а не скрипта.

deploy() {
  local env="$1"
  local version="$2"
  echo "Деплой $version в $env"
  echo "Всего аргументов: $#"
}

deploy prod v1.2.3

local и scope

x="глобально"

set_x() {
  local x="локально"     # ВАЖНО: иначе перезапишет глобальную
  echo "внутри: $x"
}

set_x
echo "снаружи: $x"      # "глобально"

Возврат значения

В bash функция не возвращает строку. return N — это код возврата (0–255). Чтобы «вернуть» значение — выводи в stdout и забирай через command substitution.

get_count() {
  echo 42                # «возврат» через stdout
}

n=$(get_count)
echo "got $n"

# Для статуса успех/провал — return
is_root() {
  [[ "$EUID" -eq 0 ]] && return 0 || return 1
}

if is_root; then echo "root"; fi

Передача массива в функцию

print_items() {
  local arr=("$@")
  for i in "${arr[@]}"; do
    echo "- $i"
  done
}

fruits=(apple banana cherry)
print_items "${fruits[@]}"
// Совет
Всегда используй local для всех переменных внутри функции — иначе скрипт превращается в кашу из глобальных переменных, и баги ловятся месяцами.

Минимальный шаблон скрипта с аргументами

#!/usr/bin/env bash
set -euo pipefail

if [[ $# -lt 2 ]]; then
  echo "Usage: $0 <env> <version>" >&2
  exit 1
fi

env="$1"
version="$2"
shift 2                           # «сдвинуть» — $3 становится $1

extra_args=("$@")                 # всё что осталось
echo "deploy env=$env ver=$version extras=${extra_args[*]}"

$@ vs $* — разница

show() {
  echo "@: $#"
  for a in "$@"; do echo "  [$a]"; done
  echo "*:"
  for a in "$*"; do echo "  [$a]"; done
}
show "hello world" "foo bar"
# "$@" сохранит границы — 2 элемента
# "$*" склеит — 1 элемент "hello world foo bar"

getopts — короткие флаги (-h, -v, -f)

while getopts ":he:v:f:" opt; do
  case "$opt" in
    h) echo "usage: $0 -e ENV -v VER -f FILE"; exit 0 ;;
    e) env="$OPTARG" ;;
    v) version="$OPTARG" ;;
    f) file="$OPTARG" ;;
    \?) echo "Неизвестный флаг: -$OPTARG" >&2; exit 1 ;;
    :)  echo "Флаг -$OPTARG требует значение" >&2; exit 1 ;;
  esac
done
shift $((OPTIND - 1))

echo "env=$env ver=$version file=$file"

Длинные опции (--flag) — ручной парсинг

while [[ $# -gt 0 ]]; do
  case "$1" in
    --env)     env="$2"; shift 2 ;;
    --version) version="$2"; shift 2 ;;
    --dry-run) dry_run=1; shift ;;
    -h|--help) echo "usage..."; exit 0 ;;
    *)         echo "Неизвестный аргумент: $1" >&2; exit 1 ;;
  esac
done
// Заметка
Для серьёзных CLI пиши на Python с argparse или на Go с cobra. Bash хорош до ~150 строк — потом разрастается и становится непроверяемым.

// 10Коды возврата $?, set -e, trap

Что такое exit code

Любая команда возвращает число 0–255. 0 = успех, всё остальное = ошибка. Это основа всех проверок и пайплайнов.

ls /etc >/dev/null
echo $?                  # 0

ls /nope 2>/dev/null
echo $?                  # 2 (No such file)

Типичные коды

КодЗначение
0Успех
1Общая ошибка
2Неверный синтаксис / отсутствует файл
126Команда найдена, но не исполняема
127Команда не найдена (опечатка в имени)
128+NЗавершено сигналом N (130 = Ctrl+C / SIGINT)
137Убит SIGKILL (часто — OOM)

Принудительный выход

if [[ -z "$API_KEY" ]]; then
  echo "API_KEY обязателен" >&2
  exit 1
fi

set -e, set -u, set -o pipefail — святая троица

set -euo pipefail
# -e          выйти при первой ошибке
# -u          ошибка на использование неопределённой переменной
# -o pipefail в "a | b | c" статус берётся от первого упавшего, а не последнего

Эта строка должна быть в каждом твоём скрипте — это +90% надёжности. По умолчанию bash «прощает» ошибки и едет дальше, что приводит к тихим багам в продакшене.

set -x — debug

set -x          # включить трассировку — каждая команда печатается перед запуском
ls /etc
set +x          # выключить

# Или из консоли:
bash -x script.sh

trap — обработка сигналов и cleanup

tmp=$(mktemp)
trap 'rm -f "$tmp"; echo "cleanup done"' EXIT

# EXIT срабатывает при любом выходе — нормальном, через exit, через сигнал

# Несколько сигналов:
trap 'echo "получен SIGINT"; exit 130' INT
trap 'echo "получен SIGTERM"; exit 143' TERM
// Шаблон скрипта
Любой production-скрипт начинай так: #!/usr/bin/env bash → set -euo pipefail → trap cleanup EXIT. Это сэкономит тебе десятки часов отладки.

// 11Работа с файлами и read

Чтение файла построчно

while IFS= read -r line; do
  echo "> $line"
done < /etc/hosts
  • IFS= — отключить разделитель, чтобы строки не «обрезались».
  • -r — не интерпретировать backslash escape'ы.
  • < file — перенаправить файл на stdin цикла.

Чтение из пользовательского ввода

read -rp "Имя сервера: " server
echo "вы ввели: $server"

read -rsp "Пароль: " pass   # -s — не показывать ввод
echo

read -rp "OK? (y/N) " ans
[[ "$ans" =~ ^[Yy]$ ]] || { echo "отмена"; exit 1; }

Создание и запись

echo "строка 1"  >  out.txt    # перезапись
echo "строка 2"  >> out.txt    # дописать

# Heredoc — многострочный текст
cat > config.yml <<EOF
server: web-01

EOF

# <<'EOF' (в кавычках) — БЕЗ подстановки переменных
cat > raw.txt <<'EOF'
$HOME останется буквальным
EOF

Временные файлы

tmp=$(mktemp)
tmpdir=$(mktemp -d)
trap 'rm -rf "$tmp" "$tmpdir"' EXIT

# mktemp создаст уникальное имя в /tmp — безопаснее чем хардкодить

Базовые утилиты для файлов

cat file.txt                  # вывести
head -n 20 file.txt           # первые 20 строк
tail -n 20 file.txt           # последние 20
tail -f /var/log/syslog       # следить за дописываемым логом

cut -d: -f1 /etc/passwd       # первый столбец, разделитель :
tr 'a-z' 'A-Z' < file.txt    # верхний регистр

// 12Pipe и перенаправление потоков

Три стандартных потока

FDИмяЧто это
0stdinВвод (обычно клавиатура)
1stdoutОбычный вывод
2stderrОшибки и диагностика

Перенаправление

cmd > out.txt              # stdout в файл (перезапись)
cmd >> out.txt             # stdout в файл (дописать)
cmd 2> err.txt             # stderr в файл
cmd > out.txt 2> err.txt   # раздельно
cmd > all.txt 2>&1         # stderr В stdout, оба в файл
cmd &> all.txt             # bash-сокращение для предыдущего
cmd < input.txt            # файл на stdin
cmd > /dev/null            # выкинуть stdout
cmd &> /dev/null            # выкинуть оба

Pipe — соединение команд

ps aux | grep nginx | grep -v grep | awk '{print $2}'

# stdout первой команды становится stdin следующей.
# Каждая команда в pipe выполняется в подпроцессе!

Особые операторы перенаправления

# Here-string — короткая «подача» строки
grep kirill <<< "$line"

# Process substitution — подсунуть вывод как «файл»
diff <(ls dir1) <(ls dir2)

# tee — записать в файл И вывести дальше
cmd | tee out.log | grep ERROR
cmd | tee -a all.log         # append вместо overwrite

Запуск в фоне

long_task &             # в фон, $! = PID
echo "PID: $!"

wait $!                 # дождаться завершения

# Параллельный запуск над списком
for srv in "${servers[@]}"; do
  ping_check "$srv" &
done
wait                    # дождаться ВСЕХ дочерних
// Ловушка pipefail
Без set -o pipefail код возврата пайплайна = код последней команды. false | true вернёт 0, и ошибка false потеряется. Всегда включай pipefail.

// 13grep — поиск по тексту

grep ищет строки по шаблону (регулярное выражение). Это первая утилита, к которой ты потянешься 100500 раз в день.

Базовое

grep "error" app.log          # все строки со словом error
grep -i "error" app.log       # без учёта регистра
grep -v "DEBUG" app.log       # инвертировать — все КРОМЕ DEBUG
grep -c "error" app.log       # только количество совпадений
grep -n "error" app.log       # с номерами строк
grep -l "error" *.log         # только имена файлов с совпадением

Контекст

grep -A 3 "error" app.log     # 3 строки после
grep -B 3 "error" app.log     # 3 строки до
grep -C 3 "error" app.log     # 3 строки до И после

Регулярки

grep -E "^ERROR|FATAL" app.log      # -E = extended regex (alt: egrep)
grep -E "^[0-9]{4}-[0-9]{2}-[0-9]{2}" log  # строки начинающиеся с даты
grep -P "\b[A-Z0-9]+\b" file       # -P = Perl-style (доступно в GNU grep)
grep -F "1.2.3.4" log             # -F = фиксированная строка, регулярки не парсятся

Рекурсия

grep -r "TODO" ./src              # рекурсивно
grep -rn --include="*.py" "def main" .   # только в .py
grep -r --exclude-dir=node_modules "key" .

Реальный пример

# Топ-10 IP в access.log
grep -oE "^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+" access.log \
  | sort | uniq -c | sort -rn | head -10
// Альтернатива
Для рекурсивного поиска в коде попробуй ripgrep (rg) — в 10–50 раз быстрее, по умолчанию игнорирует .gitignore. Не везде установлено, но если ты часто шаришь по большим репозиториям — поставь.

// 14sed — потоковый редактор

sed читает текст построчно, применяет команды и выводит результат. Идеально для подстановок и быстрых правок файлов.

Замена строк

sed 's/old/new/'  file.txt        # первое вхождение в строке
sed 's/old/new/g' file.txt        # все вхождения
sed 's/old/new/gi' file.txt       # + case insensitive

# Изменить файл на месте (in-place)
sed -i 's/old/new/g' file.txt
sed -i.bak 's/old/new/g' file.txt   # с бэкапом file.txt.bak

# macOS BSD-sed требует пустую строку после -i:
sed -i '' 's/old/new/g' file.txt

Другие разделители

# Когда в шаблоне есть /, бери другой разделитель
sed 's|/var/log|/tmp|g' file
sed 's#http://old.com#https://new.com#g' file

Удаление и адресация

sed '3d' file              # удалить 3-ю строку
sed '/^#/d' file           # удалить строки начинающиеся с #
sed '/^$/d' file           # удалить пустые строки
sed -n '10,20p' file       # напечатать только строки 10–20 (-n = тишина)
sed '1,3s/foo/bar/' file   # замена только в строках 1–3

Группы и обратные ссылки

# Поменять местами слова
echo "kirill devops" | sed -E 's/(\w+) (\w+)/\2 \1/'
# devops kirill

# Извлечь номер версии
echo "app version 1.2.3 released" | sed -E 's/.*version ([0-9.]+).*/\1/'
# 1.2.3

Несколько команд

sed -e 's/foo/bar/g' -e 's/baz/qux/g' file
sed 's/foo/bar/g; s/baz/qux/g' file
// Различия GNU и BSD
На Linux — GNU sed, на macOS — BSD sed. -i у них работает по-разному (BSD требует параметр). Если пишешь скрипт для обеих платформ — используй sed -i.bak и потом rm бэкап.

// 15awk — обработка колонок

awk — это мини-язык программирования. Читает текст построчно, разбивает каждую строку на поля и применяет программу. Идеален для логов, CSV, табличных данных.

Базовое

awk '{print $1}' file              # первое поле каждой строки
awk '{print $1, $3}' file          # 1-е и 3-е
awk '{print NF}' file              # количество полей в строке
awk '{print NR, $0}' file          # NR = номер строки, $0 = вся строка

Разделитель полей

awk -F: '{print $1}' /etc/passwd        # разделитель :
awk -F, '{print $1, $3}' data.csv       # CSV
awk 'BEGIN{FS=":"; OFS=" -> "} {print $1, $7}' /etc/passwd
# FS — input separator, OFS — output

Условия

awk '$3 > 100 {print $0}' file              # строки где 3-е поле > 100
awk '/error/ {print $1, $2}' log            # только строки с "error"
awk 'NR > 1 {print}' file                   # пропустить заголовок CSV
awk 'length($0) > 80 {print NR": "$0}' code # длинные строки

Агрегация — реальные задачи

# Сумма значений колонки
awk '{sum += $1} END {print sum}' numbers.txt

# Среднее
awk '{sum += $1; n++} END {print sum/n}' numbers.txt

# Группировка — подсчёт IP в логах
awk '{ips[$1]++} END {for (ip in ips) print ips[ip], ip}' access.log \
  | sort -rn | head

# Топ ошибок
awk '/ERROR/ {count[$4]++} END {for (k in count) print count[k], k}' app.log

BEGIN и END блоки

awk '
  BEGIN { print "=== START ===" }
  /error/ { errors++ }
  /warn/  { warns++ }
  END {
    print "errors:", errors
    print "warns:",  warns
  }
' app.log
// Правило выбора
grep — найти строки. sed — заменить текст. awk — разобрать по полям и посчитать. Если ты пишешь grep | awk — часто достаточно одного awk: awk '/pattern/ {print $2}'.

cron — классика

Демон cron запускает задачи по расписанию. Конфиг — таблица crontab из 5 колонок времени + команда.

systemd timers — современная замена

В современных дистрибутивах cron часто заменяют на systemd timers. Плюсы: интеграция с журналом (journalctl), управление зависимостями, удобный синтаксис.

Нужно два файла: service (что запускать) и timer (когда).

# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh

# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily at 03:00

[Timer]
OnCalendar=*-*-* 03:00:00
Persistent=true

[Install]
WantedBy=timers.target

systemctl daemon-reload
systemctl enable --now backup.timer
systemctl list-timers       # все таймеры
journalctl -u backup.service # логи выполнения
cronsystemd timer

// 17Best practices и ShellCheck

Скелет «правильного» скрипта

#!/usr/bin/env bash
#
# backup.sh — ежедневный бэкап /var/data
# Usage: backup.sh [--dry-run]

set -euo pipefail
IFS=$'\n\t'                                  # безопасный IFS

readonly SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
readonly LOG="/var/log/backup.log"

log()  { echo "[$(date +%F\ %T)] $*" | tee -a "$LOG"; }
die()  { echo "FATAL: $*" >&2; exit 1; }

cleanup() {
  local rc=$?
  log "cleanup, exit=$rc"
  rm -rf "${tmp:-}"
}
trap cleanup EXIT

main() {
  tmp=$(mktemp -d) || die "mktemp failed"
  log "backup started in $tmp"
  # ...
  log "done"
}

main "$@"

Правила, которые сэкономят тебе нервы

  1. Всегда set -euo pipefail в начале.
  2. Всегда кавычки вокруг переменных: "$var", "$@", "${arr[@]}".
  3. Имена переменных: UPPER_CASE для констант/окружения, lower_case для локальных.
  4. Функции — всё через local.
  5. Проверки в начале: что нужные команды есть (command -v jq >/dev/null || die "jq missing"), что переменные заданы.
  6. Не пиши скрипты длиннее ~200 строк — переходи на Python.
  7. Логируй с timestamp в файл, ошибки в stderr (>&2).
  8. Идемпотентность: запуск дважды должен давать тот же результат.

ShellCheck — обязательный линтер

# Установка
sudo apt install shellcheck
brew install shellcheck

# Запуск
shellcheck script.sh

ShellCheck подскажет почти все типичные ошибки: незакавыченные переменные, неэкранированные регулярки, cd "$dir" || exit и т.д. Запускай его перед каждым коммитом — в идеале как pre-commit hook или в CI.

// Главное правило
Bash — это инструмент для «склейки» утилит, не для бизнес-логики. Когда начинают появляться вложенные структуры, JSON, парсинг — переходи на Python. Bash для оркестрации в 50 строк, не для приложений.
