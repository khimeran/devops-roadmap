1. Обзор проекта

В данном руководстве описан полный процесс развёртывания Python FastAPI приложения на локальном хосте под управлением Fedora 44 с использованием Docker. 2. Подготовка — клонирование репозитория

Первый шаг при работе с чужим приложением — получить исходный код. Мы использовали Git для клонирования репозитория с GitHub.

2.1 Команды

cd ~

git clone https://github.com/iboiarkin96/etr_study_api.git

cd etr_study_api

ls -la

2.2 Разбор команд

cd ~ — переход в домашнюю директорию пользователя. Символ ~ является сокращением для /home/имя_пользователя. Это удобное место для хранения проектов.

git clone URL — скачивает репозиторий со всеми файлами, историей коммитов и ветками. Создаёт папку с именем репозитория.

ls -la — показывает список файлов с подробной информацией:

  • -l — длинный формат (права, владелец, размер, дата)

  • -a — показывать скрытые файлы (начинающиеся с точки)

2.3 Что изучить в первую очередь в незнакомом репозитории

При получении незнакомого приложения всегда смотрите в следующем порядке:

  1. README.md — главный документ проекта. Там описано как запускать, какие зависимости нужны

  2. Dockerfile — инструкция по сборке образа. Показывает базовый образ, зависимости, порт

  3. docker-compose.yml — какие сервисы нужны (БД, кэш, очереди)

  4. requirements.txt / package.json / go.mod — зависимости приложения

  5. env/example или .env.example — переменные окружения которые нужно настроить

  6. scripts/ — вспомогательные скрипты запуска и настройки

3. Настройка переменных окружения

Переменные окружения — это настройки приложения которые передаются извне, без изменения кода. Это стандартная практика для конфигурирования приложений в разных средах (dev, qa, prod).

3.1 Создание .env файла

cp env/example .env

cp — команда копирования файлов. Мы копируем шаблон env/example в файл .env который будет использоваться приложением.

⚠️ Файл .env содержит секреты (пароли, ключи API). Никогда не добавляйте его в Git! Всегда проверяйте что .env есть в .gitignore.

3.2 Ключевые переменные проекта

Переменная Значение Описание
APP_HOST 0.0.0.0 Слушать на всех интерфейсах (не только localhost)
APP_PORT 8000 Порт приложения
APP_ENV dev Окружение: dev, qa, prod
SQLITE_DB_PATH /data/study_app.db Путь к файлу базы данных
LOG_LEVEL INFO Уровень логирования
METRICS_ENABLED true Включить метрики Prometheus
GRAFANA_ADMIN_USER admin Логин Grafana
GRAFANA_ADMIN_PASSWORD admin Пароль Grafana (сменить в продакшне!)

⚠️ APP_HOST=127.0.0.1 означает что приложение доступно только с самого сервера. Для доступа из сети нужно 0.0.0.0.

4. Сборка Docker образа

Docker образ — это упакованное приложение со всеми зависимостями. Образ неизменяем — из него создаются контейнеры.

4.1 Команда сборки

docker build -t study-app-api:local .

4.2 Разбор параметров

Параметр Описание
docker build Команда сборки образа
-t study-app-api:local -t означает tag (метка). Формат: имя:версия. local — наша локальная версия
. Контекст сборки — текущая директория. Docker ищет Dockerfile здесь

4.3 Разбор Dockerfile приложения

FROM python:3.11-slim-bookworm

Базовый образ. slim — облегчённая версия без лишних пакетов (~150 МБ вместо ~900 МБ). bookworm — кодовое имя Debian 12.

ENV PYTHONUNBUFFERED=1 PYTHONDONTWRITEBYTECODE=1 PIP_NO_CACHE_DIR=1

Устанавливает переменные окружения для оптимизации Python в контейнере:

  • PYTHONUNBUFFERED=1 — вывод сразу в терминал без буферизации. Важно для логов в Docker

  • PYTHONDONTWRITEBYTECODE=1 — не создавать .pyc файлы. Не нужны в контейнере, уменьшают образ

  • PIP_NO_CACHE_DIR=1 — не кэшировать пакеты pip. Уменьшает размер образа

RUN useradd --create-home --uid 1000 appuser

Создаёт непривилегированного пользователя. Никогда не запускайте приложения от root в контейнере — это угроза безопасности.

COPY requirements.txt .

RUN pip install --upgrade pip && pip install -r requirements.txt

Сначала копируем только requirements.txt и устанавливаем зависимости. Хитрость: Docker кэширует слои. Если код изменился но requirements.txt нет — этот слой берётся из кэша и сборка намного быстрее.

EXPOSE 8000

Документирует какой порт использует приложение. НЕ открывает порт реально — это только метаданные для разработчиков.

4.4 Концепция слоёв Docker

Образ состоит из слоёв — каждая команда в Dockerfile создаёт новый слой. Слои кэшируются. Правило: сначала то что меняется редко (зависимости), потом то что меняется часто (код).

Слой Команда Кэшируется?
1 FROM python:3.11-slim Да, пока не меняем базовый образ
2 RUN useradd appuser Да
3 COPY requirements.txt Да, пока файл не изменился
4 RUN pip install Да, пока requirements.txt не изменился
5 COPY app/ Нет, код меняется часто

4.5 Свой образ vs чужой — когда что использовать

Тип Когда использовать Риски
Собственный (docker build) Своё приложение, кастомизация Нужно следить за обновлениями
Официальный (docker pull) nginx, postgres, redis, prometheus Минимальные — проверено сообществом
Сторонний неизвестный Никогда в продакшне Вредоносный код, уязвимости

⚠️ Всегда используйте конкретные версии образов: postgres:16.2-alpine, а не postgres:latest. В продакшне latest может неожиданно обновиться и сломать приложение.

5. Запуск приложения

5.1 Команда запуска контейнера

docker run -d \

--name study-app \

-p 8000:8000 \

--env-file .env \

study-app-api:local

5.2 Разбор параметров

Параметр Описание
docker run Создать и запустить контейнер из образа
-d Detached mode — запустить в фоне. Без этого контейнер блокирует терминал
--name study-app Дать контейнеру имя. Без имени Docker даёт случайное вроде silly_einstein
-p 8000:8000 Пробросить порт. Формат: порт_хоста:порт_контейнера
--env-file .env Передать все переменные из файла .env в контейнер
study-app-api:local Имя образа из которого создаём контейнер

5.3 Проверка работоспособности

curl http://localhost:8000/live

Ожидаемый ответ: {"status":"alive","app_env":"dev"}

curl http://localhost:8000/ready

Ожидаемый ответ: {"status":"ready","checks":{"database":"ok"}}

💡 Эндпоинты /live и /ready — стандарт для production приложений. /live проверяет что процесс жив, /ready что приложение готово принимать запросы (БД подключена и т.д.).

6. Запуск мониторинга

Стек мониторинга включает Prometheus (сбор метрик), Grafana (визуализация) и Blackbox Exporter (проверка доступности). Все три сервиса описаны в docker-compose.observability.yml.

6.1 Команда запуска

docker compose -f docker-compose.observability.yml up -d

6.2 Разбор параметров

Параметр Описание
docker compose Инструмент для управления несколькими контейнерами
-f docker-compose.observability.yml -f (file) — указать конкретный файл. По умолчанию ищет docker-compose.yml
up Создать и запустить все сервисы из файла
-d Detached — запустить в фоне

6.3 Другие полезные команды Docker Compose

Команда Описание
docker compose ps Статус всех сервисов
docker compose logs -f Следить за логами всех сервисов
docker compose logs -f prometheus Логи конкретного сервиса

7. Проблемы и их решения

В процессе развёртывания мы столкнулись с рядом проблем характерных для Linux и Docker. Каждая проблема подробно описана ниже.

7.1 Проблема: Permission Denied на конфигурационных файлах

Симптомы

err="open /etc/prometheus/prometheus.yml: permission denied"

Prometheus и Blackbox Exporter падали с ошибкой доступа к файлам конфигурации которые мы монтировали через volumes.

Причина

Docker контейнеры запускаются от пользователя с другим UID (65534 — nobody для Prometheus). Файлы принадлежали root и не имели прав на чтение для других пользователей.

Диагностика

docker logs study-app-prometheus

ls -la ops/prometheus/

В логах ищите: permission denied, cannot open, access denied

Решение 1 — изменить права доступа

chmod -R 644 ops/prometheus/

chmod -R 644 ops/grafana/

chmod — изменить права доступа к файлам. -R — рекурсивно. 644 — владелец: чтение+запись, остальные: только чтение.

Решение 2 — изменить SELinux контекст (специфично для Fedora/RHEL)

sudo chcon -Rt svirt_sandbox_file_t ops/

chcon — изменить SELinux контекст. -R — рекурсивно. svirt_sandbox_file_t — тип который разрешает Docker контейнерам читать файлы хоста.

⚠️ На Ubuntu/Debian SELinux обычно не установлен — там достаточно только chmod. На Fedora/RHEL/CentOS нужны оба шага.

7.2 Проблема: host.docker.internal не работает на Linux

Симптомы

Prometheus не мог достучаться до приложения через адрес host.docker.internal который прописан в конфиге.

Причина

host.docker.internal — специальный DNS адрес который Docker создаёт только на Mac и Windows для обращения к хост-машине из контейнера. На Linux этот адрес не создаётся автоматически.

Решение

Заменить host.docker.internal на реальный IP адрес хост-машины в файле ops/prometheus/prometheus.yml:

targets: ["192.168.0.108:8000"]

Узнать IP хост-машины: ip a | grep inet

ОС host.docker.internal Решение
macOS Работает Ничего не нужно
Windows Работает Ничего не нужно
Linux НЕ работает Использовать реальный IP или --add-host
Linux (альтернатива) НЕ работает docker run --add-host=host.docker.internal:host-gateway

7.3 Проблема: приложение недоступно по внешнему IP

Симптомы

Приложение работало через localhost но не было доступно по IP адресу 192.168.0.108 из браузера Fedora. С телефона всё работало нормально.

Причина

Особенность маршрутизации на Linux — обращение к собственному внешнему IP с самого хоста проходит иначе чем с других устройств. Это нормальное поведение.

Диагностика

ss -tlnp | grep 8000

Если видите 0.0.0.0:8000 — приложение слушает на всех интерфейсах и доступно снаружи.

sudo firewall-cmd --list-all

Проверить открытые порты в firewall.

Решение

sudo firewall-cmd --zone=FedoraWorkstation --permanent --add-port=8000/tcp

sudo firewall-cmd --reload

💡 Если приложение доступно с телефона но не с самого хоста по внешнему IP — это нормально. Используйте localhost для локальной проверки.

7.4 Проблема: SSH на нестандартном порту блокируется SELinux

Симптомы

status=255/EXCEPTION

SSH демон не мог запуститься после смены порта с 22 на 2222.

Причина

SELinux знает какие порты разрешены для каждого сервиса. Порт 2222 не был в списке разрешённых для SSH.

Решение

sudo semanage port -a -t ssh_port_t -p tcp 2222

semanage — управление SELinux политиками. port -a — добавить порт. -t ssh_port_t — тип порта для SSH. -p tcp 2222 — протокол и номер порта.

8. SELinux — подробное руководство

SELinux (Security-Enhanced Linux) — система принудительного контроля доступа. Присутствует на Fedora, RHEL, CentOS. На Ubuntu/Debian используется AppArmor — похожая но другая система.

8.1 Основные команды диагностики SELinux

Команда Описание
getenforce Показать режим SELinux: Enforcing/Permissive/Disabled
sudo setenforce 0 Временно перевести в Permissive (только логирует, не блокирует)
sudo setenforce 1 Вернуть в Enforcing (блокирует нарушения)
ls -Z файл Показать SELinux контекст файла
ps auxZ | grep nginx Показать SELinux контекст процесса
sudo ausearch -m avc -ts recent Показать последние блокировки SELinux
sudo ausearch -m avc -ts today Блокировки за сегодня

8.2 Типичные проблемы SELinux с Docker

Симптом Причина Решение
permission denied на mounted volume Неверный SELinux контекст файла chcon -Rt svirt_sandbox_file_t папка/
SSH не запускается на новом порту Порт не разрешён для ssh_port_t semanage port -a -t ssh_port_t -p tcp ПОРТ
Nginx не может читать файлы Контекст файлов не httpd_content_t chcon -Rt httpd_content_t /var/www/
Приложение не может писать в папку Нет прав на запись по SELinux chcon -t container_file_t папка/

8.3 SELinux на разных дистрибутивах

Дистрибутив Система MAC По умолчанию
Fedora SELinux Enforcing
RHEL/CentOS/Rocky SELinux Enforcing
РЕД ОС SELinux Enforcing
Ubuntu/Debian AppArmor Enabled
Arch Linux Нет Отсутствует

9. Firewall — подробное руководство

9.1 firewalld — основные команды

Команда Описание
sudo firewall-cmd --list-all Показать все правила активной зоны
sudo firewall-cmd --permanent --add-port=8000/tcp Открыть порт навсегда
sudo firewall-cmd --permanent --remove-port=8000/tcp Закрыть порт навсегда
sudo firewall-cmd --reload Применить изменения
sudo firewall-cmd --get-active-zones Показать активные зоны
sudo firewall-cmd --permanent --add-service=http Открыть сервис (http=80)

⚠️ Параметр --permanent сохраняет правило навсегда. Без него правило действует только до перезагрузки. После --permanent всегда выполняйте --reload.

9.2 Firewall на разных дистрибутивах

Дистрибутив Firewall Команды
Fedora/RHEL/РЕД ОС firewalld firewall-cmd
Ubuntu (новые) ufw ufw allow 8000/tcp
Ubuntu (старые) iptables iptables -A INPUT -p tcp --dport 8000 -j ACCEPT
Debian nftables/iptables nft add rule inet filter input tcp dport 8000 accept

10. Диагностика — на что смотреть в логах

10.1 Общая стратегия диагностики

  1. Проверьте статус контейнера: docker ps — Running или Restarting?

  2. Смотрите логи: docker logs имя_контейнера — последние строки покажут ошибку

  3. Проверьте порты: ss -tlnp | grep PORT — слушает ли приложение?

  4. Проверьте сеть: curl http://localhost:PORT — отвечает ли приложение?

  5. Проверьте firewall: firewall-cmd --list-all — открыт ли порт?

  6. Проверьте SELinux: ausearch -m avc -ts recent — есть ли блокировки?

10.2 Ключевые слова для поиска в логах

Ключевое слово Что означает Что делать
permission denied Нет прав доступа к файлу или порту chmod, chcon, или semanage
connection refused Порт закрыт или приложение не слушает Проверить запущен ли сервис, открыт ли порт
no such file or directory Файл или директория не существует Проверить пути, создать директорию
address already in use Порт занят другим процессом ss -tlnp | grep PORT, убить процесс
OOMKilled Контейнер убит из-за нехватки памяти Увеличить лимит памяти в docker-compose
exit code 1 Приложение завершилось с ошибкой Смотреть логи подробнее
exit code 137 Контейнер убит сигналом SIGKILL (OOM) Увеличить память
cannot connect to database Нет подключения к БД Проверить DATABASE_URL, запущена ли БД

10.3 Полезные команды диагностики

# Статус всех контейнеров

docker ps -a

# Последние 50 строк логов

docker logs --tail 50 имя_контейнера

# Следить за логами в реальном времени

docker logs -f имя_контейнера

# Зайти внутрь контейнера для диагностики

docker exec -it имя_контейнера bash

# Посмотреть потребление ресурсов

docker stats

# Детальная информация о контейнере

docker inspect имя_контейнера

11. Алгоритм работы с незнакомым приложением

Когда вам дают незнакомое приложение и просят развернуть — следуйте этому алгоритму:

Шаг 1 — Разведка (15-30 минут)

  • Прочитайте README.md полностью

  • Найдите Dockerfile — есть ли он? Если нет — нужно написать

  • Найдите docker-compose.yml — какие сервисы нужны?

  • Найдите .env.example или env/example — какие переменные нужны?

  • Посмотрите requirements.txt — определите стек технологий

  • Найдите скрипты запуска в scripts/ или Makefile

Шаг 2 — Вопросы к разработчику

  • На каком порту слушает приложение?

  • Какая база данных нужна и какие учётные данные?

  • Есть ли внешние зависимости (Redis, очереди, сторонние API)?

  • Какие переменные окружения обязательны?

  • Как запустить миграции БД?

  • Как проверить что приложение работает (healthcheck endpoint)?

Шаг 3 — Локальный запуск без Docker

Сначала запустите приложение напрямую — это поможет понять как оно работает:

python3 -m venv venv && source venv/bin/activate

pip install -r requirements.txt

python3 app.py # или uvicorn app.main:app

Шаг 4 — Написание Dockerfile (если нет)

Шаблон Dockerfile для Python FastAPI приложения:

FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

Шаг 5 — Написание docker-compose.yml

Минимальный docker-compose.yml для FastAPI + PostgreSQL:

services:

build: .

ports:

- "8000:8000"

- postgres

postgres:

environment:

POSTGRES_DB: myapp

POSTGRES_USER: myapp

POSTGRES_PASSWORD: secret

volumes:

- postgres_data:/var/lib/postgresql/data

volumes:

postgres_data:

Шаг 6 — Проверка и отладка

  • Запустите: docker compose up

  • Смотрите логи: docker compose logs -f

  • Проверьте healthcheck: curl http://localhost:8000/health

  • Проверьте Swagger: http://localhost:8000/docs

На что обратить особое внимание

Аспект Что проверить Потенциальные проблемы
База данных Подключение, миграции, данные Неверный URL, не запущена БД, не применены миграции
